## 目的

本文定义 `v1.3.1` scripts 和 gates 的 authority boundary。

它不删除 scripts，不重跑 proof snapshots，也不重新分类每个具体 script 文件。v1.2 script-surface classification 仍是历史 script-topology baseline。

本文分类的是 scripts 和 gates 被允许决定什么。

## 核心规则

Scripts 和 gates 支持 Agent / Skills-led mainline。它们不替代主线。

Scripts 可以证明 mechanical facts 和 evidence availability。Gates 可以阻止不安全推进或封顶 claims。二者都不能单独决定 product success、architecture quality、implementation value、test meaning 或 investment worth。

短规则：

```text
scripts prove mechanics;
gates protect boundaries;
Agent / Skills judge value;
evidence caps claims.
```

## Script Authority Classes

| Class | Purpose | May prove | Must not decide |
| --- | --- | --- | --- |
| `always-on safety net` | 保持 default path 结构安全 | required files exist、schemas parse、commands are available、obvious contract shape is valid | product correctness、architecture adequacy、implementation value |
| `background regression` | 发现 retained cases 或重复 surfaces 上的 drift | expected outputs remain present、known invariants still hold、forbidden wording 没有重新出现 | new artifact 是否有价值，或 phase 是否应该 close |
| `strict-proof engine` | 提供 high-evidence validation 和 claim ceiling support | trace/evidence closure、runtime/test execution、output-contract validity、package inclusion、formal claim support/failure | business truth、design truth、remediation ownership、release approval |
| `developer diagnostic` | 在 remediation 中帮助定位 failures | failure class、missing artifact、stale path、broken dependency、weak evidence signal | final verdict、phase success、是否值得 further investment |
| `legacy/support` | 保留 compatibility 或 historical helper behavior | historical behavior still runs，或 compatibility path exists | current mainline authority、semantic correctness、default control direction |

## Gate Authority Classes

| Class | Purpose | May decide | Must not decide |
| --- | --- | --- | --- |
| `entry / exit boundary gate` | required inputs 或 outputs 缺失时阻止 phase progression | 基于 declared contract presence 判断 continue、return、blocked 或 review-bound | product、architecture、implementation 或 validation judgment 的实质价值 |
| `mechanical contract gate` | 强制 schemas、IDs、trace shape、package shape 或 command shape | declared mechanical contract 的 pass/fail | artifact 在 business/design terms 下是否有用或正确 |
| `runtime evidence gate` | 证明选定 install/build/start/test/runtime paths | supplied commands/evidence 是否运行，以及适用 environment ceiling | tests 是否 meaningful，或 product value 是否真实 |
| `strict-proof / claim-ceiling gate` | 验证 formal claims 的 evidence sufficiency | supported claim、capped claim、return reason、minimum rerun boundary | upstream product truth、architecture truth、implementation repair、production authorization |
| `review routing gate` | 保留 finding ownership 和 rerun boundary | category、owning phase、required action、review-bound carryover | 静默修复 upstream artifacts 或改变 accepted truth |

## Allowed Script Claims

Scripts 可以声明：

- file existence；
- schema validity；
- mechanical contract compatibility；
- trace ID 或 artifact ID presence；
- install / build / start / test execution，如果命令可用；
- package inclusion；
- output-contract validity；
- evidence aggregation；
- strict-proof pass / return evidence；
- stated environment 内的 formal claim support 或 failure。

## Forbidden Script Claims

Scripts 不能决定：

- product goal correctness；
- business value validity；
- user pain strength；
- narrowest valuable wedge；
- architecture tradeoff quality；
- design elegance 或 adequacy；
- implementation value；
- core-path usability；
- test meaning 或 assertion quality；
- 是否值得 further investment；
- production approval、real owner sign-off、go-live、rollback readiness、monitoring handover 或 production risk acceptance。

## Pass Wording Rule

当 script 或 gate 写 `pass` 时，周围 human-facing wording 必须说明通过的是哪一层。

可接受：

- `schema pass`；
- `output-contract pass`；
- `runtime smoke pass`；
- `strict-proof evidence pass`；
- `package-inclusion pass`；
- `development / pre-production evidence` 内的 `testing-validation pass`。

避免：

- `product pass`；
- `architecture pass`；
- `implementation value pass`；
- `business success pass`；
- 没有 evidence ceiling 的 `delivery success`；
- 没有 explicit external authority 的 `release approved`。

## Routing Rule

如果 gate 发现 semantic defect，应 route 而不是 repair：

- source truth / product-spec gap -> pre-P1 admission 或 P1；
- architecture gap -> P2；
- implementation patch 或 runtime/test evidence gap -> P3；
- validation evidence-consumption 或 closure-judgment defect -> P4；
- external production / owner-signoff / online UAT evidence gap -> 默认不属于 WFF authority，除非作为 external evidence 提供。

## Change Surface Rule

`WO-119A.11` 默认 central-only，除非某个 release-facing surface 明确赋予 scripts 或 gates 与本文冲突的 semantic authority。

不要为了重复此分类而编辑 P1-P4 skills。

不要重写 v1.2 script surface classification；它仍是 topology baseline。本文负责 v1.3 authority 和 wording。

## 验收问题

如果本文正确把 scripts 和 gates 分类为 support layers，则可以接受 `WO-119A.11`：

- scripts 证明 mechanics 和 evidence availability；
- gates 保护 boundaries、路由 defects 并封顶 claims；
- Agent / Skills 保留 product、business、architecture、implementation、test-meaning 和 review judgment；
- script/gate pass wording 必须说明实际通过的是哪一层；
- 不授予 production 或 owner-signoff authority。
