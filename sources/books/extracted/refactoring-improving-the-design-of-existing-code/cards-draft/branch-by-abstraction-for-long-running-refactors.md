# 长期重构策略：用 Branch by Abstraction 降低替换风险

- 类型：方法 / 风险控制
- 适用阶段：开发实现；维护 / 重构
- 来源：`sources/books/original/phase-others/refactoring-improving-the-design-of-existing-code.md`（第2章 2.4 长期重构）
- 摘要：当重构规模大到需要数周时，不应让系统长期不可用。Branch by Abstraction 的要点是：先引入一层抽象兼容新旧实现；让调用方迁移到抽象；最后替换底层实现。

## 输入条件

- 需要替换库/组件/依赖结构或拆分大型模块，无法在短时间完成

## 关键步骤

1. 定义抽象层（接口/适配器）以同时支持旧实现与新实现。
2. 让调用方逐步迁移到抽象层；每一步都保持系统可工作并可验证。
3. 当调用方完全迁移后，替换底层实现（或移除旧实现）。
4. 全程保持安全网：关键路径测试 + 监控/回归验证。

## 输出结果

- 大型替换被拆成可控的小步；系统持续可用

## 适用边界

- 抽象层不是永久架构；迁移完成后要及时清理多余的双实现复杂度。

## 常见错误

- 直接大重写并长时间冻结交付
- 引入抽象后不迁移调用方，导致“新旧并存”长期化
