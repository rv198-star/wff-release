# 基于故事地图做 MVP 切片

- 类型：方法
- 适用阶段：需求拆解 / 发布规划
- 来源：`sources/books/original/用户故事地图.md`（第2章、第11章）
- 摘要：故事地图的一个核心用途是按用户活动和目标切分 MVP、发布版本与后续迭代，而不是仅按功能列表做粗暴优先级排序。

## 输入条件
- 已存在可用的故事地图主干与任务细节
- 已有明确目标，如 MVP、首发版本或某次交付范围

## 关键步骤
1. 基于用户活动主干识别最小可用体验范围。
2. 优先围绕成果和学习目标切片，而不是围绕功能堆叠切片。
3. 区分 MVP、后续发布路线与更远期想法。
4. 用切片结果反查是否仍保持完整用户体验闭环。

## 输出结果
- MVP 范围说明
- 发布路线切片
- 以完整体验为中心的优先级结构

## 适用边界
- 适用于需要控制首发范围、避免过度开发的产品/业务场景
- 不适用于只有技术重构目标、缺少用户体验闭环定义的工作项

## 常见错误
- 把 MVP 当成“最小功能列表”而不是“最小可用体验”
- 只按功能优先级切，不按用户成果切
- 跳过全景图，直接讨论 MVP，导致切片失真

## 关联模板
- MVP 说明
- 路线图说明
- 需求拆解结果

## 下游使用建议
- 可被未来“需求拆解 Skill”引用作为发布切片方法
- 可支持“架构设计 Skill”理解首批范围与非首批范围
- 可作为阶段 gate：没有说明最小可用体验边界时，不得宣布 MVP 已定义
