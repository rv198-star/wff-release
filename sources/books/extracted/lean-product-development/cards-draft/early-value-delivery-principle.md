# 更早交付价值原则

- 类型：概念
- 适用阶段：需求分析 / 交付策略 / 规划
- 来源：`sources/books/original/精益产品开发_原则_方法与实施.md`（第1章）
- 摘要：敏捷/精益产品开发首先不是为了“更快写代码”，而是为了更早交付价值。这里的“快”是时间上的更早，而不是单点开发速度更高。

## 输入条件
- 团队需要定义交付策略、迭代策略或范围切片逻辑
- 当前项目面临时间窗口、市场变化或反馈时效压力

## 关键步骤
1. 从业务视角重新定义“快”，即更早获得价值，而不是更晚一次性交付全部。
2. 将交付策略从单次大交付转向分批和渐进式交付。
3. 让规划、切片和阶段 gate 服务于早价值交付，而不是服务于一次性完整性幻觉。

## 输出结果
- 更符合业务目标的交付策略
- 支持分批交付和早期反馈的规划原则

## 适用边界
- 适用于大多数需要持续获得反馈与业务结果的产品场景
- 不意味着“越快越好”或“不需要质量与边界控制”

## 常见错误
- 把“更快”误解为开发人员压缩工期
- 仍以一次性交付全部范围为默认目标
- 只关注开发效率，不关注价值到达时间

## 下游使用建议
- 可被未来“需求拆解 Skill”用于解释为什么要做切片
- 可被“架构设计 Skill”用于界定首批/后续能力边界
- 可强化阶段 gate：若产物无法说明早价值交付策略，则规划不完整
