# Mindthus v0.3 Design

Status: approved design for implementation planning
Date: 2026-04-29

## Goal

v0.3 要解决两个已经暴露出来的产品化问题：

1. Mindthus skills 的自动唤起率不够理想。用户不理解 SELA、3L5S、EDSP、WAE 时，agent 仍应能从普通问题表达中识别合适的方法镜头。
2. `tplan` 的严格运行时契约直接暴露给 LLM 输出，容易因为 JSON 字段、类型、格式波动而卡住。

v0.3 的目标是：

> 让 Mindthus 更容易被正确唤醒，让 tplan 更不怕结构化输出失败。

Final implementation note for v0.3:

- Passive activation is intentionally limited to `3l5s`, `edsp`, `sela`, and `wae`.
- `tplan` and `tvg` are manual-only / explicitly routed tools. They activate only when a human, agent, script, hook, or workflow explicitly names `tplan` / `Mission runtime` / `TVG`.
- Ordinary long-task, task-tree, persistent-state, shallow-document, audit, or value-deepening language must not passively activate `tplan` or `tvg`.
- Older draft examples in this design and its implementation plan that show `tplan` or `tvg` as passive route targets are superseded by this final boundary.

## 3L5S Framing

### Baseline / 定基线

v0.2 已经建立了较稳的 `tplan` 状态机：

- Mission / Task / SubTask / Step 层级已经明确。
- Evidence、Logs、Archive 已经分层。
- 结构操作开始由脚本控制，agent 负责判断。
- Stop report 可以在无法安全继续时优雅停止。

但当前还有两个弱点：

- `AGENTS.md`、`using-mindthus` 和各 skill `description` 更像方法论说明，不够像场景触发器。
- `apply_decision.py` 直接要求完整 hook output JSON，缺少 LLM loose output 到 strict runtime contract 之间的适配层。

### Target / 立目标

用户不用懂方法论名称，也能在常见表达下触发正确 skill：

- 问题混乱、反复返工、任务过大时，默认触发 `3l5s`。
- A/B 都有道理、边界不清、定性判断悬而不决时，默认触发 `edsp`。
- 旧方案局部优势真实但新方案系统效率明显更高时，默认触发 `sela`。
- 流程、agent、脚本、证据边界不清时，默认触发 `wae`。
- AI 产物结构完整但内容空、判断薄时，不被动触发 `tvg`；只有显式点名或 workflow 明确路由时进入 `tvg`。
- 长任务、任务树、状态持久化、恢复上下文等表述，不被动触发 `tplan`；只有显式点名或 workflow 明确路由时进入 `tplan`。

同时，`tplan` 继续保持严格核心，但不直接要求 LLM 一次性写对完整 JSON：

- 先 validate / prepare，再 apply。
- 常用状态操作提供窄命令。
- 修复失败后生成中文 Stop report，请人介入。

### Gap / 析差距

当前缺少四类能力：

- 场景化路由表：从普通用户表达映射到 Mindthus skill。
- 更强的 skill frontmatter description：让自动 skill selection 更容易命中。
- tplan decision 输出预校验和修复层：把坏 JSON、缺字段、错类型变成可修复反馈。
- 回归测试：同时覆盖 skill activation 和 tplan contract resilience。

### Strategy / 定策略

v0.3 分两条主线：

1. Activation Layer：改造 `AGENTS.md`、`using-mindthus` 和 skill descriptions，让 `3l5s`、`edsp`、`sela`、`wae` 能从普通场景自动唤醒，同时让 `tplan`、`tvg` 明确拒绝被动唤醒。
2. Contract Resilience：在 LLM 输出和 `tplan` 严格运行时之间增加适配、预校验、窄命令和失败降级。

不把 v0.3 做成新的大 runtime。优先补接口层、触发层和测试层。

## Scope

### In Scope

- 更新 `AGENTS.md` 的场景化路由表。
- 更新 `skills/using-mindthus/SKILL.md`，让它成为更明确的 router。
- 重写各 Mindthus skill frontmatter `description`，补充普通用户表达和边界信号。
- 新增 tplan decision 预校验或准备脚本。
- 新增常用 tplan 状态窄命令。
- 新增结构化输出失败的修复/停止规则。
- 新增 activation routing 和 contract resilience 测试。

### Out Of Scope

- 不重写 SELA、3L5S、EDSP、WAE、TVG 的核心方法论。
- 不让 `tplan` 替代语义判断。
- 不做多模型适配器或外部 LLM API 调用。
- 不做完整自然语言解析引擎。
- 不把所有普通任务都自动升级为 `tplan` Mission。

## Work Items

### T1: Mindthus Scenario Router

Modify:

- `AGENTS.md`
- `skills/using-mindthus/SKILL.md`

Add a scenario-first route table:

- User expression
- Default skill
- Why this skill
- Do-not-trigger boundary

Acceptance:

- A user can read the table without knowing method names.
- `using-mindthus` clearly says it is a router, not a required chain.
- `tplan` and `tvg` are documented as manual-only / explicitly routed tools.

### T2: Skill Description Rewrite

Modify:

- `skills/sela/SKILL.md`
- `skills/3l5s/SKILL.md`
- `skills/edsp/SKILL.md`
- `skills/wae/SKILL.md`
- `skills/tvg/SKILL.md`
- `skills/tplan/SKILL.md`

Rewrite frontmatter descriptions so they include:

- common user-language triggers
- the core problem pattern
- one boundary hint to avoid overuse

Acceptance:

- Each description is understandable without reading the full method.
- Each description improves automatic matching from ordinary user language.
- No description relies mainly on the method name itself.

### T3: tplan Decision Prepare / Validate Script

Add:

- `skills/tplan/scripts/prepare_decision.py` or `skills/tplan/scripts/validate_decision.py`

Behavior:

- Read a candidate decision JSON file.
- Validate hook output without mutating Mission state.
- Return machine-readable and user-readable feedback:
  - `valid`
  - `errors`
  - `repair_template`
  - `next_action`

Acceptance:

- Missing fields, wrong types, unsupported recommendation, bad mutation shape, and invalid JSON are reported without traceback.
- The script never writes `mission.json`.
- The repair template is small enough to feed back to an LLM.

### T4: tplan Narrow State Commands

Add narrow scripts for common deterministic operations:

- `set_active_task.py`
- `complete_task.py`
- `block_task.py`
- `pause_task.py`

Behavior:

- Use existing runtime helpers for legal state transitions.
- Avoid requiring full hook output JSON for simple state changes.
- Keep high-impact semantic decisions routed through decision hooks.

Acceptance:

- Common status changes can be executed without writing `proposed_mutations`.
- Unknown task IDs and illegal statuses fail without partial writes.
- Scripts do not decide whether the transition is semantically correct.

### T5: Contract Failure Repair And Stop Rule

Add guidance and tests for a two-step failure policy:

1. First contract failure: return a repair template and ask the agent to repair the structured output.
2. Second contract failure: use `stop_report.py` and request human intervention.

Acceptance:

- The rule is documented in `tplan` runtime guidance.
- Tests cover one repair-success path and one repeated-failure stop path.
- Stop report remains concise Chinese by default.

### T6: Activation Regression Tests

Add a lightweight routing regression suite.

The suite should contain ordinary user expressions and expected skills, for example:

- "这个需求越做越乱，不知道该先解决什么" -> `3l5s`
- "两个方案都对，但边界到底在哪" -> `edsp`
- "旧方案体验很好，新方案成本效率高很多" -> `sela`
- "这里到底该脚本控制还是 agent 判断" -> `wae`
- "这份 AI 文档很完整但没什么用" -> no passive skill, unless explicitly routed to `tvg`
- "我要跑一个长期目标，并保留任务状态和证据" -> no passive `tplan`; route to `wae` only if the request asks a real control-boundary question

Acceptance:

- Each auto-trigger skill has at least 5 positive examples.
- `tplan` and `tvg` each have manual-only anti-overuse examples.
- Tests verify the route table and skill descriptions contain enough trigger coverage.

## Architecture

v0.3 keeps strict runtime state at the bottom and adds a softer adapter above it:

```text
ordinary user expression
  -> AGENTS.md / using-mindthus scenario router
  -> selected auto-trigger Mindthus skill, or null for manual-only tplan/tvg-shaped prompts
  -> optional explicitly routed tplan runtime packet
  -> prepare_decision / validate_decision
  -> apply_decision or narrow state command
  -> stop_report if repeated contract failure blocks progress
```

Control boundary:

- `AGENTS.md` and `using-mindthus` control route selection hints.
- Individual skills control method judgment.
- `tplan` scripts control deterministic state and validation.
- Repair templates help LLMs fix shape; they do not prove semantic correctness.
- Stop report hands off when continuing would require inventing intent, authority, or contract shape.

## Testing

Required verification:

```bash
python3 -m unittest discover -s tests/tplan -v
python3 -m unittest tests.test_packaging_docs -v
python3 -m py_compile skills/tplan/scripts/*.py
git diff --check
```

New tests should cover:

- activation routing examples
- description trigger coverage
- malformed decision JSON
- missing required hook fields
- wrong field types
- invalid mutation shape
- prepare/validate without mutation
- narrow command success and failure paths
- repeated contract failure leading to Stop report guidance

## Implementation Order

1. T1 and T2 first, because they improve daily skill activation without touching runtime state.
2. T3 next, because it creates the adapter layer before adding more commands.
3. T4 after T3, using the same validation and runtime helpers.
4. T5 after T3/T4, because the repair/stop rule depends on clear validation errors.
5. T6 throughout, but final routing tests should be tightened after T1/T2 text stabilizes.

## Risks

- Over-triggering `tplan`: any mention of "task" or "plan" must not automatically create a Mission runtime.
- Over-triggering `tvg`: not every document needs depth enhancement.
- Router text becoming too long: the route table must stay scannable.
- Repair templates becoming a hidden semantic validator: they should fix shape only.
- Narrow commands bypassing important review gates: high-impact decisions must still route through hooks.

## Exit Criteria

v0.3 is complete when:

- Mindthus has a scenario-first activation layer.
- Auto-trigger skill descriptions are easier to trigger from ordinary user language.
- `tplan` and `tvg` descriptions act as hard negative gates for passive activation.
- `tplan` has a pre-apply decision validation path.
- Common state transitions have narrow commands.
- Repeated contract failure has a documented graceful stop path.
- Regression tests cover both activation and contract resilience.
