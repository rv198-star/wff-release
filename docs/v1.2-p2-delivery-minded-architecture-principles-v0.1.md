# v1.2 P2 Delivery-Minded Architecture Principles（v0.1）

## 1. Purpose

This document freezes the `v1.2.2` top-level design correction for `P2`:

> **P2 should behave like a delivery-minded architect with ROI-gated extensibility.**

It is a companion to:

- `docs/v1.2-design-principles-v0.1.md`
- `docs/v1.2-phase-thinking-mode-split-v0.1.md`
- `runtime-deps/mindthus/source/skills/wae/SKILL.md`
- `docs/phases/phase-2/engineering-spec-pack-v0.1.md`

This document answers a specific question:

> After P1 deepens and freezes the business world, how should P2 balance architecture depth, delivery efficiency, extensibility, and over-design risk?

---

## 2. Core Role Definition

`P2` is not a pure architecture exercise.
It is also not a narrow implementation checklist.

The default P2 role is:

> **delivery-minded architect with ROI-gated extensibility**

That means P2 should:

- think like an architect about boundaries, evolution, quality attributes, data ownership, dependency posture, and extensibility
- think like an engineering delivery owner about first-delivery path, implementation risk, validation cost, operational load, and contributor cognitive load
- use ROI to decide which forward-looking architecture boundaries deserve to be paid for now
- keep optional architecture excellence explicit when the ROI is not yet strong enough for the default mainline

Chinese short rule:

> **P2 是以交付为约束、以 ROI 为门槛的架构判断者。**

---

## 3. What P2 Optimizes For

P2 should optimize for:

- implementation-facing clarity
- sufficient architecture for first delivery
- honest realizability
- explicit data and interface contracts
- bounded extensibility when ROI justifies it
- low unnecessary implementation and maintenance cost
- downstream handoff that lets P3 build without inventing key design truth

P2 should not optimize for:

- maximal architecture elegance
- broad future-state completeness
- generic enterprise architecture appearance
- optional expansion presented as default mainline
- local expert brilliance that raises average-case cost without clear system-level benefit

Short rule:

> **P2 should produce sufficient, realizable, selectively extensible architecture—not minimal architecture and not maximal architecture.**

---

## 4. Workflow / Agentic Boundary In P2

The general Workflow / Agentic split remains unchanged.

In P2, Workflow should keep:

- stage order
- ESP section contract
- completeness checklist
- trace absorption checks
- data/interface existence checks
- realizability gate existence
- handoff state
- evidence aggregation
- phase verdict recording

In P2, Agentic should handle:

- architecture boundary choice
- why-this-boundary-not-that reasoning
- module/service decomposition judgment
- data invariant and ownership reasoning
- external dependency realism
- quality attribute trade-offs
- delivery economy and complexity trade-offs
- extensibility ROI judgment
- anti-overengineering review

Scripts may expose missing design truth, but must not manufacture design judgment by generic field completion.

Short rule:

> **P2 Workflow keeps the design package ordered; P2 Agentic decides whether the architecture is the right support for the chosen business world.**

---

## 5. P2 Loop Posture

P2 should use loops, but not P1-style broad deep loops.

P1 loops are mainly value-discovery loops:

- is the business world deep enough?
- is the value mechanism strong enough?
- is the output still demo-like?
- would another round create positive business value gain?

P2 loops are targeted design-convergence loops:

- is this boundary choice defensible?
- is this data model implementable?
- are state, permission, and interface contracts explicit enough?
- is dependency realism honest?
- is the design too thin for P3 to implement?
- is the design too heavy for the current ROI?
- can a lighter boundary preserve future option value without dragging first delivery?

P2 should loop when at least one of these is true:

1. P3 would need to invent key design truth.
2. a major boundary cannot answer `why-this-not-that`.
3. the data/interface/storage posture is insufficient for implementation.
4. external dependency or runtime feasibility is unclear.
5. extensibility pressure is real but the ROI and implementation cost have not been judged.
6. the current design looks elegant but may slow first delivery or raise average-case cognitive load.

P2 should stop looping when:

- the ESP is sufficient for P3 implementation;
- major design decisions are resolved or explicitly review-bound;
- realizability is honest;
- another loop is unlikely to materially improve implementation leverage, delivery economics, robustness, or justified extensibility.

Short rule:

> **P2 thinks deeply at decision pressure points, not everywhere by default.**

## 6. ROI-Gated Extensibility

P2 must allow some forward-looking architecture and extensibility.

But extensibility enters the default mainline only when ROI justifies paying design cost now.

### 6.1 Allowed Extensibility

Extensibility may enter P2 mainline when:

1. the pressure comes from the frozen P1 thesis, proof track, or explicit business path;
2. not reserving the boundary now would create materially higher later migration cost;
3. the current reservation cost is low relative to likely future benefit;
4. the boundary can be expressed lightly, such as through module seams, interface contracts, schema evolution posture, ADRs, event naming, or deferred adapter contracts;
5. it does not force P3 to implement complexity that first delivery does not need;
6. it lowers system-level cost, risk, compliance burden, or future change cost.

### 6.2 Disallowed Extensibility

Extensibility should not enter the default mainline when it is justified mainly by:

- “maybe later” speculation;
- architecture elegance;
- broad future-state completeness;
- desire to look enterprise-grade;
- P2 inventing future business that P1 did not select;
- optional capability being written as mandatory implementation path;
- current environment lacking a credible way to build or validate the design.

### 6.3 Three-Lane Classification

P2 should classify extensibility decisions into three lanes:

1. `Mainline Required`
   - Required for first delivery or P3 would need to guess.
2. `ROI-Justified Extension Boundary`
   - Not fully required for first delivery, but low-cost reservation now has clear ROI.
3. `Optional / Expert Lane`
   - Potentially valuable, but current ROI is insufficient for default mainline.

Short rule:

> **Architecture sufficiency does not mean minimum architecture; it means ROI-bounded architecture.**

---

## 7. System Efficiency As A Stronger P2 Tiebreaker

The repo-wide system-efficiency principle matters especially in P2.

Architecture decisions often have attractive local advantages:

- cleaner abstraction
- more general future architecture
- stronger expert-lane expression
- more elegant decomposition
- richer integration posture

These advantages can be real and still not deserve the default mainline.

When two options both satisfy the minimum quality floor, P2 should prefer the option with better full-chain system economics:

- lower average implementation cost
- lower maintenance burden
- lower validation cost
- lower contributor cognitive load
- faster iteration
- stronger reuse across ordinary projects
- clearer handoff to P3

This does not ban forward-looking architecture.
It asks whether the forward-looking cost is ROI-positive at the system level.

Short rule:

> **In P2, local architecture elegance loses to system-level delivery economics unless the elegance materially lowers future cost or risk.**

---

## 8. Anti-Overengineering Review

P2 should add an anti-overengineering review as the architecture counterpart to P1 anti-demo review.

Ask:

1. Does this boundary serve the selected P1 thesis, or did P2 invent future business?
2. Is this service/module split needed now, or would a simpler seam preserve the option?
3. Does this abstraction reduce average-case cost, or only improve local elegance?
4. Does this optional capability slow P3 first delivery?
5. Is traceability making implementation clearer, or obscuring weak design judgment?
6. Is an external dependency honestly realizable in the target environment?
7. Are we paying full implementation cost for something that only needs a deferred boundary?

If the answer reveals over-design, P2 should either simplify the mainline or move the item into `Optional / Expert Lane`.

---

## 9. P2 Exit Rule

P2 may freeze when all of the following are true:

1. P1 business/release truth is inherited without being rewritten as new product truth.
2. The ESP is implementation-facing enough for P3 to start.
3. Major architecture boundaries explain `why-this-not-that`.
4. Data ownership, state, interface, storage, compliance, and operational posture are explicit where relevant.
5. Realizability is honestly judged.
6. Extensibility decisions are classified as `Mainline Required`, `ROI-Justified Extension Boundary`, or `Optional / Expert Lane`.
7. Optional architecture excellence does not obscure the default implementation path.
8. Another loop is unlikely to materially improve implementation leverage, delivery economics, robustness, or ROI-justified extensibility.

P2 must not freeze when:

- P3 would still need to invent key design truth;
- major boundaries are present but unjustified;
- a clean-looking architecture is not honestly buildable;
- optional future architecture is mixed into the default mainline;
- system efficiency is sacrificed for local elegance without ROI.

---

## 10. One-Line Rule

> **P1 expands and selects the business world; P2 designs the smallest sufficient, honestly realizable, ROI-extensible architecture that can support that world.**
