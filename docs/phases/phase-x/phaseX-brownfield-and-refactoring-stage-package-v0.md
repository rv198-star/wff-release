# PhaseX — 历史项目接入与技术重构阶段 Skill 包草案（v0）

## 状态标记

- 当前状态：`incubation-draft`
- 定位：P1-P4 主链之外的特殊扩展轨，不是默认主生命周期阶段
- 目的：为 brownfield / legacy / refactoring / migration 场景提供可编排 Skill 池
- 当前建议：先按 Wave-1 落地最小可用集（PX-SK-01 / PX-SK-04 / PX-SK-07 / PX-SK-06 partial），不要一次性展开 10 个 Skill

## 文档目的

这份文档是给人类评审的 `v0` 大纲草案，用于说明 **PhaseX（棕地项目阶段）** 的定位、Skill 池设计、编排模型、场景覆盖、与 P1-P4 的衔接机制。

PhaseX 不是一条固定流水线，而是一组 **可独立调用、有前置依赖的 Skills + 轻量编排层**，用于解决 P1-P4 当前未覆盖的核心问题：**已有系统如何接入这套技能链路。**

> **核心设计原则**：PhaseX 不强制线性 Stage 顺序，而是提供 Skill 菜单 + 依赖图 + 推荐 Profile，允许按项目特征裁剪组合。

---

## 一、为什么需要 PhaseX

### 1.1 P1-P4 的隐含假设

当前 P1-P4 链路假设的起点是 **绿地项目**：有一份原始需求描述，从零开始生成 PRD，再从白纸设计架构。但现实中大量场景是 **棕地项目**：

- 已有代码库、已有架构、已有技术债
- 业务在跑，不能停机重写
- 改造范围不确定，需要先梳理现状
- 改造目标多样——可能是业务升级，也可能只是技术层面的代码质量提升

这类项目无法直接进入 P1，因为缺少一个 **"从现状到目标的桥接阶段"**。

### 1.2 PhaseX 解决什么

PhaseX 负责回答：
- **现在是什么**（AS-IS 基线）
- **哪里有问题**（健康度/合规/性能评估）
- **要变成什么**（TO-BE 目标，如果有的话）
- **差距在哪里**（GAP 分析）
- **怎么安全地改**（迁移策略/重构策略/测试护网）

PhaseX 的输出物被 P1-P4 消费：
- 如果是业务变更 → 输出变更需求包，**重入 P1**（P1 需要感知基线约束）
- 如果是纯技术重构 → 输出重构任务包，**直接进入 P3**
- 如果是技术栈迁移 → 输出迁移计划，**进入 P3 + P4**

## 二、PhaseX 与线性 Stage 模型的区别

### 2.1 为什么不用 Stage 1→2→3→4

P1-P4 能用线性 Stage 走，是因为输入确定、路径唯一。PhaseX 面对的场景本质上是 **条件分支 + 按需组合**：

- 有的项目只需要"代码梳理 → 补测试 → 分模块重构"（3 个 Skill）
- 有的项目需要"全量四层基线 → 目标架构 → GAP → 拆需求变更 → 重入 P1"（6 个 Skill）
- 有的项目介于两者之间——梳理完发现某个模块需要业务改造，其他模块只需要技术重构
- 需求变更场景甚至可能跳过全量基线，直接从变更点切入

硬塞进线性 Stage 结构会制造不必要的约束。

### 2.2 PhaseX 的结构模型

```
PhaseX = Skill 池 + 依赖图 + 编排层 + Profile 推荐
```

- **Skill 池**：10 个可独立调用的 Skill，每个有明确的输入/输出/前置条件
- **依赖图**：Skill 之间的硬依赖和软依赖关系
- **编排层**：根据项目特征决定"这个项目需要哪几个 Skill"
- **Profile**：预定义的常见走法（推荐路径），但允许自由组合

## 三、PhaseX Skill 池（10 个 Skill）

### 3.1 Skill 总览

#### 核心 Skills（通用，大多数场景都会用到）

| Skill ID | Skill 名称 | 英文标识 | 一句话目标 |
|---|---|---|---|
| PX-SK-01 | 代码库基线提取 | `codebase-baseline-extraction` | 从代码仓库中提取 AS-IS 应用架构、模块结构、依赖关系、技术栈 |
| PX-SK-02 | 数据库基线提取 | `database-baseline-extraction` | 从数据库/DDL 中提取 AS-IS 数据架构、表结构、关系、数据特征 |
| PX-SK-03 | 业务架构梳理 | `business-architecture-discovery` | 从文档/访谈/代码行为中还原 AS-IS 业务流程、领域模型、角色权限 |
| PX-SK-04 | 技术健康度评估 | `technical-health-assessment` | 对代码质量、测试覆盖、依赖风险、安全合规、性能瓶颈、耦合度做量化评估 |
| PX-SK-05 | 目标架构设计 | `target-architecture-design` | 定义 TO-BE 架构（业务/应用/数据/技术四层） |
| PX-SK-06 | GAP 分析与变更拆分 | `gap-analysis-and-change-decomposition` | 对比 AS-IS 与 TO-BE，识别差距，拆分为可被下游消费的变更需求或改造任务 |
| PX-SK-07 | 测试护网构建 | `safety-net-test-construction` | 为现有系统补充单元/接口/契约测试，建立重构安全网 |
| PX-SK-08 | 重构策略制定 | `refactoring-strategy-definition` | 制定模块级重构计划（顺序、边界、验证标准、回滚策略） |

#### 扩展 Skills（复杂场景按需启用）

| Skill ID | Skill 名称 | 英文标识 | 一句话目标 |
|---|---|---|---|
| PX-SK-09 | 迁移策略与切换计划 | `migration-strategy-and-cutover-plan` | 制定技术栈/数据迁移的灰度、双写、切换、回滚方案 |
| PX-SK-10 | 多系统实体映射 | `multi-system-entity-mapping` | 跨系统的实体对齐、冲突检测、主数据治理方案 |

### 3.2 各 Skill 详细定义

#### PX-SK-01：代码库基线提取

| 项目 | 内容 |
|---|---|
| Skill 名称 | `codebase-baseline-extraction` |
| 目标 | 从已有代码仓库中提取应用架构基线，形成可被后续 Skill 消费的结构化画像 |
| 典型输入物 | 代码仓库（Git）、构建配置、部署脚本、已有技术文档（可选） |
| 典型输出物 | 应用架构画像（模块清单、依赖图、技术栈清单、入口/边界识别、API 清单）|
| 前置条件 | 代码可获取；如代码无法运行（如烂尾项目），需先执行可运行性预检 |
| 自动化能力 | 静态分析（AST/依赖树）、框架识别、API 路由提取、模块边界推断 |
| Profile 变体 | 后端 profile（重点：服务/模块/依赖）、前端 profile（重点：组件树/路由/状态管理）、全栈 profile |
| SOP 重点 | 识别入口点 → 提取模块结构 → 构建依赖图 → 识别技术栈 → 标注已知文档与代码的偏差 |
| 可运行性预检 | 尝试构建/启动 → 记录失败点 → 标注环境依赖 → 输出可运行性报告 |

#### PX-SK-02：数据库基线提取

| 项目 | 内容 |
|---|---|
| Skill 名称 | `database-baseline-extraction` |
| 目标 | 从数据库连接或 DDL 中提取数据架构基线 |
| 典型输入物 | 数据库连接信息、DDL 文件、ORM 模型定义、数据字典（可选） |
| 典型输出物 | 数据架构画像（表清单、字段定义、关系图/ER 图、索引策略、数据量级估算、敏感字段标注） |
| 前置条件 | 至少有 DDL 或 ORM 定义可用；有连接则可做数据量级和特征采样 |
| 自动化能力 | Schema dump、ER 图生成、数据量级统计、敏感字段识别（PII/支付等）、外键/索引分析 |
| SOP 重点 | 获取 Schema → 分析表关系 → 评估数据量级 → 标注敏感字段 → 识别冗余/反范式 → 输出数据画像 |

#### PX-SK-03：业务架构梳理

| 项目 | 内容 |
|---|---|
| Skill 名称 | `business-architecture-discovery` |
| 目标 | 还原 AS-IS 业务流程、领域模型、角色权限体系 |
| 典型输入物 | 产品文档、用户手册、代码行为分析结果（PX-SK-01 输出）、访谈记录、工单/缺陷历史 |
| 典型输出物 | 业务架构画像（业务流程图、领域模型、角色权限矩阵、业务规则清单、业务痛点清单） |
| 前置条件 | 至少有代码或文档之一可用 |
| 自动化能力 | 从代码中推断业务流程（路由→控制器→服务调用链）、从 UI 中提取角色/权限模型 |
| SOP 重点 | 识别核心业务流程 → 梳理角色与权限 → 提取业务规则 → 标注隐式逻辑（代码中有但文档中没有的规则） → 形成业务架构画像 |
| 注意事项 | 这是最依赖人工输入的 Skill——代码能告诉你"系统做了什么"，但不一定能告诉你"业务为什么这样做" |

#### PX-SK-04：技术健康度评估

| 项目 | 内容 |
|---|---|
| Skill 名称 | `technical-health-assessment` |
| 目标 | 对现有系统进行多维度健康度量化评估 |
| 典型输入物 | PX-SK-01 输出、PX-SK-02 输出（可选）、CI/CD 配置、测试报告（如有） |
| 典型输出物 | 技术健康报告（评分卡 + 债务清单 + 风险矩阵） |
| 前置条件 | PX-SK-01 已完成 |
| 评估维度（默认） | 代码质量、测试覆盖率、依赖健康（过期/漏洞）、架构耦合度、文档完整度 |
| Profile: 性能 | 增加维度：响应时间基线、吞吐量、资源利用率、慢查询、N+1 问题 |
| Profile: 合规 | 增加维度：法规条文→技术映射（等保/GDPR/数据本地化）、敏感数据处理、审计日志覆盖 |
| Profile: 耦合度 | 增加维度：模块间调用频次、共享数据库表、循环依赖、变更影响半径 |
| SOP 重点 | 运行静态分析 → 收集指标 → 按维度评分 → 识别 Top-N 债务项 → 输出风险矩阵 |
| 输出格式 | 健康评分卡（每个维度 1-10 分 + 权重）+ 技术债务清单（按优先级排序）+ 风险矩阵 |

#### PX-SK-05：目标架构设计

| 项目 | 内容 |
|---|---|
| Skill 名称 | `target-architecture-design` |
| 目标 | 定义 TO-BE 目标架构（业务/应用/数据/技术四层） |
| 典型输入物 | PX-SK-01~03 输出（AS-IS 基线）、业务目标/改造目标、技术约束 |
| 典型输出物 | 目标架构文档（四层架构图 + 关键设计决策 + 约束清单 + 不变量声明） |
| 前置条件 | 至少 PX-SK-01 已完成；PX-SK-03 在业务变更场景下为必需 |
| 四层结构 | 业务架构（流程/领域/规则变化）→ 应用架构（模块/服务/边界变化）→ 数据架构（Schema/存储/迁移变化）→ 技术架构（栈/基础设施/部署变化） |
| SOP 重点 | 明确改造目标 → 逐层设计 TO-BE → 标注每层的变与不变 → 记录关键决策 → 输出约束清单（"哪些不能改"） |
| 不变量声明 | 显式列出改造过程中必须保持不变的约束（如：某些 API 必须向后兼容、某些表结构不能改、某些业务规则不能变） |

#### PX-SK-06：GAP 分析与变更拆分

| 项目 | 内容 |
|---|---|
| Skill 名称 | `gap-analysis-and-change-decomposition` |
| 目标 | 对比 AS-IS 与 TO-BE，识别差距，拆分为可被下游消费的变更需求或改造任务 |
| 典型输入物 | AS-IS 基线（PX-SK-01~03）、TO-BE 目标（PX-SK-05）|
| 典型输出物 | GAP 清单 + 变更需求包（每条标注影响范围：新增/修改/废弃） |
| 前置条件 | PX-SK-05 已完成（全量模式）；或至少有变更点描述（局部模式） |
| 运行模式 | **全量模式**：四层逐层对比 AS-IS vs TO-BE → **局部模式**：仅在变更点做局部 GAP，跳过全量基线 |
| 输出与 P1 的衔接 | 变更需求包的格式应可被 P1 消费——每条变更需求包含：变更描述、影响模块、基线约束、验收标准 |
| SOP 重点 | 逐层对比 → 识别差异点 → 分类（新增/修改/废弃/不变）→ 评估影响范围 → 拆分为独立的变更需求 → 排优先级 |
| 兼容性约束输出 | 标注哪些接口/数据结构必须保持向后兼容 |

#### PX-SK-07：测试护网构建

| 项目 | 内容 |
|---|---|
| Skill 名称 | `safety-net-test-construction` |
| 目标 | 为现有系统补充测试覆盖，建立重构/迁移的安全网 |
| 典型输入物 | PX-SK-01 输出、PX-SK-04 输出（测试覆盖率数据）、已有测试代码 |
| 典型输出物 | 测试护网清单（需补测试的模块 + 优先级）+ 测试代码/用例骨架 |
| 前置条件 | PX-SK-01 已完成 |
| SOP 重点 | 识别核心路径 → 评估现有覆盖率 → 确定护网优先级（先保护要改的模块）→ 补充接口测试/契约测试/关键路径集成测试 → 验证护网有效性 |
| 优先级策略 | 先覆盖即将被重构/迁移的模块 → 再覆盖核心业务路径 → 最后补充边缘场景 |
| 护网类型 | 接口契约测试（API 不变性）、关键路径端到端测试（业务不变性）、数据一致性校验（数据不变性） |

#### PX-SK-08：重构策略制定

| 项目 | 内容 |
|---|---|
| Skill 名称 | `refactoring-strategy-definition` |
| 目标 | 制定模块级重构计划 |
| 典型输入物 | PX-SK-04 输出（健康度报告 + 债务清单） |
| 典型输出物 | 重构计划（模块顺序、每模块改造边界、验证标准、回滚策略） |
| 前置条件 | PX-SK-04 已完成 |
| SOP 重点 | 从债务清单中筛选目标 → 确定重构顺序（依赖少的先动）→ 为每个模块定义改造边界 → 定义"完成"标准 → 定义回滚策略 |
| 典型重构目标 | 提高可扩展性、提升性能、抽取公共模块、降低耦合度、升级框架版本 |
| 与 PX-SK-07 的关系 | 重构策略的执行前提是测试护网已就位——先有护网，再动刀 |

#### PX-SK-09：迁移策略与切换计划

| 项目 | 内容 |
|---|---|
| Skill 名称 | `migration-strategy-and-cutover-plan` |
| 目标 | 制定技术栈迁移或数据迁移的完整切换方案 |
| 典型输入物 | AS-IS 基线、TO-BE 目标、PX-SK-04 输出 |
| 典型输出物 | 迁移计划（阶段划分、灰度策略、双写方案、数据校验方案、回滚触发条件、切换检查清单） |
| 前置条件 | PX-SK-01 + PX-SK-05 已完成 |
| 适用场景 | 技术栈迁移（Java→Go / 单体→微服务）、数据库迁移（MySQL→PostgreSQL / 分库分表）、云迁移 |
| SOP 重点 | 评估迁移复杂度 → 选择迁移策略（大爆炸/绞杀者/并行运行）→ 设计灰度方案 → 设计数据同步/双写 → 定义回滚条件 → 定义切换检查清单 |
| 关键输出 | 回滚触发条件必须可量化（如：错误率 > 1%、延迟 P99 > 500ms） |

#### PX-SK-10：多系统实体映射

| 项目 | 内容 |
|---|---|
| Skill 名称 | `multi-system-entity-mapping` |
| 目标 | 跨系统的实体对齐、冲突检测、主数据治理方案 |
| 典型输入物 | 多个系统的 PX-SK-01 + PX-SK-02 输出 |
| 典型输出物 | 实体映射表（同一业务实体在不同系统的定义对照）、冲突清单、主数据方案建议 |
| 前置条件 | 至少 2 个系统的 PX-SK-01 和/或 PX-SK-02 已完成 |
| 适用场景 | 并购后系统整合、多套业务系统统一、主数据平台建设 |
| SOP 重点 | 收集各系统实体定义 → 按业务语义对齐 → 识别冲突（同名异义 / 异名同义 / 字段类型不一致）→ 确定主数据源 → 设计映射规则 |

## 四、Skill 依赖图与编排规则

### 4.1 依赖图

```
PX-SK-01 ─┬──→ PX-SK-04 ──→ PX-SK-08 ──────→ [P3 入口]
(代码基线)  │    (健康度)      (重构策略)          ↑
           │                                     │
           ├──→ PX-SK-07 ────────────────────────┘
           │    (测试护网)
           │
PX-SK-02 ──┤
(数据基线)  │
           │
PX-SK-03 ──┴──→ PX-SK-05 ──→ PX-SK-06 ──────→ [P1 入口]
(业务架构)       (目标架构)    (GAP 分析)      (约束模式)
                    │
                    └──→ PX-SK-09 ──────────→ [P3 入口]
                         (迁移策略)

(PX-SK-01 + PX-SK-02) × N ──→ PX-SK-10 ──→ PX-SK-05
                               (多系统映射)
```

### 4.2 依赖规则表

| Skill | 硬依赖（必须先完成） | 软依赖（有则更好） | 可并行 |
|---|---|---|---|
| PX-SK-01 | 无 | — | 可与 PX-SK-02、PX-SK-03 并行 |
| PX-SK-02 | 无 | — | 可与 PX-SK-01、PX-SK-03 并行 |
| PX-SK-03 | 无 | PX-SK-01（代码行为辅助理解业务） | 可与 PX-SK-01、PX-SK-02 并行 |
| PX-SK-04 | PX-SK-01 | PX-SK-02 | — |
| PX-SK-05 | PX-SK-01 | PX-SK-02、PX-SK-03 | — |
| PX-SK-06 | PX-SK-05（全量模式）或 变更点描述（局部模式） | PX-SK-01~03 | — |
| PX-SK-07 | PX-SK-01 | PX-SK-04（覆盖率数据） | 可与 PX-SK-08 并行 |
| PX-SK-08 | PX-SK-04 | PX-SK-07（护网就绪确认） | 可与 PX-SK-07 并行（但执行时护网应先就位） |
| PX-SK-09 | PX-SK-01 + PX-SK-05 | PX-SK-02 | — |
| PX-SK-10 | 多系统的 PX-SK-01/02 | PX-SK-03 | — |

### 4.3 编排层职责

编排层不是一个独立 Skill，而是一个轻量决策逻辑：

1. **输入**：项目特征描述（驱动力、范围、约束）
2. **输出**：推荐的 Skill 组合 + 执行顺序 + 出口指向
3. **决策依据**：
   - 业务需求是否要变？→ 决定是否需要 PX-SK-03 + PX-SK-05 + PX-SK-06
   - 涉及几个系统？→ 决定是否需要 PX-SK-10
   - 是否涉及技术栈/数据迁移？→ 决定是否需要 PX-SK-09
   - 改造范围是全量还是局部？→ 决定 PX-SK-06 的运行模式

## 五、场景 Profile（推荐走法）

### Profile A：纯技术重构

> **场景特征**：业务不变，代码质量差 / 性能低 / 框架过时 / 需要抽取公共模块

```
PX-SK-01 → PX-SK-04 → PX-SK-07 + PX-SK-08（并行）→ P3 → P4
```

| 步骤 | Skill | 产出 |
|---|---|---|
| 1 | PX-SK-01 代码库基线提取 | 应用架构画像 |
| 2 | PX-SK-04 技术健康度评估 | 健康报告 + 债务清单 |
| 3a | PX-SK-07 测试护网构建 | 测试护网 |
| 3b | PX-SK-08 重构策略制定 | 重构计划 |
| 4 | → P3（分模块重构执行） | 重构后代码 |
| 5 | → P4（回归验证） | 验证报告 |

- **不需要**：PX-SK-02（除非涉及数据层改动）、PX-SK-03（业务不变）、PX-SK-05（无 TO-BE）、PX-SK-06（无 GAP）
- **P1 不需要重入**

### Profile B：历史项目现代化

> **场景特征**：业务要大改，现有架构撑不住，需要重新梳理 + 重新设计

```
PX-SK-01 + PX-SK-02 + PX-SK-03（并行）
    → PX-SK-04
    → PX-SK-05
    → PX-SK-06
    → P1（约束模式）→ P2 → P3 → P4
```

| 步骤 | Skill | 产出 |
|---|---|---|
| 1a | PX-SK-01 代码库基线提取 | 应用架构画像 |
| 1b | PX-SK-02 数据库基线提取 | 数据架构画像 |
| 1c | PX-SK-03 业务架构梳理 | 业务架构画像 |
| 2 | PX-SK-04 技术健康度评估 | 健康报告 |
| 3 | PX-SK-05 目标架构设计 | TO-BE 四层架构 |
| 4 | PX-SK-06 GAP 分析与变更拆分 | 变更需求包 + 基线约束包 |
| 5 | → P1（接受约束输入） | 变更 PRD |
| 6-8 | → P2 → P3 → P4 | 设计 → 实现 → 验证 |

- **P1 需要扩展**：接受"基线约束包 + 变更需求"而非仅"原始需求描述"

### Profile C：局部需求变更

> **场景特征**：改某个模块的业务逻辑，其他模块不动

```
PX-SK-01（仅目标模块）→ PX-SK-06（局部模式）→ P1 → P2 → P3 → P4
```

| 步骤 | Skill | 产出 |
|---|---|---|
| 1 | PX-SK-01 代码库基线提取（局部） | 目标模块架构画像 |
| 2 | PX-SK-06 GAP 分析（局部模式） | 变更需求 + 兼容性约束 |
| 3 | → P1（约束模式） | 变更 PRD |
| 4-6 | → P2 → P3 → P4 | 设计 → 实现 → 验证 |

- **跳过**：全量基线、健康度评估、目标架构设计
- **PX-SK-06 运行在局部模式**：接受变更点描述而非完整 TO-BE

### Profile D：技术栈迁移

> **场景特征**：Java→Go / 单体→微服务 / 换数据库，业务逻辑基本不变

```
PX-SK-01 + PX-SK-02（并行）
    → PX-SK-04
    → PX-SK-05
    → PX-SK-09
    → PX-SK-07
    → P3 → P4
```

| 步骤 | Skill | 产出 |
|---|---|---|
| 1a | PX-SK-01 代码库基线提取 | 应用架构画像 |
| 1b | PX-SK-02 数据库基线提取 | 数据架构画像 |
| 2 | PX-SK-04 技术健康度评估 | 健康报告 |
| 3 | PX-SK-05 目标架构设计 | TO-BE 架构（新技术栈） |
| 4 | PX-SK-09 迁移策略与切换计划 | 灰度/双写/回滚方案 |
| 5 | PX-SK-07 测试护网构建 | 迁移验证测试 |
| 6 | → P3（迁移执行） | 新栈代码 |
| 7 | → P4（回归 + 切换验证） | 验证报告 |

### Profile E：安全合规改造

> **场景特征**：等保/GDPR/数据本地化要求，主要是技术约束驱动的改造

```
PX-SK-01 + PX-SK-02 + PX-SK-03（并行）
    → PX-SK-04（合规 profile）
    → PX-SK-06
    → P1 → P2 → P3 → P4
```

- **PX-SK-04 使用合规 profile**：法规条文→技术映射

### Profile F：接管烂尾项目

> **场景特征**：外包走了，代码没文档，要接着做

```
PX-SK-01（含可运行性预检）+ PX-SK-02 + PX-SK-03（并行）
    → PX-SK-04
    → PX-SK-05
    → PX-SK-06
    → P1 → P2 → P3 → P4
```

- **PX-SK-01 包含可运行性预检**：先确认代码能不能跑起来
- 类似 Profile B，但 PX-SK-03 的业务梳理更加依赖代码逆向推断

### Profile G：模块抽取 / 平台化

> **场景特征**：把支付/用户中心从业务系统中抽出来做公共服务

```
PX-SK-01 + PX-SK-02 + PX-SK-03（并行）
    → PX-SK-04（耦合度 profile）
    → PX-SK-05
    → PX-SK-06
    → P1 → P2 → P3 → P4
```

- **PX-SK-04 使用耦合度 profile**：重点分析调用链、共享数据库表、循环依赖

### Profile H：多系统整合

> **场景特征**：收购/合并后多套系统要统一

```
(PX-SK-01 + PX-SK-02 + PX-SK-03) × N（对每个系统并行）
    → PX-SK-10
    → PX-SK-05
    → PX-SK-06
    → P1 → P2 → P3 → P4
```

- **PX-SK-10 负责跨系统实体映射和冲突检测**

### Profile I：性能专项优化

> **场景特征**：不改功能，但要扛住 10x 流量

```
PX-SK-01 + PX-SK-02（并行）
    → PX-SK-04（性能 profile）
    → PX-SK-08
    → PX-SK-07
    → P3 → P4
```

- **PX-SK-04 使用性能 profile**：包括压测基线采集、瓶颈定位

## 七、对 P1-P4 的影响与扩展需求

### 7.1 P1 需要扩展的能力

PhaseX Profile B/C/E/F/G/H 的出口是 P1，但 P1 当前只接受"原始需求描述"。需要扩展：

| 扩展点 | 说明 |
|---|---|
| **输入扩展** | 除了"原始需求描述"，还要接受 **基线约束包**（哪些不能改、哪些 API 保持兼容、哪些表结构冻结） |
| **需求类型扩展** | 从纯"新建需求"扩展到"变更需求"——每条需求标注影响范围（新增 / 修改 / 废弃）和约束来源 |
| **影响范围标注** | 每条需求需标注：涉及哪些现有模块、是否破坏现有接口、是否需要数据迁移 |
| **约束透传** | 基线约束需透传到 P2，使架构设计在 AS-IS 基础上演进而非从白纸开始 |

### 7.2 P2 需要感知的变化

| 扩展点 | 说明 |
|---|---|
| **架构演进模式** | 从"从零设计"扩展到"在已有架构基础上演进"——需要感知现有模块、现有接口、不变量约束 |
| **约束输入** | 接受来自 PhaseX 的"不变量声明"（哪些接口/Schema/模块不能改） |

### 7.3 P3 的变化

| 扩展点 | 说明 |
|---|---|
| **重构模式** | 从"全新实现"扩展到"在已有代码上改造"——需要处理遗留代码、分支策略、渐进式替换 |
| **迁移执行** | 接受 PX-SK-09 的迁移计划作为执行指导 |

### 7.4 P4 的变化

| 扩展点 | 说明 |
|---|---|
| **回归验证** | 除了新功能验证，增加回归验证——确认改造没有破坏原有行为 |
| **迁移验证** | 对技术栈迁移场景，增加数据一致性校验、性能对比验证 |

## 十、开放问题（评审讨论点）

## 十一、当前 v0 判断

- PhaseX 是 P1-P4 体系的必要补充，解决了"棕地项目如何接入"的核心问题
- 不采用线性 Stage 模型，改用 **Skill 池 + 依赖图 + Profile 推荐** 的灵活编排模型，这是对现有框架的一次结构性创新
- 10 个 Skill 覆盖了 12 种已识别场景，验证矩阵全部通过
- 建议分三波落地，先跑通 Profile A（纯技术重构）和 Profile C（局部需求变更）
- P1 需要同步扩展"约束输入"能力，这是最关键的下游改造点
- 当前草案足够做整体评审和分工，但每个 Skill 的 SOP、模板、引用素材尚需后续细化
