# v1.3 Autopilot / Strict-Proof 词汇边界（v0.1）

状态：`WO-119A.03` 本地验证记录
日期：2026-05-03
所属工单：`WO-119A.03`
控制边界：`docs/governance/v1.3-product-flow-control-boundary-v0.1.md`

## 目的

本文澄清 `v1.3.1` 中 `autopilot`、`guided` 和 `strict-proof` 的词汇边界。

它不创建新的交付模式，不改变已接受的 `WO-119A.02` 控制边界，不重跑 proof snapshots，也不批准生产发布。

目标是避免后续 docs、skills、tests 或 release-package 语言误把 `strict-proof` 重新写成与 `autopilot` 平级的主线模式。

## 规范关系

已接受的 `v1.3.1` 关系保持为：

```text
pre-P1 source input -> autopilot mainline -> strict-proof gate -> capped delivery claim
```

解释 v1.3.1 product flow 时应使用这个关系。

不要把流程描述成：

```text
autopilot / guided / strict-proof
```

这种三列说法会让 proof machinery 看起来像平级 delivery path，并可能把 WFF 拉回 script-led artifact closure。

## 词汇定义

### Autopilot Mainline

`autopilot mainline` 指 source truth 足够后的默认 P1-P4 Agent / Skills-led 生命周期路径。

它表示：

- Agent / Skills 在 phase boundaries 内拥有 product、business、architecture、implementation、test-meaning 和 review judgment。
- Workflow 保持 order、entry / exit contracts、convergence 和 rerun boundaries。
- Scripts 提供 mechanical checks、diagnostics、regression safety 和 proof support。
- Evidence 封顶 formal claims。

它不表示：

- 没有 review breakpoints 的盲目执行；
- 跳过 evidence；
- 隐藏 uncertainty；
- 让 scripts 决定 value；
- 移除 human oversight opportunities。

### Guided

`guided` 不是 `v1.3.1` delivery mode。

如果后续文字保留这个词，它应表示围绕 Agent-led mainline 的校准、采用、调试或 human oversight 行为。

可接受用法：

- guided onboarding into the WFF lifecycle；
- guided debugging of a failed run；
- guided calibration of source truth or review expectations；
- phase breakpoints 上的 human-guided oversight。

避免把 `guided` 用作：

- 与 `autopilot` 平级的模式；
- 默认 human-led collaboration model；
- Agent / Skills judgment 的替代品；
- 削弱 strict-proof 或 claim ceilings 的理由。

### Strict-Proof Gate

`strict-proof gate` 指 proof 和 claim-ceiling validation。

它可以检查的 evidence 包括：

- source-to-claim links；
- contract / test / code traces；
- install、build、start、runtime、migration、persistence 或 test evidence，如果可用；
- output-contract validity；
- package inclusion；
- formal delivery-gate state。

它不拥有：

- product goal correctness；
- business value validity；
- architecture tradeoff quality；
- implementation value；
- test meaning；
- upstream remediation。

如果 strict-proof 失败，它记录 failed evidence、claim ceiling、return reason 和 minimum rerun boundary，然后把问题路由给 owning phase。

## Release-Facing 用词规则

使用：

- `autopilot mainline`；
- `strict-proof gate`；
- `claim ceiling`；
- `phase review breakpoint`；
- `human oversight opportunity`；
- `script support layer`。

避免：

- 暗示平级 delivery path 的 `strict-proof mode`；
- 把 `autopilot / guided / strict-proof` 写成 mode table；
- `proof-driven mainline`；
- `script-led closure`；
- `gate-owned remediation`；
- 把 `human-led continuous delivery` 作为默认模型。

如果旧表面必须保留 `strict-proof mode` 作为历史用词，附近文字必须澄清 runtime authority 是 gate / claim-ceiling mechanism，而不是 peer mainline controller。

## Change Surface Rule

本文在 `WO-119A.03` 中属于 central-only。

只有当某个 phase、skill、template 或 reference-package surface 满足以下情况时，才把短规则镜像进去：

- 呈现 delivery modes；
- 告诉 agent 什么时候运行 strict proof；
- 描述 phase completion authority；
- 给 scripts 或 gates 赋予 product、architecture、implementation 或 review judgment 的 semantic authority。

不要为了重复这组词汇而编辑 P1-P4 skills。

## 本地验证问题

如果本文满足以下条件，则 `WO-119A.03` 完成：

- 保持已接受的 `WO-119A.02` 控制边界；
- 防止 `strict-proof` 被描述为与 `autopilot` 平级的 mainline mode；
- 不把 `guided` 放进正式 delivery-mode model；
- 保持 phase review breakpoints 作为 oversight opportunities；
- 要求 strict-proof failure 路由回 upstream，而不是修复 upstream truth；
- 不开启 `v1.3.5` P1 source-truth admission work。
