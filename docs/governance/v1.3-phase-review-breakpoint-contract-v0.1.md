## Purpose

This document defines the `v1.3.1` phase review breakpoint contract for P1-P4.

The contract adds explicit human oversight opportunities at phase completion while preserving the accepted Agent / Skills-led autopilot mainline. It does not make human-led continuous collaboration the default delivery model.

## Control Principle

Each P phase must actively present a review opportunity at completion.

The breakpoint is a review and routing surface. It is not a stage artifact editor, not a proof gate, and not a new phase.

The phase may continue only after the breakpoint records one of these states.

Human-facing review actions should be shown with Chinese labels. Stable English keys may be kept in structured records for scripts, tests, or cross-language references:

- `批准` (`approved`): the reviewer accepts the phase output within the stated evidence ceiling.
- `带保留项批准` (`approved-with-review-bound-items`): the reviewer accepts downstream continuation while preserving named open items.
- `要求修改` (`changes-requested`): the reviewer requests bounded changes inside the same phase.
- `要求返回` (`return-requested`): the reviewer sends the issue to an upstream or current owning phase with a rerun boundary.
- `提供干预输入` (`intervention-supplied`): the reviewer supplies new source truth, constraints, artifacts, or decisions; the phase must treat them as new input and rerun or remediate rather than patching final artifacts silently.
- `明确跳过审核` (`explicitly-skipped`): the reviewer declines review; the phase may continue only with the skip, reviewer identity if available, reason, and unchanged claim ceiling recorded.

## Required Breakpoint Payload

At phase completion, the phase should present:

- phase output root and primary artifact list;
- phase summary and intended downstream consumer;
- value-bearing judgment for that phase;
- open `review-bound` items and their downstream effect;
- evidence basis and missing evidence;
- current formal claim ceiling;
- recommended next step;
- minimum rerun or return boundary if the reviewer rejects, intervenes, or requests return.

The payload may be expressed in the phase execution report, handoff package, closeout summary, or a small review-breakpoint note. A new artifact is not required when the existing phase output already carries the required fields.

## Non-Mutation Rule

The breakpoint must not silently mutate generated stage artifacts, proof outputs, or upstream truth.

If review changes product truth, architecture truth, implementation behavior, or validation evidence, the change must be recorded as new input and routed through the owning phase.

Allowed actions at the breakpoint:

- record reviewer decision and notes;
- record new input supplied by the reviewer;
- route remediation to the owning phase;
- cap or preserve claims;
- explicitly skip review with a reason.

Disallowed actions at the breakpoint:

- editing retained generated proof snapshots by hand;
- rewriting upstream artifacts without rerun / remediation;
- turning missing evidence into pass wording;
- treating reviewer optimism as evidence;
- claiming production approval, online UAT, go-live, real owner sign-off, rollback readiness, monitoring handover, or production risk acceptance without external evidence.

## Phase-Specific Intent

| Phase | Breakpoint asks | Downstream effect |
| --- | --- | --- |
| P1 | Is the product / business / user / acceptance truth valuable and clear enough for architecture? What remains review-bound? | Continue to P2, return to P1 deepening, or defer to `v1.3.5` source-truth admission when the gap belongs before P1. |
| P2 | Is the architecture / delivery path / constraint response / implementation-facing handoff good enough for implementation? | Continue to P3, run bounded P2 remediation, or return to P1 if missing product truth forces design invention. |
| P3 | Is the delivery runnable and meaningful enough for validation within the evidence ceiling? | Continue to P4, run bounded P3 remediation, or return to P2/P1 for upstream truth gaps. |
| P4 | Is validation closure honest, are risks visible, and is the claim ceiling correct? | Close testing-validation, preserve review-bound items, route remediation, or optionally enter Stage-04 only when explicitly requested. |

## Human Oversight Boundary

The breakpoint gives the human or reviewer a decision point.

It does not require the human to co-author every stage, continuously steer the Agent, or replace Agent / Skills judgment.

Autopilot remains the mainline; the breakpoint is an oversight and intervention opportunity.

## Runtime Mirror Rule

The minimum runtime-facing mirror belongs in the official phase entry skills:

Only mirror into stage SOPs, templates, or reference packages if a later child work item proves that the phase entry skill is not enough for runtime execution.

## Acceptance Question

Accept `WO-119A.04` if this contract correctly defines phase review breakpoints as:

- required phase-completion review opportunities for P1-P4;
- oversight and routing surfaces, not continuous human-led collaboration;
- non-mutating decision points that preserve generated artifacts and upstream truth;
- places where Chinese human-facing action labels such as `批准`, `要求修改`, `提供干预输入`, `要求返回`, and `明确跳过审核` are recorded while stable protocol keys remain available for structured records;
- claim-ceiling preservers, not proof or production-approval substitutes;
- narrow enough that only the four official phase entry skills need minimum runtime mirrors for now.
