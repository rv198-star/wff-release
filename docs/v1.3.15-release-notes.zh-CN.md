## v1.3.15 改了什么

v1.3.15 收口 reader translation restart 线。它保持 P1/P2 canonical source-of-truth 文档为英文，再从 canonical English source 生成供人工审核的本地化 reader 文档。

本版本声明范围刻意保持收窄：

- localized reader first-pass quality 显著提升并达到可用；
- P1/P2 reader 生成保留 canonical English source truth 和 reader packaging 边界；
- 当前 `main` 通过 fresh release-packaged GEO + PetClinic P1-P4 strict-runtime 快照；
- 不声明 stable `90+` polished long-document localized-reader quality。

## LLM API 前提

v1.3.15 引入了一个真实运行前提：要生成可供人工审核的 localized reader 翻译文档，必须配置 OpenAI-compatible LLM API。

翻译运行器会读取：

- `OPENAI_API_KEY` 作为 API key；
- `OPENAI_BASE_URL` 作为 OpenAI-compatible endpoint，除非 `config/generated-output-policy.json` 已提供 `reader_translation.api_base_url`；
- `WFF_TRANSLATION_MODEL` 作为模型覆盖，除非 `config/generated-output-policy.json` 已提供 `reader_translation.model`。

当前入库默认策略使用 `deepseek-v4-flash` 和 `https://api.deepseek.com/v1`，但仍通过 `OPENAI_API_KEY` 读取密钥。

这不是全局生命周期 blocker。如果没有提供 API，WFF 仍可保留 canonical English source truth，运行正常 P1-P4 生命周期，并生成属于既有确定性管线的非 LLM 本地化 audit surfaces。无法完成的是 v1.3.15 新增的、由 LLM 生成并用于人工审核的 localized reader translation evidence。

当翻译运行器在没有可用 API 的情况下被调用时，它会报告 `translation-failed` 并返回非零退出码。凡是依赖 localized reader translation 的发布或评审声明，都应标记为缺少 reader evidence / translation required，不能静默替换成伪本地化生成文档。

## 后续规划

v1.3.15 暂时保留旧的确定性 `.zh-CN.md` audit mirror，因为 P1 report、skill-family audit、runtime snapshot evidence 和测试仍把它作为兼容表面引用。

但该 mirror 不应继续作为长期 reader 机制。后续规划候选已记录在 [`superpowers/plans/2026-05-17-v1316-reader-artifact-boundary-cleanup.zh-CN.md`](superpowers/plans/2026-05-17-v1316-reader-artifact-boundary-cleanup.zh-CN.md)。目标是让 canonical English artifacts 成为唯一主线 source truth，让 LLM localized reader artifacts 成为可供人工审核的翻译表面，并降级或移除旧 `.zh-CN.md` mirror。

即使完成替代后，localized reader translation 也仍应是 optional evidence。缺少 LLM API 或缺少 localized reader 输出，只能封顶 reader/localization 相关声明，不能阻断 P1-P4 生命周期。

## 声明边界

v1.3.15 只证明开发 / 预生产生命周期内的生成与验证闭环。

它不声明真实线上 UAT、生产部署、生产发布批准、真实负责人签字、预算批准、上线授权、回滚就绪、生产监控交接、生产风险承接、stable `90+` localized-reader polish，也不声明生成产物完美无缺。
