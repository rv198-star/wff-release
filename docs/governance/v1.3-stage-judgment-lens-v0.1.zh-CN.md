## 目的

本文定义 P1-P4 的 `v1.3.1` Stage Judgment Lens。

该 lens 是内部 Agent / Skills judgment aid。它帮助每个 phase 判断 artifact 是否有实际价值、可审查、并能被下游 phase 使用。

它不是公开 named-agent layer，不是 role-play output，不是新的用户可见 persona，也不是往每个 artifact 加 checklist-only 文本的许可。

## 核心规则

每个 phase 中，Agent judgment 必须回答该 phase 真正拥有的 value question。

Workflow 可以确认 order 和 artifact presence。Scripts 可以确认 mechanical facts。Evidence 可以封顶 claims。

这些都不能替代 phase 自己的 judgment lens。

## Lens Table

| Phase | Judgment lens | Agent 必须判断什么 | 什么本身不算充分 |
| --- | --- | --- | --- |
| P1 | product / business / user / acceptance value | product wedge 是否有价值、有 source grounding、足够具体，并值得进入 architecture？pain strength、status quo、business value、user value 和 acceptance meaning 是否足够清楚，能供 P2 使用？ | PRD section completeness、trace existence、generic business language，或 clean convergence gates 本身 |
| P2 | architecture tradeoff / delivery path / constraint response | architecture 是否把 P1 truth 转译成可实现的 delivery path，并提供有用边界、明确 tradeoffs、risk control 和 implementation leverage？ | diagram count、ADR volume、service names、wrapper pass，或没有 delivery economics 的 architecture elegance |
| P3 | runnable delivery / core path / implementation value / test meaning | implementation 是否在 selected core path 上产生 runnable value，证明 meaningful behavior，并让下游 developer 或 Agent 能继续？ | code volume、endpoint existence、unit-count green、generated runtime green，或没有 runtime 与 assertion meaning 的 script pass |
| P4 | validation risk / evidence sufficiency / claim ceiling / return routing | validation 是否诚实暴露 risk、消费可用 evidence、封顶 formal claims，并把 defects 路由给 owning phase，而不是修复 upstream truth？ | acceptance template completion、proof report existence、Stage-03 pass wording，或没有 supplied external evidence 的 release-shaped language |

## 如何使用 Lens

在 phase closure、review breakpoints、rerun triage 和 focused remediation 中使用该 lens。

每个 phase：

1. 识别 phase-owned value question。
2. 对照 source truth 和 downstream consumer need 检查 artifact。
3. 说明已经存在的最强 positive value。
4. 说明仍未解决的最弱 value risk。
5. 判断下一步是 continue、bounded remediation、upstream return、review-bound carryover，还是 blocked。

该 lens 应产生更好的 judgment，而不是更长的 artifacts。

## Anti-Role-Play Boundary

不要把该 lens 暴露成四个公开 agents 或 personas。

避免：

- `P1 Product Agent`；
- `P2 Architect Agent`；
- `P3 Implementation Agent`；
- `P4 Validation Agent`；
- role-play scripts；
- decorative reviewer voices；
- 只是为了显示调用了 lens 而存在的 output sections。

允许：

- 简洁的 phase-closure judgment；
- 绑定到 phase-owned value question 的 reviewer notes；
- 基于 lens 的 rerun / return routing；
- 当 lens 发现真实 value loss 时进行 targeted artifact improvement。

## Evidence Boundary

lens 可以判断 value 强或弱，但 formal claims 仍需要 evidence。

如果 lens 发现 product、design、implementation 或 validation value 强，但 supporting evidence 不足，应把 judgment 保留为 review-bound 并封顶 claim。

如果 scripts pass 但 lens 发现 value 薄弱，该 phase 不得把 artifact 提升为 substantively complete。

## Change Surface Rule

`WO-119A.05` 是 central-only。

不要为了重复这张表而编辑 P1-P4 skills。

只有后续工作证明某个 phase 无法执行 value-bearing closure，才把该 lens targeted mirror 到对应 phase skill。
