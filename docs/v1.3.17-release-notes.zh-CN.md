## v1.3.17 带来了什么变化

v1.3.17 取代了 v1.3.16，成为本地化读者证据版本。

本次发布主题是：

- 将英文 P1/P2/P3/P4 规范工件作为生命周期的单一事实来源；
- 将本地化读者翻译作为非阻塞证据通道运行；
- 当配置了兼容 OpenAI 的 LLM API 时，为 P1 PRD、P2 ESP 和 P3 action cards 生成中文读者工件；
- 读者翻译不可用时，保留翻译进度和清单证据，但不阻塞规范的 P1-P4 生命周期；
- 发布时随包提供已固定的 Mindthus 运行时依赖，使正常启动不再依赖首次运行时的 GitHub 克隆。

## Reader lane 范围澄清

v1.3.17 历史中可能出现 `relax` 这类听起来较宽的 integrity / structure check 表述。在本发布线上，这类调整只限 reader lane：容忍度变化只作用于本地化读者翻译的结构 / immutable-token 检查，不放宽 canonical P1-P4 主线 gate、phase verdict、strict-runtime evidence 或 claim ceiling。

## LLM API 边界

可供人工复核的本地化读者翻译需要兼容 OpenAI 的 LLM API。

读者翻译运行器会读取 `config/generated-output-policy.json` 中的 `reader_translation`，以及已配置的 API key 环境变量。如果密钥存放在本地配置中，它会被注入运行时环境，但不会被记录日志。

这个 API 不是 WFF 生命周期的全局依赖。如果没有配置，WFF 仍然可以正常运行 P1-P4。在没有 API key 的非交互环境中，读者预检会跳过读者翻译，并返回 `no-api-key-noninteractive`，而不是阻塞 P1-P4。

## 运营成本

被接受的 R11 证据表明，完整的发布候选验证并不是一次短暂冒烟：

- P1-P4 主线：约 `8m49s`；
- 主线之后完成完整读者通道：约 `66-79m`，取决于案例；
- 人工可观察的从启动到进入复核窗口：约 `1h30m`。

如果发布决策需要完整读者证据和人工 Review，请至少预留 `90min`。
