# v1.2 P3 Behavior Card Contract Boundary

## One-Line Rule

Behavior cards are the P3 enhanced interface contract: implementation code and test code must consume the same card-derived evidence, error, state, and source obligations.

## Source Order

1. P1 business/value truth
2. P2 operation/source/risk/action-card bridge
3. P3 behavior card
4. P3 implementation and tests

P3 may refine executable details, but it must not invent missing P1/P2 business truth or create a P3-local trace authority.

## Test Boundary

- Unit service tests prove behavior-card evidence, service response projection, and service error semantics.
- Repository unit tests prove behavior-card step methods and persistence/effect boundaries.
- Contract tests prove HTTP contract, payload enrichment, error forcing, and response shape.
- Scenario tests prove operation chains, context propagation, and business scenario continuity.
- Runtime foundation tests prove generated runtime store helpers.

A direct service unit test must not pretend to prove generated runtime store persistence. Runtime store helpers belong in runtime/foundation tests or started-service tests.

## TVG Role

Thinking Value-Gain is a value-preservation hook. It may check whether behavior-card value was carried into code/tests, but it must not invent P1/P2 requirements or replace behavior cards.

For final v1.2.3 review, P3 should be evaluated with both:

- behavior cards plus current TVG value-preservation hook
- behavior cards with P3 TVG disabled

If the no-TVG path is comparable, keep behavior cards as the P3 mainline and demote TVG to a diagnostic or expert-lane check.

## Runtime Honesty

P3 may claim only the evidence that actually ran:

- script/generator regression
- typecheck/build
- contract/unit/scenario/schema runtime tests
- persistence/init/restore proof when persistence is claimed
- manual code review when business fullness is the question

Passing scripts alone do not prove business implementation depth.
