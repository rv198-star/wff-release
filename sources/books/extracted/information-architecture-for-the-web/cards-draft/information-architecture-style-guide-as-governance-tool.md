# IA Style Guide：用“why+how”文档防止架构漂移

- 类型：治理
- 适用阶段：设计收敛
- 来源：`sources/books/original/phase-02-design-architecture/Information_Architecture_For_The_Web.md`（Information Architecture Style Guides）
- 摘要：信息环境会持续变化。IA style guide 记录组织方式与扩展规则，为维护者提供“做与不做”的依据与保护伞。

## 输入条件
- 上线后内容新增导致 taxonomy/nav/label 被随意破坏（architectural drift）

## 关键步骤
1. Why：记录 mission/vision、受众与目标、信息需求假设、内容政策（包含/不包含）。
2. How：标准（必须做）、指南（建议做）、维护流程（词表新增/导航扩展）。
3. 附上可复用交付物：sitemap/wireframe/词表/元数据矩阵，并提供解释。
4. 设计 style guide 本身：面向不同读者（作者/设计/工程/运营）分层呈现。

## 输出结果
- 一份可执行的 IA 治理手册，能回答“为什么不能把我部门内容置顶”。

## 适用边界
- style guide 不是一次性文档，需要与 administration 阶段同步更新。

## 常见错误
- 只写原则不写操作流程；或只写规则不写 rationale

## 关联模板
- IA style guide 章节大纲（why/how/artifacts/procedures）

## 下游使用建议
- Stage-2：把 style guide 当作交付物候选（尤其多团队内容平台）。
