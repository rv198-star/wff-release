# 通过探索建立共识

- 类型：方法
- 适用阶段：需求分析 / 需求澄清 / 工作坊协作
- 来源：`sources/books/original/用户故事地图.md`（第14章）
- 摘要：在复杂需求场景中，探索的核心目标不是立刻产出软件，而是通过围绕活动、讨论和工件的协作过程建立团队共识，为后续需求定义、切片和设计决策提供稳定基础。

## 输入条件
- 存在需要澄清的业务机会、用户问题或方案方向
- 多角色参与者需要对问题与目标形成共享理解

## 关键步骤
1. 明确探索对象：机会、场景、目标或待验证方向。
2. 通过讨论与可视化工件，把不同角色的理解显式化。
3. 在探索过程中不断发现分歧、补足缺失信息并修正误解。
4. 形成可供后续需求/设计阶段使用的共识基础，而不是仓促定稿。

## 输出结果
- 共享的需求理解
- 更稳定的故事地图或需求结构草案
- 为下游阶段提供一致的上下文基线

## 适用边界
- 适用于存在多角色、多观点、需求理解尚未稳定的场景
- 不适用于已经具备稳定结构化输入、仅需机械转写的简单任务

## 常见错误
- 把探索误当成直接开发
- 把探索活动做成汇报会，而不是共识构建过程
- 在共识未建立时就过早固化需求或设计

## 关联模板
- 需求分析工作坊记录
- 故事地图草案
- 机会分析记录

## 下游使用建议
- 可被未来“需求分析 Skill”引用，作为多人协作澄清需求的标准方法
- 可进入阶段 gate：若关键参与方尚未形成共识，不进入设计阶段
- 可作为工作坊类 Skill 的核心方法卡
