## Purpose

This document defines the `v1.3.1` Stage Judgment Lens for P1-P4.

The lens is an internal Agent / Skills judgment aid. It helps each phase ask whether the artifact is practically valuable, reviewable, and usable by the next phase.

It is not a public named-agent layer, not role-play output, not a new user-facing persona, and not permission to add checklist-only prose to every artifact.

## Core Rule

At each phase, Agent judgment must answer the value question that the phase actually owns.

Workflow may confirm order and artifact presence. Scripts may confirm mechanical facts. Evidence may cap claims.

None of those replaces the phase's judgment lens.

## Lens Table

| Phase | Judgment lens | What the Agent must decide | What does not count by itself |
| --- | --- | --- | --- |
| P1 | product / business / user / acceptance value | Is the product wedge valuable, source-grounded, specific enough, and worth sending to architecture? Are pain strength, status quo, business value, user value, and acceptance meaning clear enough for P2? | PRD section completeness, trace existence, generic business language, or clean convergence gates alone |
| P2 | architecture tradeoff / delivery path / constraint response | Does the architecture translate P1 truth into a realizable delivery path with useful boundaries, explicit tradeoffs, risk control, and implementation leverage? | Diagram count, ADR volume, service names, wrapper pass, or architecture elegance without delivery economics |
| P3 | runnable delivery / core path / implementation value / test meaning | Does the implementation produce runnable value on the selected core path, prove meaningful behavior, and leave a downstream developer or Agent able to continue? | Code volume, endpoint existence, unit-count green, generated runtime green, or script pass without runtime and assertion meaning |
| P4 | validation risk / evidence sufficiency / claim ceiling / return routing | Does validation honestly expose risk, consume available evidence, cap formal claims, and route defects to the owning phase without repairing upstream truth? | Acceptance template completion, proof report existence, Stage-03 pass wording, or release-shaped language without supplied external evidence |

## How To Use The Lens

Use the lens during phase closure, review breakpoints, rerun triage, and focused remediation.

For each phase:

1. Identify the phase-owned value question.
2. Compare the artifact against the source truth and downstream consumer need.
3. Name the strongest positive value already present.
4. Name the weakest unresolved value risk.
5. Decide whether the next action is continue, bounded remediation, upstream return, review-bound carryover, or blocked.

The lens should produce better judgment, not longer artifacts.

## Anti-Role-Play Boundary

Do not expose the lens as four public agents or personas.

Avoid:

- `P1 Product Agent`;
- `P2 Architect Agent`;
- `P3 Implementation Agent`;
- `P4 Validation Agent`;
- role-play scripts;
- decorative reviewer voices;
- output sections that exist only to show that the lens was invoked.

Allowed:

- concise phase-closure judgment;
- reviewer notes tied to the phase-owned value question;
- rerun / return routing based on the lens;
- targeted artifact improvement when the lens finds real value loss.

## Evidence Boundary

The lens may identify that value is strong or weak, but formal claims still need evidence.

If the lens finds strong product, design, implementation, or validation value without supporting evidence, preserve the judgment as review-bound and cap the claim.

If scripts pass but the lens finds weak value, the phase must not promote the artifact as substantively complete.
