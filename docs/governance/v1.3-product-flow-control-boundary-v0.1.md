## Purpose

This document defines the central control boundary for `v1.3.1 Agentic mainline rebalance`.

It is the release-facing rule that should guide later skill, phase, template, and test updates. It does not by itself change generated proof outputs, rerun retained release cases, or approve production release.

## Version Goal

`v1.3.1` makes WFF's P1-P4 default lifecycle mainline explicitly Agent / Skills-led, with workflow preserving order, scripts preserving mechanical safety, evidence preserving claim honesty, and strict-proof acting as a gate rather than the mainline controller.

## Core Control Relationship

The v1.3 control relationship is:

```text
pre-P1 source input -> autopilot mainline -> strict-proof gate -> capped delivery claim
```

Meaning:

- `pre-P1 source input` provides normalized source truth before formal P1 execution.
- `autopilot mainline` is the default P1-P4 Agent / Skills-led lifecycle path once source truth is sufficient.
- `strict-proof gate` validates evidence sufficiency and caps formal claims; it does not own product, architecture, implementation, or remediation judgment.
- `capped delivery claim` is the strongest claim supported by the weakest available environment and evidence bridge.

`strict-proof` must not be described as a peer runtime mode beside `autopilot`. It is a proof and claim-ceiling gate.

`guided` should not be promoted as a v1.3.1 delivery mode. If the term is retained in later docs, it should mean calibration, adoption, debugging, or human oversight behavior around the Agent-led mainline.

## WAE Mapping

The v1.3.1 control boundary is:

```text
Workflow shell + Agentic core + Evidence bridge
```

### Workflow Shell

Workflow should control:

- P1 -> P2 -> P3 -> P4 order.
- Phase entry and exit boundaries.
- Required artifact families.
- Phase review breakpoint presentation.
- State recording and rerun boundaries.
- Deterministic transforms and packaging.
- Repeatable verification command surfaces.

Workflow should not decide business truth, architecture quality, implementation value, or whether product value is real.

### Agentic Core

Agent / Skills should control:

- Product goal judgment.
- Business value and user value judgment.
- Scenario, acceptance, and anti-demo judgment.
- Architecture tradeoffs and delivery economics.
- Implementation value and core-path usability.
- Test meaning and assertion quality.
- Review finding classification and upstream routing.
- Whether value-bearing artifact closure is substantive rather than decorative.

Agentic judgment must be bounded by phase responsibilities and evidence ceilings. It must not invent missing upstream truth.

### Evidence Bridge

Evidence should control:

- Claim-to-source links.
- Contract -> test -> code traces.
- Runtime proof.
- Formal claim ceilings.
- Delivery boundary statements.
- Strict-proof pass / return evidence.

Evidence is not the mainline goal. Evidence makes claims reviewable and honestly capped.

### Script Support Layer

Scripts may control or prove:

- File existence.
- Schema validity.
- Mechanical contract compatibility.
- Install / build / start / test execution where commands are available.
- Package inclusion.
- Evidence aggregation.
- Strict-proof reports.
- Formal claim support or failure.

Scripts must not decide:

- Product goal correctness.
- Business value validity.
- User pain strength.
- Architecture tradeoff quality.
- Implementation value.
- Test meaning.
- Whether further investment is warranted.

Script pass is a support signal, not product success.

## Phase Mainline Responsibilities

P1-P4 should be understood as an Agent / Skills-led product-development lifecycle, not an artifact factory.

| Phase | Agent / Skills judgment owns | Workflow / scripts support |
| --- | --- | --- |
| P1 | product value, business value, user value, scenario leverage, acceptance meaning, source truth sufficiency | input structure, PRD assembly, convergence reports, mechanical gates, trace preservation |
| P2 | architecture tradeoffs, delivery path, constraints, risk, evolution, implementation leverage | stage order, ESP structure, trace absorption, schema / contract checks, realizability evidence |
| P3 | runnable delivery value, core path usability, implementation correctness, test meaning, handoff value | contract/test order, build/test/runtime smoke, evidence reports, delivery gate records |
| P4 | validation risk, evidence sufficiency, claim ceiling, return routing, review-bound carryover | acceptance catalogs, evidence aggregation, output-contract checks, strict-proof reports |

The table is a judgment boundary. It is not permission to mechanically edit every P1-P4 file.

## Phase Review Breakpoints

`v1.3.1` requires phase review breakpoints as a control principle, but the detailed breakpoint contract is owned by `WO-119A.04`.

At this boundary level:

- each P phase should be able to present a review opportunity at completion;
- human review may approve, reject, intervene, request return, or explicitly skip;
- the breakpoint does not itself mutate generated artifacts;
- the breakpoint does not make human continuous collaboration the default delivery model.

## Strict-Proof Failure Routing

When strict-proof fails:

- record the failed evidence, claim ceiling, and return reason;
- route the problem to the owning phase;
- state the minimum rerun boundary;
- do not let the proof gate repair upstream product truth, architecture truth, implementation truth, or evidence gaps by itself.

Default ownership:

- source truth / product value gap -> P1 or pre-P1 admission;
- architecture / contract / design gap -> P2;
- implementation / runtime evidence gap -> P3;
- validation consumption / closure judgment defect -> P4;
- external production / owner-signoff gap -> remain outside WFF default authority unless supplied as external evidence.

## Non-Goals

`v1.3.1` does not:

- redesign `wff-req-chat`;
- define the final `P1 source input packet`;
- implement fresh-context P1 source brief review;
- implement `wff-help`;
- add named agents as a public role layer;
- add customization overlays;
- add `wff-project-context`;
- rerun retained v1.2 proof snapshots;
- manually edit generated proof outputs;
- claim real online UAT, production release approval, owner sign-off, go-live authorization, rollback readiness, monitoring handover, or production risk ownership.

## Mirror Rule

This document should be the central source for v1.3.1 control-boundary wording.

Later child work should mirror only the minimum runtime-affecting rule into specific phase, skill, template, or reference-package surfaces.

Do not copy this whole document into P1/P2/P3/P4 skills.

Do not edit phase surfaces just to show coverage.

If a phase skill is edited, the edit must name the runtime behavior it changes or clarifies.

## Acceptance Question

Accept `WO-119A.02` if this document correctly freezes the v1.3.1 control boundary:

- Agent / Skills are the mainline judgment layer.
- Workflow controls order and convergence.
- Scripts provide mechanical safety and proof support without semantic authority.
- Evidence caps claims without becoming the mainline goal.
- Strict-proof is a gate, not a peer mainline mode.
- Human review breakpoints exist as oversight opportunities, not continuous human-led delivery.
- v1.3.5 P1 source-truth admission work remains out of scope.
