# v1.2 Script Surface Classification（v0.1）

## 这份文档是做什么的

这份文档把仓库当前脚本面做第一次 `v1.2` 分类冻结。

它的目的不是为了“删脚本而删脚本”。
它的目的是：

- 找出哪些脚本确实在保秩序、保 gate、保证据
- 找出哪些脚本已经退出主线，应该降级为 optional lane
- 找出哪些脚本只是把一个逻辑拆得过碎，应该 merge
- 找出哪些脚本已经不值得继续保留

这份文档首先服务 `v1.2` 的实际改造，不是历史回顾。

## 分类规则

### `must keep`

必须保留为 `v1.2` 主线或必要底座的一部分。

典型特征：

- deterministic transform
- hard gate
- execution/report closure
- evidence emission / aggregation
- downstream handoff truth

### `optional`

能力有价值，但不应继续压在默认主线里。

典型特征：

- extra hardening
- advanced review
- fallback lane
- multi-worker / dispatch lane
- richer side proof

### `merge`

功能本身仍需要，但不值得继续维持为独立脚本。

典型特征：

- wrapper 套 wrapper
- 同一条链路被拆成过多中间脚本
- 语义和输入输出高度耦合，完全可以并回更少的执行面

### `retire`

当前 `v1.2` 方向下不再值得继续保留的独立脚本。

典型特征：

- 主要在维护过厚控制面
- 作用已被更上层/更直接的脚本吸收
- 保留只会继续放大理解成本

### 补充语义：顶层 script surface vs internal support module

这份文档优先分类的是**顶层脚本面**，不是把所有 support helper 都当成独立脚本对待。

因此：

- 某个顶层脚本被标成 `retire`
  - 表示它不再应作为独立入口或独立认知面存在
- 如果相应能力被下沉到 `scripts/phase3/`
  - 那是 `merge/downshift` 后保留为 internal helper
  - 不是“能力彻底删除”

短规则：

> **退场的是独立入口，不一定是底层能力本身。**

## 第一波结论

### `P1`

当前判断：

- 应保留主线 runner、深度输出生成、PRD 组装/收敛、关键 gates、execution report
- 应把若干 gate 合并，减少“一个现象一个 gate”的碎片化
- `v1.2` 的重点不是删光 `P1` 脚本，而是把 thinking runtime 从“存在但不主导”提升为“真正主导”

### `P2`

当前判断：

- 脚本面不算最大问题
- 应控制 runner 数量和附属 evidence 面，不继续变重
- 重点是把 checklist-first + focused review 固化，而不是继续扩脚本

### `P3`

当前判断：

- 这是第一波最需要真实收口的区域
- 应保留 backend-first 主线所需的 compile / scaffold / verify / gate / report
- fallback UI、dispatch、worker packet、wave planning、更强 review/hardening 应退出默认主线
- 多个 dispatch/work-packet 脚本已经明显适合 merge 或 retire

### `P4`

当前判断：

- 主线仍然清楚
- 关键是把 `validation closure` 和 `release readiness` 语义彻底拆开
- 脚本面不需要大动刀，但要防止 future creep
- `run_phase4_first_version.py` 也应保持 thin orchestration，不再让 metadata/quality/mainline-summary 这类 runner 控制面回流到顶层入口
