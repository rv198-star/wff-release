# 问题级需求优先于方案级需求

- 类型：边界
- 适用阶段：需求调研 / 需求分析
- 来源：`sources/books/original/有效需求分析（第2版）.md`（第1章，第2章）
- 摘要：在 B 端需求分析中，用户提出的往往是方案级需求，而真正应优先分析的是问题级需求。先分析问题、业务影响与场景，再给出成本合适的解决方案，能显著降低执行变形与返工。

## 输入条件
- 已收到用户或业务部门提出的功能诉求
- 当前需求表述中含有明显的实现方案倾向

## 关键步骤
1. 判断当前表述是方案级、问题级还是更深层的人性级需求。
2. 对方案级需求追问“如果没有这个功能，会对当前业务造成什么影响”。
3. 将原始诉求改写为用户视角、业务视角的问题级描述。
4. 在问题级需求清晰后，再讨论多种可选方案及其成本。

## 输出结果
- 更稳定、可验证的问题级需求描述
- 避免被用户提出的初始方案绑死

## 适用边界
- 适用于组织应用类软件系统、B 端系统与变更优化需求
- 对 ToC 产品的人性级需求分析仅有部分适用性，需补充动机研究

## 常见错误
- 直接围绕界面、字段、按钮问实现细节
- 把客户说的解决方案直接转给开发
- 忽视问题背后的业务后果与场景
