# v1.2 Phase Thinking Mode Split（v0.1）

## Purpose

This document freezes one `v1.2` design correction that should guide later skill updates:

> **`P1` and `P2` should not be treated as the same thinking mode with different templates**

They solve different classes of problems.
Therefore they should use different default reasoning posture, different loop pressure, and different review logic.

This document is a companion to:

- `docs/v1.2-design-principles-v0.1.md`
- `docs/internal/improvement-reports/v1.2-business-depth-sufficiency-and-superpowers-inspired-agent-loop-v0.1.md`
- `docs/internal/improvement-reports/v1.2-mainline-deliverables-and-backend-quality-bar-v0.1.md`

## 1. Core Position

`P1` is primarily about:

- value gain
- business leverage
- user experience gain
- scenario selection
- defining a credible business world

`P2` is primarily about:

- supporting `P1` well
- implementation realism
- delivery economics
- architecture coherence
- extensibility and future supportability

Short form:

> **`P1 = value-first`**

and:

> **`P2 = support-first`**

Another useful short form:

> **`P1` asks "what should exist to create better business and user outcomes?"**

> **`P2` asks "how should we support that well, quickly, economically, and sustainably?"**

---

## 2. Why They Must Be Split

If `P1` is treated like a checklist-driven fill-in phase, the likely failure is:

- structurally complete PRD
- but thin business world
- weak scenario density
- weak trade-off quality
- demo-like downstream product

If `P2` is treated like another open-ended exploration phase, the likely failure is:

- architecture drift
- slow convergence
- excessive abstraction churn
- overthinking without better implementation leverage

So the design rule is:

- `P1` needs more thinking tension and deeper loop pressure
- `P2` needs more coverage discipline and implementation-support discipline

---

## 3. P1 Thinking Mode

## 3.1 Primary Objective

`P1` should maximize:

- business value clarity
- user value clarity
- scenario leverage
- experience quality pressure
- realism of the chosen business baseline

Its main question is not:

- "have we answered every field?"

Its main question is:

- "have we chosen and deepened the right product world strongly enough that downstream work will not produce a thin or demo-like result?"

## 3.2 Default Reasoning Posture

`P1` should be:

- value-first
- scenario-first
- deep-loop-first
- trade-off-aware
- anti-demo-aware
- evidence-calibrated

This is where the repo should allow the highest thinking tension.

## 3.3 P1 Topology Profile Routing

`P1` should not use one universal depth model for every business.

Before deepening the business world, build a `topology profile`, not a fixed case label.

The profile has two layers:

- `topology_archetype`
  - `execution-centric | decision-centric | hybrid`
- `depth-axis profile`
  - first-wave reusable axes:
    - `operational-chain`
    - `exception-state`
    - `role-coordination`
    - `substitute-positioning`
    - `proof-evidence`
    - `buyer-budget-continuation`

Design rule:

- archetype gives the coarse routing posture
- depth axes decide what kind of depth the loop must actually earn
- the axis set is extensible; do not force every future case into the first-wave list

Implication:

- execution-heavy cases become demo-like when the operating world is too thin
- decision-heavy cases become demo-like when the economic decision loop is too thin

So `ordinary real-world baseline` is profile-dependent, not universal.

## 3.4 Typical P1 Reasoning Tasks

- identify high-value scenarios
- compare alternative product directions
- choose what belongs in the baseline
- identify what would make the product feel fake or thin
- calibrate against ordinary real-world practice
- decide where additional detail has positive business value gain

## 3.5 P1 Source Posture

`P1` method assets may legitimately include more:

- conceptual PM methods
- value framing
- user empathy framing
- decision and trade-off methods
- industry/domain reality calibration

That means `P1` references may naturally feel more "conceptual" or "strategic" than `P2`.
That is not a flaw.
It reflects the type of thinking `P1` is responsible for.

## 3.6 P1 Runtime Pattern

Recommended default:

- topology-profile routing
- structured draft
- bounded deepening loop
- anti-demo review
- real-world baseline calibration
- integration round
- freeze / return / blocked resolution

Short form:

> **`P1` should be loop-heavy before freeze-heavy**

## 3.7 P1 Depth Modes

`P1` now has two depth modes:

- `Baseline Mode`
  - default
  - must reach ordinary real-world baseline
  - must reduce downstream guessing below the handoff threshold
- `Creative Mode`
  - only when the user explicitly asks for stronger ideation or business-model exploration
  - can continue beyond baseline until another round no longer yields additional `positive business value gain` under the repo's existing canonical definition

Important boundary:

> **`Creative Mode` does not replace `Baseline Mode`; it starts only after baseline sufficiency exists**

`Creative Mode` is where the repo may deliberately explore:

- stronger business leverage
- better user-experience shape
- additional scenario families with high value potential
- possible new operating or business models

But its outputs should be layered:

- baseline truth
- optional creative discoveries

so downstream phases do not confuse exploratory opportunities with frozen mainline truth.

## 4. P2 Thinking Mode

## 4.1 Primary Objective

`P2` should maximize:

- support quality for `P1`
- implementation-driving specificity
- architecture coherence
- delivery realism
- future extensibility
- fast/cheap/good balance under real constraints

Its main question is not:

- "what else could the product become?"

Its main question is:

- "given the chosen business world, how do we support it well without overbuilding, underbuilding, or creating fragile delivery paths?"

## 4.2 Default Reasoning Posture

`P2` should be:

- support-first
- checklist-first
- completeness-driven
- consistency-driven
- realizability-driven
- focused-review-driven

`P2` still needs judgment.
But its judgment should be more selective and more implementation-bound than `P1`.

## 4.3 Typical P2 Reasoning Tasks

- architecture boundary selection
- service/module cut quality
- data invariant definition
- contract shape stabilization
- non-functional risk balancing
- extensibility and supportability planning
- cost / complexity / timeline trade-off

## 4.4 P2 Source Posture

`P2` method assets will usually feel more:

- practical
- architecture-oriented
- implementation-oriented
- concrete
- constraint-aware

That is expected.
`P2` should not need the same broad exploratory pressure as `P1`.

## 4.5 P2 Runtime Pattern

Recommended default:

- consume frozen `P1` truth
- run coverage/checklist completion
- run consistency and realizability review
- trigger focused decision review only on high-risk points
- converge into implementation entry

Short form:

> **`P2` should be checklist-heavy with focused high-value reviews, not broad deep-loop exploration by default**

## 4.6 P2 Depth-Mode Boundary

Current `v1.2` canonical stance:

- `P2` uses `Baseline Mode` only
- `P2` does **not** support `Creative Mode` as a current formal capability

Reason:

- the value of `P2` comes from stronger support quality, not open-ended invention
- allowing creative-mode expansion too early risks over-design, abstraction drift, and inflated technical posture

There is one narrow reservation:

- future versions may revisit whether a tightly bounded experimental form is useful on rare high-risk decision points

But that is **not** the current default, and should not shape current skill behavior.

---

## 5. Comparative Table

| Dimension | `P1` | `P2` |
|---|---|---|
| Main orientation | value-first | support-first |
| Main problem type | discover and shape the right business world | support that world with the right engineering design |
| Main risk | thin product world / demo-like output | weak implementation support / incoherent architecture |
| Default runtime | deep loop | checklist + focused review |
| Main judgment style | scenario and value trade-offs | architecture and delivery trade-offs |
| Source posture | partly conceptual / product-thinking heavy | mostly practical / engineering-method heavy |
| Freeze pressure | after sufficient deepening | after sufficient completeness/consistency/realizability |

## 6. What This Means For Skill Design

## 6.1 `wff-req`

`wff-req` should explicitly allow:

- stronger bounded deepening
- anti-demo review
- business baseline calibration
- value-gain-oriented iteration

It should not collapse into:

- template filling only
- gate passing only
- one-pass enrich and freeze

## 6.2 `wff-arch`

`wff-arch` should explicitly favor:

- checklist completeness
- implementation-driving specificity
- focused architecture decision reviews
- consistent support of `P1`
- delivery-minded architecture judgment
- ROI-gated extensibility
- anti-overengineering review

It should not behave like:

- a second broad product-discovery phase
- a second freeform scenario exploration phase
- a maximal future-state architecture exercise
- a shortest-path implementation note that ignores known extension pressure

---

## 7. Focused Review Rule For P2

`P2` is not mindless.
It still needs deeper review at selected pressure points.

Those points should include:

- architecture boundary choice
- data invariants that affect correctness
- contract choices that affect implementation shape
- external dependency posture
- deployment/runtime posture
- extensibility and blast-radius concerns
- ROI of paying extension-boundary cost now
- overengineering risk when local elegance raises average system cost

But those reviews should be:

- bounded
- problem-specific
- decision-focused

not broad exploratory loops by default.

The default `P2` posture is:

> **delivery-minded architect with ROI-gated extensibility**

Meaning:

- allow forward-looking boundaries when the pressure comes from `P1` truth or credible delivery risk
- prefer light extension boundaries over premature full implementation
- move attractive but low-ROI future architecture into optional / expert lanes
- use system efficiency as a stronger tiebreaker once the architecture quality floor is met

Detailed rules:

- `docs/v1.2-p2-delivery-minded-architecture-principles-v0.1.md`

## 8. P1 Positive Business Value Gain vs P2 Support Gain

The notion of "positive gain" is different across the two phases.

### `P1`

The key question is:

- does this new iteration create material positive business value gain?

Meaning:

- better business outcome
- better user outcome
- stronger baseline scenario choice
- lower risk of demo-like product

### `P2`

The key question is:

- does this design choice materially improve support quality for the chosen product world?

Meaning:

- better implementation leverage
- better delivery economics
- lower coupling / higher cohesion
- better extensibility
- better realizability under constraints

This is another reason `P1` and `P2` should not share the same loop pressure.

---

## 9. Anti-Patterns

The following should be treated as drift signals:

- using `P2` to rediscover product value that `P1` should have already frozen
- using `P1` to over-index on engineering structure before the business world is credible
- forcing `P2` into broad brainstorming because `P1` was too thin
- pretending `P1` can be sufficiently solved by checklist completion
- pretending `P2` should use the same loop intensity as `P1`

---

## 10. One-line Rule

> **`P1` should think harder about value, scenarios, and business reality; `P2` should think harder about support quality, delivery economy, and extensibility.**
