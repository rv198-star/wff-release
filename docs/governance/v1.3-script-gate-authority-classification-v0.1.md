## Purpose

This document defines the `v1.3.1` authority boundary for scripts and gates.

It does not delete scripts, rerun proof snapshots, or reclassify every concrete script file. The v1.2 script-surface classification remains the historical script-topology baseline.

This v1.3 document classifies what scripts and gates are allowed to decide.

## Core Rule

Scripts and gates support the Agent / Skills-led mainline. They do not replace it.

Scripts may prove mechanical facts and evidence availability. Gates may block unsafe progression or cap claims. Neither may decide product success, architecture quality, implementation value, test meaning, or investment worth by itself.

Short rule:

```text
scripts prove mechanics;
gates protect boundaries;
Agent / Skills judge value;
evidence caps claims.
```

## Script Authority Classes

| Class | Purpose | May prove | Must not decide |
| --- | --- | --- | --- |
| `always-on safety net` | keep the default path structurally safe | required files exist, schemas parse, commands are available, obvious contract shape is valid | product correctness, architecture adequacy, implementation value |
| `background regression` | detect drift across retained cases or repeated surfaces | expected outputs remain present, known invariants still hold, no forbidden wording reappears | whether a new artifact is valuable or whether a phase should close |
| `strict-proof engine` | provide high-evidence validation and claim ceiling support | trace/evidence closure, runtime/test execution, output-contract validity, package inclusion, formal claim support/failure | business truth, design truth, remediation ownership, release approval |
| `developer diagnostic` | help localize failures during remediation | failure class, missing artifact, stale path, broken dependency, weak evidence signal | final verdict, phase success, whether further investment is warranted |
| `legacy/support` | preserve compatibility or historical helper behavior | historical behavior still runs or a compatibility path exists | current mainline authority, semantic correctness, default control direction |

## Gate Authority Classes

| Class | Purpose | May decide | Must not decide |
| --- | --- | --- | --- |
| `entry / exit boundary gate` | prevent phase progression when required inputs or outputs are missing | continue, return, blocked, or review-bound based on declared contract presence | the substantive value of product, architecture, implementation, or validation judgment |
| `mechanical contract gate` | enforce schemas, IDs, trace shape, package shape, or command shape | pass/fail on declared mechanical contract | whether the artifact is useful or correct in business/design terms |
| `runtime evidence gate` | prove selected install/build/start/test/runtime paths | whether supplied commands/evidence ran and what environment ceiling applies | whether tests are meaningful or whether product value is real |
| `strict-proof / claim-ceiling gate` | validate evidence sufficiency for formal claims | supported claim, capped claim, return reason, minimum rerun boundary | upstream product truth, architecture truth, implementation repair, production authorization |
| `review routing gate` | preserve finding ownership and rerun boundary | category, owning phase, required action, review-bound carryover | silently fixing upstream artifacts or changing accepted truth |

## Allowed Script Claims

Scripts may claim:

- file existence;
- schema validity;
- mechanical contract compatibility;
- trace ID or artifact ID presence;
- install / build / start / test execution where commands are available;
- package inclusion;
- output-contract validity;
- evidence aggregation;
- strict-proof pass / return evidence;
- formal claim support or failure within the stated environment.

## Forbidden Script Claims

Scripts must not decide:

- product goal correctness;
- business value validity;
- user pain strength;
- narrowest valuable wedge;
- architecture tradeoff quality;
- design elegance or adequacy;
- implementation value;
- core-path usability;
- test meaning or assertion quality;
- whether further investment is warranted;
- production approval, real owner sign-off, go-live, rollback readiness, monitoring handover, or production risk acceptance.

## Pass Wording Rule

When a script or gate says `pass`, the surrounding human-facing wording must name what passed.

Acceptable:

- `schema pass`;
- `output-contract pass`;
- `runtime smoke pass`;
- `strict-proof evidence pass`;
- `package-inclusion pass`;
- `testing-validation pass within development / pre-production evidence`.

Avoid:

- `product pass`;
- `architecture pass`;
- `implementation value pass`;
- `business success pass`;
- `delivery success` without an evidence ceiling;
- `release approved` without explicit external authority.

## Routing Rule

If a gate finds a semantic defect, it should route rather than repair:

- source truth / product-spec gap -> pre-P1 admission or P1;
- architecture gap -> P2;
- implementation patch or runtime/test evidence gap -> P3;
- validation evidence-consumption or closure-judgment defect -> P4;
- external production / owner-signoff / online UAT evidence gap -> outside default WFF authority unless supplied as external evidence.

## Change Surface Rule

`WO-119A.11` is central-only unless a release-facing surface explicitly grants scripts or gates semantic authority that contradicts this document.

Do not edit P1-P4 skills merely to repeat this classification.

Do not rewrite the v1.2 script surface classification; it remains the topology baseline. This v1.3 document governs authority and wording.

## Acceptance Question

Accept `WO-119A.11` if this document correctly classifies scripts and gates as support layers:

- scripts prove mechanics and evidence availability;
- gates protect boundaries, route defects, and cap claims;
- Agent / Skills retain product, business, architecture, implementation, test-meaning, and review judgment;
- script/gate pass wording must name the actual passed layer;
- no production or owner-signoff authority is granted.
