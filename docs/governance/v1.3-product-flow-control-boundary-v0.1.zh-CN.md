# v1.3 产品流控制边界（v0.1）

状态：`v1.3.1` 验收门草案
日期：2026-05-03
所属工单：`WO-119A.02`
英文镜像：`docs/governance/v1.3-product-flow-control-boundary-v0.1.md`

## 目的

本文定义 `v1.3.1 Agentic mainline rebalance` 的中心控制边界。

它是后续 skill、phase、template 和 test 更新时应遵循的 release-facing 规则。它本身不修改 generated proof outputs，不重跑 retained release cases，也不批准生产发布。

## 版本目标

`v1.3.1` 要让 WFF 的 P1-P4 默认生命周期主线明确回到 Agent / Skills 主导：workflow 保持顺序，scripts 保持机械安全，evidence 保持 claim 诚实，strict-proof 作为 gate，而不是主线控制器。

## 核心控制关系

v1.3 的控制关系是：

```text
pre-P1 source input -> autopilot mainline -> strict-proof gate -> capped delivery claim
```

含义：

- `pre-P1 source input` 在正式 P1 执行前提供归一化 source truth。
- `autopilot mainline` 是 source truth 足够后默认的 P1-P4 Agent / Skills-led 生命周期路径。
- `strict-proof gate` 验证 evidence sufficiency 并封顶 formal claims；它不拥有产品、架构、实现或 remediation 判断权。
- `capped delivery claim` 是最弱环境证据和 evidence bridge 能支持的最强声明。

`strict-proof` 不能被描述成与 `autopilot` 平级的 runtime mode。它是 proof 和 claim-ceiling gate。

`guided` 不应在 v1.3.1 中被提升为交付模式。如果后续文档保留这个词，它应只表示围绕 Agent-led mainline 的校准、采用、调试或 human oversight 行为。

## WAE 映射

v1.3.1 控制边界是：

```text
Workflow shell + Agentic core + Evidence bridge
```

### Workflow Shell

Workflow 应控制：

- P1 -> P2 -> P3 -> P4 顺序。
- phase entry 和 exit 边界。
- required artifact families。
- phase review breakpoint 的呈现。
- state recording 和 rerun boundaries。
- deterministic transforms 和 packaging。
- repeatable verification command surfaces。

Workflow 不应决定 business truth、architecture quality、implementation value，或 product value 是否真实成立。

### Agentic Core

Agent / Skills 应控制：

- product goal judgment。
- business value 与 user value judgment。
- scenario、acceptance 和 anti-demo judgment。
- architecture tradeoffs 与 delivery economics。
- implementation value 与 core-path usability。
- test meaning 与 assertion quality。
- review finding classification 和 upstream routing。
- value-bearing artifact closure 是否实质成立，而不是装饰性完整。

Agentic judgment 必须被 phase responsibilities 和 evidence ceilings 约束。它不能编造缺失的 upstream truth。

### Evidence Bridge

Evidence 应控制：

- claim-to-source links。
- contract -> test -> code traces。
- runtime proof。
- formal claim ceilings。
- delivery boundary statements。
- strict-proof pass / return evidence。

Evidence 不是主线目标。Evidence 的作用是让 claims 可审查并被诚实封顶。

### Script Support Layer

Scripts 可以控制或证明：

- file existence。
- schema validity。
- mechanical contract compatibility。
- install / build / start / test execution，如果命令可用。
- package inclusion。
- evidence aggregation。
- strict-proof reports。
- formal claim support 或 failure。

Scripts 不能决定：

- product goal correctness。
- business value validity。
- user pain strength。
- architecture tradeoff quality。
- implementation value。
- test meaning。
- 是否值得 further investment。

Script pass 是 support signal，不是 product success。

## 阶段主线责任

P1-P4 应被理解为 Agent / Skills-led product-development lifecycle，而不是 artifact factory。

| Phase | Agent / Skills judgment owns | Workflow / scripts support |
| --- | --- | --- |
| P1 | product value、business value、user value、scenario leverage、acceptance meaning、source truth sufficiency | input structure、PRD assembly、convergence reports、mechanical gates、trace preservation |
| P2 | architecture tradeoffs、delivery path、constraints、risk、evolution、implementation leverage | stage order、ESP structure、trace absorption、schema / contract checks、realizability evidence |
| P3 | runnable delivery value、core path usability、implementation correctness、test meaning、handoff value | contract/test order、build/test/runtime smoke、evidence reports、delivery gate records |
| P4 | validation risk、evidence sufficiency、claim ceiling、return routing、review-bound carryover | acceptance catalogs、evidence aggregation、output-contract checks、strict-proof reports |

这张表是判断边界，不是机械编辑所有 P1-P4 文件的许可。

## Phase Review Breakpoints

`v1.3.1` 要求 phase review breakpoints 作为控制原则，但详细 breakpoint contract 属于 `WO-119A.04`。

在这个边界层：

- 每个 P phase 完成时都应能呈现 review opportunity；
- human review 可以 approve、reject、intervene、request return，或 explicitly skip；
- breakpoint 本身不修改 generated artifacts；
- breakpoint 不把 human continuous collaboration 变成默认交付模型。

## Strict-Proof Failure Routing

当 strict-proof 失败时：

- 记录 failed evidence、claim ceiling 和 return reason；
- 把问题路由到 owning phase；
- 说明 minimum rerun boundary；
- 不允许 proof gate 自己修复 upstream product truth、architecture truth、implementation truth 或 evidence gaps。

默认归属：

- source truth / product value gap -> P1 或 pre-P1 admission；
- architecture / contract / design gap -> P2；
- implementation / runtime evidence gap -> P3；
- validation consumption / closure judgment defect -> P4；
- external production / owner-signoff gap -> 默认不属于 WFF authority，除非作为 external evidence 提供。

## 非目标

`v1.3.1` 不做：

- 重设计 `wff-req-chat`；
- 定义最终 `P1 source input packet`；
- 实现 fresh-context P1 source brief review；
- 实现 `wff-help`；
- 增加公开 named agents role layer；
- 增加 customization overlays；
- 增加 `wff-project-context`；
- 重跑 retained v1.2 proof snapshots；
- 手工编辑 generated proof outputs；
- 声明真实 online UAT、production release approval、owner sign-off、go-live authorization、rollback readiness、monitoring handover 或 production risk ownership。

## Mirror Rule

本文应作为 v1.3.1 control-boundary wording 的中心来源。

后续子工单只应把最小 runtime-affecting rule 镜像到具体 phase、skill、template 或 reference-package surfaces。

不要把整篇文档复制到 P1/P2/P3/P4 skills。

不要为了显示覆盖而编辑 phase surfaces。

如果编辑某个 phase skill，该编辑必须说明它改变或澄清了什么 runtime behavior。

## 验收问题

如果本文正确冻结了 v1.3.1 control boundary，则可以接受 `WO-119A.02`：

- Agent / Skills 是 mainline judgment layer。
- Workflow 控制 order 和 convergence。
- Scripts 提供 mechanical safety 和 proof support，但没有 semantic authority。
- Evidence 封顶 claims，但不成为 mainline goal。
- Strict-proof 是 gate，不是 peer mainline mode。
- Human review breakpoints 是 oversight opportunities，不是 continuous human-led delivery。
- v1.3.5 P1 source-truth admission work 仍在范围外。
