## Purpose

This document clarifies the `v1.3.1` vocabulary for `autopilot`, `guided`, and `strict-proof`.

It does not create a new delivery mode, change the accepted `WO-119A.02` control boundary, rerun proof snapshots, or approve production release.

The goal is to prevent later docs, skills, tests, or release-package language from accidentally turning `strict-proof` back into a peer mainline mode beside `autopilot`.

## Canonical Relationship

The accepted `v1.3.1` relationship remains:

```text
pre-P1 source input -> autopilot mainline -> strict-proof gate -> capped delivery claim
```

Use this relationship when explaining v1.3.1 product flow.

Do not describe the flow as:

```text
autopilot / guided / strict-proof
```

That three-column wording makes proof machinery look like a peer delivery path and can pull WFF back toward script-led artifact closure.

## Vocabulary

### Autopilot Mainline

`autopilot mainline` means the default P1-P4 Agent / Skills-led lifecycle path after source truth is sufficient.

It means:

- Agent / Skills own product, business, architecture, implementation, test-meaning, and review judgment within phase boundaries.
- Workflow preserves order, entry / exit contracts, convergence, and rerun boundaries.
- Scripts provide mechanical checks, diagnostics, regression safety, and proof support.
- Evidence caps formal claims.

It does not mean:

- blind execution without review breakpoints;
- skipping evidence;
- hiding uncertainty;
- letting scripts decide value;
- removing human oversight opportunities.

### Guided

`guided` is not a `v1.3.1` delivery mode.

If retained in later wording, it should mean calibration, adoption, debugging, or human oversight behavior around the Agent-led mainline.

Acceptable uses:

- guided onboarding into the WFF lifecycle;
- guided debugging of a failed run;
- guided calibration of source truth or review expectations;
- human-guided oversight at phase breakpoints.

Avoid using `guided` as:

- a peer mode beside `autopilot`;
- a default human-led collaboration model;
- a replacement for Agent / Skills judgment;
- a reason to weaken strict-proof or claim ceilings.

### Strict-Proof Gate

`strict-proof gate` means proof and claim-ceiling validation.

It may check evidence such as:

- source-to-claim links;
- contract / test / code traces;
- install, build, start, runtime, migration, persistence, or test evidence where available;
- output-contract validity;
- package inclusion;
- formal delivery-gate state.

It does not own:

- product goal correctness;
- business value validity;
- architecture tradeoff quality;
- implementation value;
- test meaning;
- upstream remediation.

If strict-proof fails, it records the failed evidence, claim ceiling, return reason, and minimum rerun boundary, then routes the issue to the owning phase.

## Release-Facing Wording Rules

Use:

- `autopilot mainline`;
- `strict-proof gate`;
- `claim ceiling`;
- `phase review breakpoint`;
- `human oversight opportunity`;
- `script support layer`.

Avoid:

- `strict-proof mode` when it implies a peer delivery path;
- `autopilot / guided / strict-proof` as a mode table;
- `proof-driven mainline`;
- `script-led closure`;
- `gate-owned remediation`;
- `human-led continuous delivery` as the default model.

If an older surface needs to keep `strict-proof mode` as historical wording, the nearby text must clarify that the runtime authority is a gate / claim-ceiling mechanism, not a peer mainline controller.

## Change Surface Rule

This vocabulary record is central-only for `WO-119A.03`.

Mirror a short rule into a phase, skill, template, or reference-package surface only when that surface:

- presents delivery modes;
- tells an agent when to run strict proof;
- describes phase completion authority;
- gives scripts or gates semantic authority over product, architecture, implementation, or review judgment.

Do not edit P1-P4 skills merely to repeat this vocabulary.
