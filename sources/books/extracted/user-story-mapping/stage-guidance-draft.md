# 用户故事地图 Stage Guidance Draft

## Goal
- 为需求分析、需求拆解、需求验证提供一套以故事地图为核心的上游方法指导。
- 避免把用户故事退化为条目管理工具，而保持其作为“全景图 + 协作 + 切片 + 学习”的方法价值。

## Recommended methods
- 先构建全景图，再拆分故事
- 通过探索建立共识，而不是先写完整需求
- 通过故事地图做 MVP / release slicing
- 用故事地图支撑验证性学习循环
- 通过故事工作坊持续提炼，而不是一次性写完故事

## Key decision rules
- 没有全景图时，不应直接进行大规模故事拆分
- 没有形成共享理解时，不应把故事视作稳定输入
- 没有最小可用体验边界时，不应宣称 MVP 已成立
- 如果故事只剩模板句式而无上下文，则应回到讨论与地图层，而非继续细化

## Common mistakes
- 先写大量故事，再试图事后拼出全景
- 用模板代替共识
- 把故事地图当静态计划图，而不是学习与切片工具
- 在未提炼到可消费粒度前就将故事交给下游阶段
