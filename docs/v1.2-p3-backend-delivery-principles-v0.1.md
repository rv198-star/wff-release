# v1.2 P3 Backend Delivery Principles（v0.1）

## 1. Purpose

This document freezes the `v1.2.3` top-level design correction for `P3`:

> **P3 should behave like a runtime-evidence-minded backend implementer with honest delivery boundaries.**

It is a companion to:

- `docs/v1.2-design-principles-v0.1.md`
- `runtime-deps/mindthus/source/skills/wae/SKILL.md`
- `docs/phases/phase-3/phase-3-runnable-delivery-correction-v0.1.zh-CN.md`
- `skills/wff-impl/SKILL.md`

P3 is different from P2.

- P2 decides whether the architecture is sufficient, realizable, and ROI-extensible.
- P3 proves the selected backend mainline can actually be built, started, migrated, exercised, and handed off with honest evidence.

---

## 2. Core Role Definition

`P3` is not a second architecture phase.
It is also not a workflow packaging exercise.

The default P3 role is:

> **runtime-evidence-minded backend implementer with honest delivery boundaries**

That means P3 should:

- consume the P2 ESP as the implementation contract, not reopen architecture by default
- implement accepted contracts test-first rather than coding first and documenting later
- ground high-value tests in P1 business scenarios through P2 contracts
- deepen test assertions until they prove real backend behavior rather than endpoint existence only
- link API docs to scenario, contract, test evidence, and evidence level
- prove infrastructure capabilities separately from mock-backed business tests
- make persistent-state tests reentrant through explicit init / rerun / restore discipline
- implement the backend-first mainline before optional UI or dispatch complexity
- prove real runtime behavior where the environment exists
- keep generated/scaffold/mock/runtime-helper paths below implementation-ready unless replaced by real service boundaries
- expose defer/fallback/review-bound truth instead of using wrappers to look complete
- use targeted agentic loops for code trade-offs, bug triage, failure diagnosis, and bounded rewrites

Chinese short rule:

> **P3 用真实运行证据证明后端主线，而不是用 workflow 包装证明完成。**

---

## 3. What P3 Optimizes For

P3 should optimize for:

- backend runtime startability
- migration execution evidence
- SQL/persistence write-read or state-transition proof
- reentrant persistence proof for stateful tests
- started-service interface verification
- P1/P2 traceable scenario or contract tests
- API docs linked to test evidence
- API Docs consumer executability: documented OpenAPI requests must run without hidden behavior-card helper enrichment
- test-obligation coverage: generated tests must satisfy explicit obligations from OpenAPI, behavior cards, action cards, P1/P2 scenarios, replay rows, and DB schema
- scenario negative-path breadth: core happy chains should also expose invalid, permission, and conflict variants when the accepted API contract documents those failures
- SQL reentry discipline: schema tests must include rollback/restore proof, not only table shape and single insert-read checks
- honest formal state
- minimal default mainline complexity
- clear handoff from implemented backend package to P4 validation

P3 should not optimize for:

- wrapper count
- sidecar volume
- dispatch sophistication
- frontend completeness by default
- generated runtime adapters that remain in core execution paths
- broad hardening before backend mainline evidence exists
- local demo polish that hides unverified backend state
- API docs that only list endpoints without evidence linkage
- API docs that omit required behavior/evidence fields while internal tests silently add them
- mock-backed tests that pretend to prove infrastructure readiness
- persistence tests that pass only once because of residual state

Short rule:

> **P3 should produce runnable backend evidence, not merely delivery-looking artifacts.**

---

## 4. Workflow / Agentic Boundary In P3

The general Workflow / Agentic split remains unchanged, but P3 assigns it differently from P2.

In P3, Workflow should keep:

- ESP handoff intake contract
- environment contract
- separate host-local developer env from container-internal compose env
- deterministic scaffold/build/test/migration commands
- artifact existence checks
- evidence aggregation
- formal state recording
- delivery gate computation
- package assembly

In P3, Agentic should handle:

- backend implementation trade-offs
- code-level boundary decisions within the accepted ESP
- business-scenario-grounded test selection
- assertion deepening for contract, persistence, scenario, unit, and replay tests
- failure diagnosis and bug triage
- targeted rewrite loops
- fallback/defer honesty
- deciding whether a failed runtime path is an implementation problem, environment constraint, or upstream design gap
- deciding whether optional lanes should remain deferred

Scripts may prove, collect, and gate runtime evidence.
They must not use generated helpers, wrappers, or sidecars to imply backend readiness that has not been executed.

Short rule:

> **P3 Workflow proves what ran; P3 Agentic decides how to make the accepted backend design actually run.**

## 4A. Contract-First Implementation

P3 must implement from accepted P2 contracts.

P2 owns:

- service boundaries
- data objects
- interface contracts
- storage strategy
- interaction flow
- implementation-facing handoff

P3 owns:

- executable contract tests
- OpenAPI/schema/migration materialization
- backend implementation that satisfies accepted contracts
- explicit review-bound or bounded return when a contract cannot be implemented honestly

P3 must not silently change accepted contracts to make implementation easier.

If a P2 contract is incomplete, contradictory, or non-buildable, P3 should record the gap and return it as a design/handoff issue rather than hiding it behind implementation changes.

Short rule:

> **P2 defines contracts; P3 implements and proves them.**

---

## 4A.1 Public Contract Skeleton Before Private Implementation

For high-risk public services and APIs, P3 should insert a public contract skeleton pass before private implementation.

The skeleton pass defines public behavior, not private logic.

It should include:

- public method / endpoint signature
- input contract
- output contract
- preconditions
- postconditions
- state effects
- persistence effects
- error semantics
- permission boundary
- idempotency / replay rule when relevant
- evidence expectation
- review-bound items

Then P3 should write contract/interface/scenario tests against the skeleton before implementing private methods.

Short rule:

> **Fix public behavior, test public behavior, then implement private logic.**

### High-risk triggers

Use the skeleton pass when any of the following is true:

1. the API/service is on a P1 core scenario or buyer/operator value path
2. the API/service performs persistent state change
3. the API/service controls permission, tenant isolation, identity, audit, or compliance behavior
4. the API/service crosses a service/module boundary owned by different components
5. the API/service publishes or consumes queue/event/async workflow messages
6. the API/service uses cache semantics that affect correctness, visibility, or freshness
7. the API/service calls an external dependency or substitute boundary
8. the API/service has idempotency, replay, retry, or concurrency requirements
9. the API/service is linked to P2 RBI / review-bound / high-risk boundary decisions
10. changing the public contract after implementation would force meaningful client, test, migration, or downstream workflow rework

### Low-risk candidates

A lightweight path is allowed when all of the following are true:

1. the API/service is internal or low-value support behavior
2. it is read-only or simple deterministic transformation
3. it does not mutate persistent state
4. it does not control permission, tenant isolation, audit, compliance, or money-like decisions
5. it does not cross an external dependency, queue, cache-correctness, or async boundary
6. it has no meaningful idempotency/replay/concurrency semantics
7. P2 contract is already explicit enough that tests can constrain public behavior without a separate skeleton pass
8. changing it later would not cause broad client or persistence migration cost

Low-risk does not mean untested.
It means a concise signature plus contract test may be enough.

### Default tie-breaker

When unsure, classify as high-risk for public contract skeleton purposes.

The cost of a lightweight public skeleton is usually lower than the cost of private implementation inventing public behavior.

## 4B. Business-Scenario-Grounded Testing（测试业务场景化）

P3 tests should not be derived only from code structure, endpoint lists, or coverage targets.

High-value P3 tests should be grounded in P1 business scenarios and anchored through P2 contracts.

Use this chain:

```text
P1 business scenario / state truth / success condition
  -> P2 service boundary / data object / interface contract / interaction flow
  -> P3 contract test / persistence test / scenario test / unit test / replay test
```

For high-value tests, record or make clear:

- P1 scenario or business rule being protected
- P2 contract, service, data object, or flow used as the execution anchor
- test type
- assertion strength
- failure meaning
- evidence level

---

## 4A.2 Traceable Behavior Card Before Tests And Code

For v1.2.3, high-risk public operations require behavior cards before tests and private implementation.

The behavior card extends the public contract skeleton into a human-reviewable bridge:

```text
P1/P2 trace + P2 contract/flow/state source
  -> traceable behavior card
  -> contract/scenario/replay/SQL/unit tests
  -> controller/service/repository implementation
```

The card must reuse `wff-base-traceability-management` for identity, bindings, and link integrity.

Short rule from the Traceability Skill:

> **Documents own meaning. Traceability Skill owns identity, bindings, and link integrity.**

Therefore, behavior cards may explain behavior but must not create a parallel trace ID universe.

Required behavior-card sections:

1. `Traceability Binding`
2. `Business Intent`
3. `Public Contract`
4. `Human-Readable Pseudocode`
5. `Error Trigger Table`
6. `State And Persistence Effects`
7. `Test Mapping`
8. `Implementation Mapping`
9. `Review-Bound Items`

The `Human-Readable Pseudocode` section is not decorative.
It is the implementation-depth bridge: service and repository code must be explainable against these steps, not merely against response shape.

If P2 source precision is missing, the card must mark the item as review-bound instead of inventing hidden behavior. If a flow/sequence/state source is not required, that decision must come from the P2-authored shared operation risk row; P3 must not locally downgrade missing P2 sources into `not_required`.

### 4A.3 Implementation Action Cards

Resolution matrix / discovery policy:

- P3 must consume `p1-value-to-p2-operation-resolution-matrix.json`, `implementation-component-catalog.json`, and `component-action-card-obligation-matrix.json` as canonical JSON bridge artifacts before claiming behavior/action-card source closure.
- The discovery policy is `canonical-json-first-markdown-diagnostic-only`: markdown sections may explain or diagnose missing bridge material, but they do not replace canonical JSON for green trace/action-card authority.
- Minimal operation resolution rows must carry `operation_id`, `api_endpoint`, `http_method`, `risk_tier`, concrete `P1-*` trace IDs, concrete `P2-*` contract/source IDs, `source_files`, `source_anchors`, and `source_requirement_status`.
- Classify missing bridge artifacts as `matrix_missing_from_p2`; classify markdown-only bridge material without canonical JSON as `matrix_present_loader_missed`; classify malformed rows or missing required fields as `matrix_present_invalid_shape`; classify rows without concrete P1 IDs as `matrix_present_no_p1_ids`.
- If the selected Phase-2 root lacks these bridge artifacts, P3 must return upstream or mark review-bound instead of inferring operation/action-card source authority from filenames, endpoint names, or P3-local trace files.


For v1.2.3, code implementation should proceed from implementation action cards, not directly from API shape.

The operation behavior card explains the public behavior. P2-authored `ACD` obligations decide which Service, Domain, Repository, and Adapter units require `slim`, `standard`, `deep`, or `split-required` implementation action cards.

P3 must not downgrade `ACD` locally. If the value-operation bridge, component catalog, or component obligation is missing, mark `p1_value_to_p2_operation_resolution_matrix_missing`, `implementation_component_catalog_missing`, or `component_action_card_obligation_missing`. If P3 observes higher complexity than P2, raise depth or mark `action_card_depth_mismatch`. If required design material is absent, mark `action_card_source_material_missing` instead of inventing behavior.

P3 should prioritize tests for:

- P1 core scenarios
- P1 anti-demo risk paths
- P1/P2 key state transitions
- P2 high-risk service boundaries
- P2 RBI / review-bound paths
- data ownership, permission, error, idempotency, and replay boundaries

Short rule:

> **P1 decides test value, P2 decides test landing point, P3 proves runtime behavior.**

## 4C. Assertion Deepening Loop

P3 needs loop/deep-thinking pressure, but the loop is about assertion value.

The assertion loop should ask:

- would this test fail if the implementation is wrong?
- does it assert business state, persisted data, permission behavior, error semantics, idempotency, replay, or boundary effects?
- is it only checking HTTP 200, endpoint existence, or schema shape?
- does it prove a P1/P2 critical path or only add coverage volume?
- does it hit a real evidence level that supports the claimed formal state?

Workflow may scaffold tests, run them, and report coverage.
Agentic review must decide whether the tests prove meaningful behavior.

For v1.2.3+, this split must be visible in release artifacts:

- `test-obligation-matrix.json` / `test-obligation-audit.json` are Workflow evidence floors. They prove required test surfaces exist and obvious weak signals are blocked.
- `test-richness-review.json` / `test-richness-review.md` are the Agentic review packet. If Agentic review is not completed, the P3 formal claim must remain `review-bound` even when Workflow gates pass.
- `phase3-quality-check.json` must report both layers and cap `recommended_formal_state` at the weakest missing evidence bridge.

Raw test counts, keyword matches, and generated helper assertions are not allowed to certify business richness by themselves.

For v1.2.3+, the Workflow floor must also reject or cap these known virtual-coverage patterns:

- forced-error-only permission / conflict / dependency branches without a derivable condition signal
- scenario negative variants that do not prove state invariance after the rejected action
- replay files without idempotency, duplicate, rerun, or second-pass stress evidence
- SQL FK/state obligations represented only by empty FK loops or transaction titles without concrete update probes

Short rule:

> **P3 test loops deepen assertion strength, not test count.**

---

## 4D. Evidence-Driven TDD

P3 should use test-first implementation, but the test hierarchy must match backend delivery risk.

Default priority:

```text
contract / migration / persistence / scenario evidence
> isolated unit coverage
> raw coverage percentage
```

Unit tests are useful, especially for fast implementation feedback, but they do not replace:

- migration execution proof
- SQL-backed write/read or state-transition proof
- started-service contract proof
- scenario integration proof
- queue/cache/auth/storage capability proof when those components are claimed

Short rule:

> **P3 is test-first, but delivery claims are evidence-first.**

---

## 4E. Evidence-Linked API Surface

P3 API docs are not decorative release documents.

They are the index connecting backend contracts, business scenarios, and runtime evidence.

For core APIs, docs should link:

- P1 scenario
- P2 contract
- endpoint
- handler/service when known
- request / response / error semantics
- test evidence
- evidence level
- review-bound or deferred status

For core business flows and high-risk paths, P3 should produce an `API Scenario Call Chain`:

```text
P1 scenario
  -> P2 interaction flow / service boundary
  -> API sequence
  -> state or data transition
  -> tests proving each step
  -> evidence level
```

Do not default to full API graph generation.
Core APIs need evidence chains; ordinary APIs need at least contract and test indexing.

Short rule:

> **API docs must link to evidence, not only list endpoints.**

---

## 4F. Infrastructure Capability Proof（基础组件能力证明）

P3 tests should separate business behavior from infrastructure capability.

Business scenario tests may begin with mocks when that helps implementation progress.
Infrastructure capability claims require real component evidence.

If the backend uses a component, prove the relevant capability:

- DB: migration plus core write/read or state-transition proof
- cache: set/get/TTL/invalidation proof when cache behavior matters
- queue: publish/consume/retry/failure-boundary proof when async behavior matters
- auth/session/token: at least one allowed path and one denied path
- development-only auth context header fallback: default-off, explicit opt-in only, and never sufficient for delivery-ready auth claims
- object storage/file upload: write/read/delete or signed access proof when claimed
- scheduler/async job: trigger/execution/failure proof when claimed
- external dependency: real verification or substitute boundary plus review-bound status

Do not test unused infrastructure for completeness theater.

Short rule:

> **Mocks may support business logic development; they cannot prove infrastructure readiness.**

---

## 4G. Reentrant Persistence Testing（可重入持久化测试）

Persistent-state tests must prove more than one lucky successful run.

Any test involving DB, cache, queue, object storage, or other persistent/shared state must declare and verify its init / seed / cleanup / restore discipline.

Consider:

- migration can run repeatedly or has an explicit reset strategy
- seed data is idempotent or namespaced by run id
- write tests use isolated tenant / namespace / fixture ids
- cleanup runs after tests or state rolls back predictably
- restore snapshot can rerun the same tests
- cache keys are isolated and TTL/invalidation does not pollute later tests
- queue topics, message groups, and consumer offsets are isolated or reset
- failed tests do not poison the next run

Evidence metadata should expose:

- `state_isolation`: `shared | namespaced | reset | snapshot-restore | ephemeral-container`
- `reentry_policy`: `idempotent-seed | cleanup-after-test | rollback | recreate-schema | restore-snapshot`
- `rerun_proof`: `single-run | repeated-run | clean-start-run | restore-run`

Short rule:

> **No persistence proof without init / rerun / restore discipline.**

---

## 4H. Evidence Level Classification

Every major P3 test/evidence claim should expose the level of evidence it provides.

Suggested levels:

- `mock-backed unit`
- `in-process integration`
- `started-service contract`
- `SQL-backed persistence`
- `cache-backed`
- `queue-backed`
- `containerized runtime`
- `external-substitute-boundary`

Passing tests do not all support the same delivery claim.

Formal state must depend on evidence level, not only on test count.

Short rule:

> **Test pass count is not evidence level.**

---

## 4I. Failure-Triage Discipline

When P3 evidence fails, diagnose before patching.

Classify the failure as one of:

- implementation bug
- weak or wrong test assertion
- environment/bootstrap issue
- P2 contract/design gap
- P1 business ambiguity
- external dependency unavailable

Do not hide failures by:

- silently weakening assertions
- replacing real paths with mocks
- changing accepted contracts without review
- adding wrappers that report green while runtime remains red

Short rule:

> **P3 failures need root-cause classification before fixes.**

---

## 5. P3 Loop Posture

P3 should use targeted execution loops, not broad design loops.

P3 loops are triggered by evidence gaps:

1. migration does not execute
2. backend service does not start
3. generated runtime remains in a core path
4. contract/scenario tests do not hit a real service boundary
5. SQL/persistence proof is missing or only mocked
6. P1/P2 state model is not represented in implementation
7. formal state claims exceed actual evidence
8. optional frontend/dispatch/hardening complexity is blocking backend mainline proof

P3 should not loop merely to add more wrappers, richer reports, or broader optional lanes.

P3 should stop looping when:

- the backend mainline evidence matches the claimed formal state
- unresolved gaps are explicitly review-bound/deferred
- another loop is unlikely to materially improve backend runtime readiness or evidence honesty

Short rule:

> **P3 loops until the backend claim matches runtime evidence, not until the artifact set looks complete.**

---

## 6. Formal State Discipline

P3 must treat formal states as evidence ceilings, not labels of aspiration.

### `foundation-ready`

Allowed when:

- ESP intake is understood
- project scaffold and implementation packets exist
- implementation can begin

Not enough for:

- claiming runnable backend implementation
- delivery readiness

### `implementation-in-progress`

Allowed when:

- real implementation has started
- some tests or scaffolds exist
- S03/runtime gates are not fully green

### `implementation-ready`

Allowed only when:

- runnable implementation code exists
- runtime entrypoint is startable
- build/lint/typecheck/backend targeted evidence is green where applicable
- `strict-runtime` has run full targeted SQL / contract / scenario / replay evidence by default; critical-targeted sampling is only an explicit fast iteration mode
- real backend interface verification hits a started service boundary
- real migration execution evidence exists for the chosen database engine
- at least one SQL-backed persistence write/read or state-transition proof exists per backend work-package family when applicable
- remaining frontend verification gaps are explicit P4 follow-up, not hidden debt

### `delivery-ready`

Allowed only when:

- `implementation-ready` is already true
- S04 audit/docs/deploy/trace/acceptance are green
- package includes a real deploy/start path
- Docker/compose packages include a directly usable compose env example whose service addresses work from inside containers
- Docker/container evidence exists when Docker assets are part of the package, or an explicit equivalent runtime-validation contract exists
- started-service API smoke evidence proves missing bearer, invalid bearer, forbidden role, and one valid authenticated HTTP path against the running backend

Short rule:

> **P3 formal state must be capped by the weakest missing runtime proof.**

---

## 7. Generated Runtime And Mock Boundary

Generated adapters, operation-support kernels, runtime-support kernels, mock services, and scaffold helpers are allowed only as bootstrap aids.

They must not remain the primary backend execution path for a completed packet.

If a core backend CRUD/state path still depends on generated or mock-backed execution:

- cap the state below `implementation-ready`
- emit a mock/dependency manifest
- mark the gap as review-bound or deferred
- do not hide it behind successful wrapper reports

Short rule:

> **Generated runtime can bootstrap P3; it cannot certify P3.**

---

## 8. Optional Lane Discipline

The P3 default mainline is backend-first.

The following should stay optional unless explicitly promoted:

- complete frontend implementation
- fallback UI
- multi-worker / dispatch orchestration
- extra hardening beyond backend readiness
- broad documentation sidecars
- release-readiness packaging

Optional lanes may be valuable, but they must not block or obscure backend mainline state.

Short rule:

> **Optional P3 work may extend delivery value, but it must not define backend readiness by default.**

---

## 9. P3 Exit Rule

P3 may freeze when all of the following are true:

1. ESP handoff is consumed without silently rewriting P2 design truth.
2. Backend mainline package is implemented to the claimed formal state.
3. Build/test/migration/start evidence matches the formal state.
4. SQL/persistence and service-boundary proofs are real where the environment exists.
5. Generated/mock paths do not certify completed backend readiness.
6. `test-obligation-audit.json` has no blocking gaps for pure OpenAPI consumer tests, backend unit tests, and required SQL/runtime obligations.
7. Frontend, dispatch, and hardening gaps are explicit optional/deferred lanes.
8. Review-bound and blocker items are honest and traceable.
9. P4 can validate from the package without guessing what actually ran.

P3 must not freeze when:

- delivery state is wrapper-green but runtime-red
- migration or persistence proof is missing for a database-backed backend claim
- service verification hits generated helpers instead of a real started boundary
- optional frontend or dispatch complexity hides backend incompleteness
- formal state claims exceed evidence

---

## 10. One-Line Rule

> **P2 designs the backend path; P3 proves the backend path runs, and caps every claim at the evidence actually produced.**

Expanded rule:

> **P3 implements accepted contracts test-first, grounds tests in business scenarios, deepens assertions until they prove real backend behavior, links APIs to evidence, proves infrastructure capabilities separately, makes persistence tests reentrant, and caps every delivery claim at the weakest runtime evidence.**

---

## 11. Quality Backtrace Loop（质量回溯闭环）

When a generated P3 output is green but weak, do not repair the one output. Repair the system path that produced and accepted it.

Use the loop:

```text
Review -> Gate -> Red Test -> Generator Fix -> Dual-case Rerun -> Manual Review
```

Apply it when review finds:

- contract tests that assert only shape while missing business/runtime effects
- API docs that list endpoints without evidence links
- scenario/replay tests that preserve fields but lose business meaning
- SQL/persistence tests that do not prove real migration, write/read, constraints, or restore discipline
- verdicts that treat artifact pass as runtime or delivery readiness

Rules:

- do not edit generated case outputs as the fix
- do not add named-case branches
- do not weaken gates or assertions to pass
- backtrace the cause to generator, handoff, gate, prompt contract, runtime evidence, or verdict wording
- prove the fix with at least two representative cases before treating it as a reusable improvement

Short rule:

> **Fix the reusable cause, not the remembered case.**
