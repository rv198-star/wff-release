# 3L5S / 三层五步

## 这是什么

3L5S 是 `Three Layers + Five Steps`，中文叫“三层五步”。它是一套通用问题处理内核，用来把混乱现象收敛成可复述、可定位、可证伪、可执行的任务。

三层是：

- `Discovery`：现在到底发生了什么。
- `Definition`：应该把问题定义成什么。
- `Resolution`：如何把问题拆成可执行方案。

五步是 `BTGSB`：

- `Baseline`：当前基线是什么。
- `Target`：目标状态是什么。
- `Gap`：差距在哪里。
- `Strategy`：用什么策略跨过差距。
- `Breakdown`：拆成哪些可验证的任务。

它的重点不是把表格填满，而是避免 agent 和人类一样犯一个基本错误：问题还没定义清楚，就开始堆方案。

## 解决什么问题

3L5S 主要解决两类问题。

第一类是“现象很多，但问题不清”。例如用户说系统不好用、文档太干、任务总失败、agent 一直跑偏。这里真正的问题可能在目标、约束、事实、执行路径、验收标准或控制边界上。直接进入方案，会把症状当成根因。

第二类是“问题清楚，但太大”。例如已经知道要写中文说明书、重构技能包、跑长任务验证，但这个目标还不能直接执行。此时 3L5S 用 `Baseline -> Target -> Gap -> Strategy -> Breakdown` 把它拆成可以排期、可以验证、可以回退的行动。

它尤其适合 agent 工作流，因为 agent 很容易被局部 prompt、文件、测试失败或用户反馈牵着走。3L5S 会强制区分观察、定义和执行，避免把这三件事混成一句“我再改一下”。

## 核心判断

3L5S 的核心判断是：先判断你处在哪一层，再决定要不要拆任务。

如果连发生了什么都不清楚，先停在 `Discovery`。如果现象清楚但问题定义仍不稳，进入 `Definition`。只有当问题已经可复述、可定位、可证伪时，才进入 `Resolution`。

进入 `Resolution` 后，BTGSB 的价值在于让方案不再漂浮：

- 没有 `Baseline`，就不知道从哪里出发。
- 没有 `Target`，就不知道什么叫完成。
- 没有 `Gap`，策略会变成泛泛建议。
- 没有 `Strategy`，任务拆解会变成随机待办。
- 没有 `Breakdown`，方案仍然不可执行。

因此，3L5S 不是复杂模板，而是防止问题处理过早跳层的最小结构。

## 怎么用

先判断要用轻量版还是完整三层。

如果问题已经被用户明确定义，只需要执行拆解，用 `Single-layer BTGSB`：

1. 写出当前基线。
2. 写出目标状态。
3. 写出关键差距。
4. 选一个策略。
5. 拆成可验证任务。

如果用户输入里混着抱怨、现象、假设和目标，用 `Three-layer 3L5S`：

1. `Discovery`：收集可观察信号，不急着解释。
2. `Definition`：把问题压成一句可以被反驳的话。
3. `Resolution`：用 BTGSB 拆执行。

一个实用标准是：如果你不能用一句话说清“我们现在要解决的不是 X，而是 Y”，通常还不该进入方案阶段。

## 具体案例

### 案例 A：用户说“系统不好用”

不要立刻进入 Resolution 改界面或加功能。先做 `Discovery`：用户真正不满意的是入口太深、流程太长、报错不清，还是关键动作找不到？再做 `Definition`：问题可能不是“系统功能不够”，而是“新用户第一次完成核心任务的路径太不稳定”。

定义清楚后再进入 BTGSB：当前基线是用户需要 8 步完成任务，目标是 3 步内完成，差距是入口、提示和错误恢复，策略是先修核心路径，最后拆成导航调整、提示文案、错误状态和验证任务。

### 案例 B：测试失败但原因不明

如果只看到 `unittest` 失败，不要直接改代码。先在 `Discovery` 中收集失败测试、报错、最近 diff 和复现条件；再在 `Definition` 中判断问题是行为回归、测试契约变化，还是文档措辞不匹配。

只有当问题能被一句话复述时，才进入 `Resolution`。否则继续改代码，往往只是把一个未知问题变成多个未知问题。

## 常见误用

第一种误用，是把 3L5S 当成固定表格。结构很完整，但每一格都是空话，这比不写结构还危险，因为它制造了“已经想清楚”的错觉。

第二种误用，是跳过 Definition。很多执行失败不是因为任务拆得不细，而是根本拆错了对象。此时继续加任务只会扩大返工。

第三种误用，是在低风险、问题已经明确的场景里强行跑完整三层。3L5S 的价值是降低判断成本，不是制造仪式感。

## 边界

3L5S 不替代领域证据、运行时验证或 stakeholder 判断。它能把问题说清楚，但不能凭空决定事实。

如果真正卡住的是“谁应该控制这段工作”，用 `WAE`。如果卡住的是“长期方向和局部优势怎么取舍”，用 `SELA`。如果卡住的是“A/B 命题结构都不稳”，用 `EDSP`。

当问题已经小到可以直接执行，不要为了形式继续拆。一个清楚的下一步，有时比完整三层更有价值。

## 与其他方法的关系

- `using-mindthus` 在 3L5S 前做前置校准，去掉二手概念。
- `EDSP` 处理 3L5S 里模糊的结构判断。
- `WAE` 处理 3L5S 产物里的控制边界。
- `TVG` 用在 3L5S 产物已经成型但仍然偏薄时。
- `tplan` 可以把 3L5S 拆出的任务放进 Mission runtime。
