# 测试验证阶段 Skill 包草案（v0）

## 文档目的

这份文档是给人类评审的 `v0` 大纲草案，用于说明测试验证阶段建议如何拆子阶段、每个子阶段包含哪些 Skill 结构、输入物、输出物、SOP 与引用素材。

---

## 先看这页：测试验证阶段总览

### 建议采用 3 个主子阶段 + 1 个可选 Stage-04 扩展

| 子阶段 | Stage Skill 名称（建议） | 目标 | 主要输入物 | 主要输出物 |
|---|---|---|---|---|
| 1 | 测试设计 / 覆盖规划 | 明确测试范围、策略、入口/出口条件 | 开发交接包、需求结果、设计结果 | 测试方案、测试用例结构、覆盖说明 |
| 2 | 测试执行 / 缺陷识别 | 执行测试并识别缺陷与风险 | 测试方案、测试用例、系统实现结果 | 测试结果、缺陷列表、风险说明 |
| 3 | 测试收口 / 交付判断 | 判断是否达到阶段收口标准 | 测试结果、缺陷列表、风险说明 | 测试结论、收口建议、是否需要启用可选 Stage-04 |
| 4（可选） | 发布收口 / 最终交接 | 在正式发布前形成 go/no-go、风险接受和交接包 | Stage-03 收口结论、风险清单、系统信息、交接清单 | 发布前收口结论、风险接受说明、最终交接包 |

---

## 一、子阶段清单（评审用）

### 子阶段 1：测试设计 / 覆盖规划

| 项目 | 内容 |
|---|---|
| 建议 Stage Skill 名称 | `acceptance-coverage-planning` |
| 主要目标 | 明确测试目标、测试范围、覆盖策略、入口/出口条件 |
| 典型输入物 | 需求结果、设计结果、开发交接包 |
| 典型输出物 | 测试方案、测试用例结构、覆盖说明 |
| SOP 重点 | 识别关键验证点、定义入口/出口 gate、安排测试优先级 |
| 需要的模板 | 测试方案模板、测试用例模板、覆盖说明模板 |
| 主要引用素材 | 《用户故事地图》验证性学习线、《启示录》产品验证逻辑、已有测试 gate 相关资产 |

### 子阶段 2：测试执行 / 缺陷识别

| 项目 | 内容 |
|---|---|
| 建议 Stage Skill 名称 | `evidence-execution-and-defect-identification` |
| 主要目标 | 按测试方案执行验证并识别缺陷和风险 |
| 典型输入物 | 测试方案、测试用例、实现结果 |
| 典型输出物 | 测试结果、缺陷列表、风险说明 |
| SOP 重点 | 执行测试、记录异常、收集证据、分类缺陷 |
| 需要的模板 | 测试执行记录模板、缺陷记录模板 |

### 子阶段 3：测试收口 / 交付判断

| 项目 | 内容 |
|---|---|
| 建议 Stage Skill 名称 | `validation-closure-and-delivery-readiness-judgment` |
| 主要目标 | 判断测试阶段是否可收口，是否需要进入可选的发布收口扩展 |
| 典型输入物 | 测试结果、缺陷列表、风险说明 |
| 典型输出物 | 测试结论、收口建议、进入 Stage-04 的条件说明 |
| SOP 重点 | 对照入口/出口 gate、检查遗留风险、给出收口判断 |
| 需要的模板 | 测试结论模板、风险接受说明模板 |

### 子阶段 4（可选）：发布收口 / 最终交接

| 项目 | 内容 |
|---|---|
| 建议 Stage Skill 名称 | `release-readiness-and-final-handoff` |
| 主要目标 | 在需要正式发布门禁时，完成风险接受、最终交接和发布准备收口 |
| 典型输入物 | Stage-03 收口结论、风险清单、部署/运行信息、交接清单 |
| 典型输出物 | 发布前收口结论、风险接受说明、最终交接包 |
| SOP 重点 | 明确 go/no-go 条件、残余风险、人工签发与交接责任 |
| 需要的模板 | 发布前检查清单、风险接受模板、最终交接模板 |

---

## 二、当前 v0 判断

- 测试验证阶段建议先冻结 3 个主子阶段，并把发布收口合并成可选 Stage-04 扩展
- 当前已经有部分测试 gate 相关资产，但整体支撑仍弱于产品/需求阶段
- 可选 Stage-04 只有在项目需要正式 release gate、风险接受和交接包时才启用，不建议默认强制进入主链
- 这一阶段的大纲目前足够做整体评审，但距离正式落地还需要更多模板和验证资产支持
