# 上下文适配优先于框架照搬

- 类型：边界
- 适用阶段：方法治理 / engineering lane 选择 / 阶段 Skill 设计
- 来源：`sources/books/original/精益产品开发_原则_方法与实施.md`（第2章、第4章）
- 摘要：精益方法与规模化框架可以提供原则和实践参考，但不能脱离组织上下文机械照搬。真正有效的是围绕用户价值和本地约束进行适配。

## 输入条件
- 团队准备引入某个框架、方法包或行业实践
- 当前存在“直接照搬成熟框架”的冲动

## 关键步骤
1. 先明确当前组织/项目的业务目标和约束。
2. 判断哪些原则是通用的，哪些实践必须本地化适配。
3. 以价值流和问题类型为基准选择做法，而不是以框架名气为基准。
4. 对引入的方法持续做裁剪和调整。

## 输出结果
- 更符合上下文的方法选择策略
- 对“框架名”与“真实有效做法”之间区别的清晰认识

## 适用边界
- 适用于准备引入敏捷、精益、SAFe、LeSS、SOMA、SOA 等方法时的前置判断
- 不适用于只是学习某个方法历史背景的场景

## 常见错误
- 用框架名称代替问题分析
- 把成熟框架直接当模板照抄
- 先选框架，再强行解释为什么项目适合它

## 关联模板
- lane decision
- 方法治理说明
- adoption note

## 下游使用建议
- 可被未来的 Engineering Lane Decision 卡片引用
- 可支撑当前项目“高阶方法资产先进入知识底座和治理层，再渐进式落地”的策略
- 可作为拒绝规则的一部分：若项目上下文未分析清楚，不得先行宣称某方法为默认主路线
