# WFF v1.4 Release Notes

发布日期：2026-05-25

状态：`released`，但有明确能力边界。

## 一句话

v1.4 把 WFF 从“主线 P1-P4 生命周期工具”扩展成更完整的可安装技能体系：补强 PX 存量项目 / 历史遗留项目接手能力，拆分 P3 实现技能，加入项目级上下文文件，并提供可选的 WFF Role Agents 实验入口。

## 主要变化

### 1. PX 存量项目接手能力

PX 当前发布面统一到 `wff-x-*` 命名：

- `wff-x-scan-code-baseline`
- `wff-x-scan-db-baseline`
- `wff-x-scan-biz-arch`
- `wff-x-scan-tech-health`
- `wff-x-intake-target-driver`
- `wff-x-design-target-arch`
- `wff-x-plan-test-protection`
- `wff-x-plan-refactor`
- `wff-x-plan-migration`

核心含义：

- 用于接手、改造、重构、迁移已有系统；
- 先把当前系统事实、数据库事实、业务流程候选、目标变化、安全网和迁移路线说清楚；
- 不把 PX 变成默认 Phase-5，也不把静态扫描说成生产真值。

### 2. `wff-project-context`

新增项目上下文技能，用于维护 `.wff/PROJECT_CONTEXT.md`。

这个文件不是 PRD 或工程设计包的替代品，而是给 AI 的“项目常识入口”：

- 项目长期要坚持的业务原则；
- 架构、设计、代码实现中的非显然约束；
- 团队技术栈、数据库、行业、用户群体等会影响决策的背景；
- 指向关键文档的少量高价值引用。

原则是精炼，不写通用最佳实践。

### 3. P3 实现技能拆分

P3 不再只是一团大的实现入口。v1.4 把它整理成三个层次：

| 层次 | 内容 | 用户心智 |
|---|---|---|
| 主入口 | `wff-impl` | 普通用户和 WFF Programmer 从这里进入 P3 |
| 可选切片能力 | 实现任务卡、数据库结构、后端、前端、API 文档、验证、评审 | 高级用户、切片安装或角色编排时使用 |
| 收口侧能力 | 安全审计、交付打包和最终交付检查 | 通常由 P3 主入口在终态审计和交付收口时调用 |

用户可以按场景安装更小的技能组合，而不是总是安装完整生命周期包。

对普通用户来说，P3 主入口仍然是 `wff-impl` 或 WFF Programmer。细分技能会随 `implementation-delivery` / `full-lifecycle` 安装组合进入运行环境，但不需要作为一级入口记忆。

### 4. 安装组合

新增面向用户场景的安装组合：

- `requirements-prd`
- `architecture-design`
- `implementation-delivery`
- `validation-closure`
- `brownfield-change`（存量项目变更）
- `full-lifecycle`
- `role-agent-companion`

这些安装组合的目标是降低首次安装和理解成本。

### 5. WFF Role Agents 实验入口

v1.4 加入可选的 role-agent companion。

角色包括：

- WFF PM
- WFF Architect
- WFF Programmer
- WFF QA Tester
- WFF Refactor Architect
- WFF Reviewer

这条线的核心不是“人格扮演”，而是把角色责任边界和可挂载 SKILLS 组合好，让用户可以用更自然的方式请求工作。

示例：

```text
@WFF Programmer 帮我基于 P2 设计实现后端 API。
@WFF Reviewer 审查当前 P3 产物是否能进入 P4。
@WFF Refactor Architect 帮我判断这个旧系统的重构边界。
```

平台适配：

- OpenCode
- Claude Code
- Codex

使用说明见：

- `docs/WFF-ROLE-AGENTS.zh-CN.md`

注意：Role Agents 仍是实验入口。v1.4 声明本地配置导出通过，并补充本机 Codex / OpenCode / Claude Code CLI 级轻量验收；不声明三个平台真实 UI 端到端验收。
