# 用户故事 3C 基础

- 类型：概念
- 适用阶段：需求分析 / 需求拆解 / 需求表达
- 来源：`sources/books/original/用户故事地图.md`（第6章）
- 摘要：用户故事不是一句模板文本，而应被理解为 Card、Conversation、Confirmation 的组合。真正的价值不在卡片本身，而在围绕卡片展开的对话与确认。

## 输入条件
- 团队正在使用用户故事或类似需求卡片
- 当前需求表达容易被误解为“写一句模板就完成了”

## 关键步骤
1. 把卡片视为讨论锚点，而不是完整需求载体。
2. 围绕卡片展开对话，补足上下文、目标、边界和细节。
3. 用确认条件帮助团队验证是否理解一致。

## 输出结果
- 对用户故事更完整的理解框架
- 更稳定的需求表达与沟通机制

## 适用边界
- 适用于基于故事卡或需求卡推进协作的场景
- 不适用于完全无对话基础、只想机械记录字段的静态归档任务

## 常见错误
- 把 Card 当成全部
- 跳过 Conversation
- 用 Confirmation 替代共同理解

## 关联模板
- 用户故事模板
- 故事工作坊记录
- 需求确认记录

## 下游使用建议
- 可作为“需求表达 Skill”的基础概念卡
- 可帮助 future refusal：如果只有卡片无对话和确认，不应视为稳定需求输入
- 可补强《用户故事地图》的方法链条完整度
