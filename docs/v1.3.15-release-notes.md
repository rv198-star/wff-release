## What v1.3.15 Changes

v1.3.15 closes the reader translation restart line. It keeps canonical P1/P2 source-of-truth documents in English, then generates localized reader artifacts from that canonical English source for human review.

The release claim is intentionally narrow:

- localized reader first-pass quality is materially improved and usable;
- P1/P2 reader generation preserves canonical English source truth and reader packaging boundaries;
- current `main` passes a fresh release-packaged GEO + PetClinic P1-P4 strict-runtime snapshot;
- stable `90+` polished long-document localized-reader quality is not claimed.

## LLM API Requirement

v1.3.15 introduces a real runtime dependency for producing human-reviewable localized reader translations: an OpenAI-compatible LLM API must be configured.

The translation runner reads:

- `OPENAI_API_KEY` for the API key;
- `OPENAI_BASE_URL` for the OpenAI-compatible endpoint, unless `config/generated-output-policy.json` provides `reader_translation.api_base_url`;
- `WFF_TRANSLATION_MODEL` for the model override, unless `config/generated-output-policy.json` provides `reader_translation.model`.

The checked-in default policy uses `deepseek-v4-flash` with `https://api.deepseek.com/v1`, and still expects the key through `OPENAI_API_KEY`.

This is not a global lifecycle blocker. If the API is not provided, WFF can still preserve canonical English source truth, run the normal P1-P4 lifecycle, and generate non-LLM localized audit surfaces that belong to the existing deterministic pipeline. What cannot be completed is the v1.3.15 LLM-produced localized reader translation evidence for human review.

When the translation runner is invoked without a working API, it reports `translation-failed` and exits non-zero. Release or review claims that depend on localized reader translation should then be marked as missing reader evidence / translation required, not silently replaced with a pseudo-localized generated document.

## Follow-up Planning

v1.3.15 intentionally leaves the legacy deterministic `.zh-CN.md` audit mirror in place because P1 reports, skill-family audit, runtime snapshot evidence, and tests still reference it as a compatibility surface.

That mirror should not remain the long-term reader mechanism. A follow-up planning candidate is recorded at [`superpowers/plans/2026-05-17-v1316-reader-artifact-boundary-cleanup.zh-CN.md`](superpowers/plans/2026-05-17-v1316-reader-artifact-boundary-cleanup.zh-CN.md). The target is to make canonical English artifacts the only mainline source truth, make LLM localized reader artifacts the human-reviewable translation surface, and downgrade or remove the old `.zh-CN.md` mirror.

Even after the replacement, localized reader translation should remain optional evidence. Missing LLM API or missing localized reader output must cap reader/localization claims, not block the P1-P4 lifecycle.

## Claim Boundary

v1.3.15 proves development / pre-production lifecycle generation and validation closure only.

It does not claim real online UAT, production deployment, production release approval, real owner sign-off, budget approval, go-live authorization, rollback readiness, production monitoring handover, production risk acceptance, stable `90+` localized-reader polish, or perfect generated artifact quality.
