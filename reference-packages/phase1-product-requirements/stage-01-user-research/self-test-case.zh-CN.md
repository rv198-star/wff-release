# Stage-01 自测案例（审计镜像）— 从非正式需求描述到结构化用户理解输出

## 1. 测试目标

用一段非正式需求描述来验证 Stage-01 是否表现符合预期：

- 先澄清，不先给方案
- 关键信息缺失时会 blocked / refusal
- 用户要求先推进时，才进入 provisional inference
- 产出符合 Stage-01 结构化输出要求

这不是完整的人机问答 benchmark，而是一份轻量自测样例。

---

## 2. 非正式输入描述

> 我想给小餐馆老板做个简单工具。他们经常在微信上回复顾客，很多问题反复出现，但容易跟丢。我希望 AI 能帮他们更快回复，也许还能整理常见问题。我现在还不确定产品最终应该长什么样，也还没有正式访谈过很多用户。

---

## 3. 预期 Stage-01 行为

### 3.1 预期追问重点
系统应优先追问：

- 主要用户到底是谁
- 哪类重复问题最关键
- 当前最强痛点是什么
- 现有证据是什么
- 是否存在硬约束或明确排除项

### 3.2 预期 blocked / provisional 行为
- 如果用户边界仍不清，不能静默 gate-pass。
- 如果用户说“你先帮我推一版”，才允许进入 provisional inference。
- 推演出的 proto-persona / story / opportunity list 必须显式标记为 provisional。

---

## 4. 预期最小输出结构

至少应包括：

- 明确的目标用户边界
- 至少一版 User Case / User Story
- 结构化问题清单
- 结构化机会清单
- assumptions
- open questions
- 若有推演内容，则必须有 provenance / confidence / verification 标记

---

## 5. 预期验收判断

### PASS 条件
- 输出不再只是零散想法
- Stage-02 可以直接基于它继续结构化需求
- 推演内容没有被当成确认事实

### FAIL 条件
- 仍没有真正的目标用户边界
- 直接跳进解决方案设计
- provisional 内容没标记
- 没有 User Case / User Story 草案

---

- 中文镜像没有弱化英文主文件里的硬规则
