# Stage-02 Charter — requirements-analysis

## 1. Authoring Goal

- Target Stage Skill: `requirements-analysis`
- Parent phase: product / requirements
- Goal: convert Stage-01 user-understanding outputs into a structured requirements view that can support Stage-03 decomposition and MVP slicing.

## 2. Problem This Stage Must Solve

This stage does not decide implementation slices yet. It is responsible for answering:

- what the product/requirements panorama actually is
- how goals, activities, tasks, and constraints relate
- what the main end-to-end backbone is
- what key constraints and high-risk points must remain visible downstream

## 3. Upstream Inputs

- Stage-01 structured research summary
- user-group boundary draft
- first-pass User Case / User Story draft
- structured problem / opportunity list
- assumptions / open questions

## 4. Downstream Handoff

- Downstream Stage: `requirements-decomposition-and-mvp-slicing`
- Minimum handoff package:
  - structured requirements analysis note
  - requirements panorama / story-map / equivalent structure view
  - key constraints list
  - first-pass priority split
  - explicitly marked high-risk validation points

## 5. Gate Focus

- Has Stage-02 built a whole-picture structure rather than a story pile?
- Are goals, activities, tasks, and constraints distinguishable?
- Is the output strong enough to support Stage-03 slicing?
- Has Stage-01 provisional uncertainty been preserved rather than silently upgraded?

## 6. Diagram Focus

- Stage-02 has a hard diagram evidence gate.
- At least one of the following must exist:
  - `story-map`
  - `requirements-structure`
- If neither exists, Stage-02 cannot pass.

## 7. Current Authoring Boundary

- This round first creates Stage-02 control artifacts.
- It does not yet finalize all runtime text in this file.
- It does not rewrite upstream source bundles.
