# IA 与敏捷并不冲突：计划可变，但规划不可或缺

- 类型：原则
- 适用阶段：设计收敛
- 来源：`sources/books/original/phase-02-design-architecture/Information_Architecture_For_The_Web.md`（Getting IA Done / agile false dilemma）
- 摘要：IA 的阶段框架帮助团队看清“大图景目标”；敏捷迭代在清晰方向下更有效。

## 输入条件
- 团队以“我们敏捷”为由拒绝研究/策略/架构规划

## 关键步骤
1. 用 IA 框架给出目标与范围（cathedral vs garage）。
2. 在构建中迭代 IA（结构与词表会随真实使用变化）。
3. 把不确定性与调整机制显式写进治理流程。

## 输出结果
- 既可迭代又不迷航的交付节奏

## 适用边界
- 不能用“迭代”当借口跳过基础结构设计。

## 常见错误
- 把“反瀑布”误解为“不做规划”

## 关联模板
- IA 迭代计划（每次迭代的结构/词表变更记录）

## 下游使用建议
- Stage-2：用于推动“先建立最小可治理 IA，再迭代扩展”。
