# 产品 / 需求阶段 Skill 包草案（v0）

## 文档目的

这份文档不是最终规范，而是一个给人类产品同事评审的 `v0` 草案。

目标是回答四个问题：

1. 软件生命周期先分成几个大阶段更合适
2. 第一阶段（产品 / 需求）适合拆成哪些子阶段 / Stage Skills
3. 是否需要单独的 POC / 验证流
4. 当前项目最小 MVP 应该先做到什么程度

---

## 先看这页：产品 / 需求阶段总览

### 推荐的大阶段位置
- 生命周期先按 5 个大阶段组织：
  1. 产品 / 需求
  2. 设计 / 架构
  3. 开发实现
  4. 测试验证
  5. 上线前收口 / 发布准备

### 产品 / 需求阶段建议拆成 4 个子阶段 / Stage Skills

| 子阶段 | Stage Skill 名称（建议） | 目标 | 主要输入物 | 主要输出物 |
|---|---|---|---|---|
| 1 | 需求调研 / 用户理解 | 理解用户、角色、问题、机会、目标用户群 | 项目背景、业务机会、用户反馈、市场信息 | 调研记录、干系人分析表、用户群划分、User Case / User Story 草案、问题/机会清单 |
| 2 | 需求分析 / 需求结构化 | 把调研结果转成结构化需求视图 | 调研记录、用户群划分、业务目标 | 需求分析说明、全景图/故事地图、约束清单 |
| 3 | 需求拆解 / MVP 切片 | 拆成可分批交付的集合，定义 MVP | 需求分析说明、结构化需求视图、产品目标 | 需求拆解结果、MVP 定义、发布切片说明 |
| 4 | 需求验证 / 概念验证 | 验证关键需求、关键方案和关键假设 | 需求拆解结果、MVP 说明、关键风险与假设 | 验证记录、低保真原型、假设验证结论、修订建议 |

### 这一阶段包的兼容原则

当前不建议先按 2B / 2C 分两套流程。

更推荐的思路是：

> 统一 4 子阶段骨架 + 可拆卸能力包。

也就是说：
- 所有项目先共用同一套阶段骨架
- 再按项目特征启用不同能力包
- 而不是先决定“这是 2B 还是 2C”，再套一整套独立流程

### 可拆卸能力包（示意）

| 子阶段 | 常见可选能力包 |
|---|---|
| 需求调研 / 用户理解 | 用户研究、用户分群、组织角色识别、业务流程调研、竞争分析、数据画像 |
| 需求分析 / 需求结构化 | 故事地图、角色矩阵、流程建模、系统边界识别、规则结构化 |
| 需求拆解 / MVP 切片 | 最小体验闭环、最小业务闭环、权限/配置闭环、实施准备闭环 |
| 需求验证 / 概念验证 | 原型验证、用户测试、场景走查、试点 / POC / pilot、多角色评审 |

### 裁剪原则
- 不要求每个项目启用全部能力包
- 也不要求先按 2B / 2C 做分类
- 更建议根据项目特征裁剪：
  - 用户是谁
  - 是否多角色决策链
  - 是否有复杂业务流程
  - 是否强调体验闭环
  - 是否强调业务可运行闭环
  - 是否需要试点 / pilot / POC

### 每个 Stage Skill 最小包含 4 类文档

| 文档 | 作用 |
|---|---|
| `skill-contract.md` | 定义目标、输入物、输出物、refusal、gate、上下游 |
| `stage-sop.md` | 定义该阶段内部的标准执行步骤 |
| `output-template.md` | 定义该阶段标准产物长什么样 |
| `source-cards.md` | 定义该阶段优先引用哪些知识卡 / 方法卡 / 边界卡 |

### 原型设计放置原则
- 需求阶段：验证型 / 低保真原型
- 设计阶段：表达型 / 交付型原型

### POC / 验证流放置原则
- 不单独做成生命周期大阶段
- 先作为产品 / 需求阶段里的一个子阶段：`需求验证 / 概念验证`

### 当前最小 MVP（建议）
- 先做一个完整但不过度铺开的“产品 / 需求阶段 Skill 包”
- 并且让它能稳定把产物交给设计阶段

### 当前控制层补充
- 产品 / 需求阶段中，优先级不应隐含存在，而应显式产出至少一版优先级结果。
- 生命周期不应只有阶段定义，还应存在阶段里程碑，用于锚定收口与流转。

## 一、生命周期大阶段建议（v0）

当前建议先按 **5 个大阶段** 组织，而不是一开始切成十几个平级 Skill。

### Stage A：产品 / 需求
回答“为什么做、做什么、先做什么”。

### Stage B：设计 / 架构
回答“系统怎么组织、边界怎么划、原型和设计怎么表达”。

### Stage C：开发实现
回答“如何按约束实现、如何组织编码、如何验证实现完成度”。

### Stage D：测试验证
回答“如何证明做对了、覆盖了、可交付”。

### Stage E：上线前收口 / 发布准备
回答“是否具备发布条件、是否已完成交接、是否风险可控”。

---

## 二、为什么建议先按 5 个大阶段组织

### 优点
- 生命周期完整，不会在“设计后直接结束”
- 对人类评审更直观
- 后续可以在每个大阶段下再继续拆 Stage Skills
- 不会一开始就因为颗粒度太细导致体系失控

### 当前取舍
- 现在不建议把开发过程省掉
- 也不建议把所有大阶段一次性拆成细颗粒 Skill
- 更建议先把“大阶段结构”定住，再讨论子阶段粒度

### 1. 需求调研 / 用户理解

#### 目标
- 理解用户、角色、业务机会、场景、问题、目标用户群

#### 对应 SOP 重点
- 识别调研目标与对象
- 选择研究方法（访谈 / 问卷 / 可用性测试 / 留置研究等）
- 听用户故事并提炼 User Case / User Story 草案
- 形成用户群与关键问题的初步认知

#### 典型输入物
- 产品背景信息
- 业务机会描述
- 已有用户反馈 / 数据
- 市场与竞争信息（可选）

#### 典型输出物
- 需求调研记录
- 用户群划分草案
- User Case / User Story 草案
- 关键问题 / 机会清单

#### 典型引用素材
- 《幕后产品》：
  - `direct-user-research-principle.md`
  - `rapid-user-group-segmentation.md`
  - `group-psychology-and-opinion-leaders.md`
- 用户画像主题组：
  - `persona-label-taxonomy.md`

#### 强化说明
- 当前建议不要把“用户理解”只做成调研结论归档。
- 更好的做法是把调研结果进一步结构化为：
  - 干系人分析表
  - User Case / User Story 草案
这样后续需求分析阶段能更直接消费这些产物。

## 四、产品 / 需求阶段是否需要 POC / 验证流

### 结论
需要，但建议不要把它做成与生命周期主干平级的单独大阶段。

### 建议定位
更合理的定位是：

> **作为产品 / 需求阶段中的“需求验证 / 概念验证”子阶段存在。**

进一步说：
- POC / pilot 不一定每个项目都启用
- 更适合被当成 `需求验证 / 概念验证` 子阶段下的一个可选能力包
- 是否启用，取决于项目特征，而不是先看它属于 2B 还是 2C

### 原因
- 在 AI 时代，验证成本下降，可以更早做低成本验证
- 但并不是所有项目都需要重型 POC
- 把它放在产品 / 需求阶段中，可以把“验证”嵌入主链，而不是额外裂出一条旁路

### 什么时候触发
- 关键假设不明确
- 方向存在争议
- 技术可行性或用户可接受性存在高不确定性
- 方案评审无法仅靠分析收敛时

## 五、原型设计放在哪个阶段

### 结论
原型不只属于一个阶段，而是分两类：

#### A. 需求阶段的原型
- 探索型原型
- 验证型原型
- 低保真原型

用于：
- 需求验证
- 方案假设验证
- 共识建立

#### B. 设计阶段的原型
- 交互原型
- 交付型原型
- 高保真原型

用于：
- 设计细化
- 实现交接
- 测试输入

所以：

> 产品 / 需求阶段应包含“验证型原型”，设计阶段再包含“表达 / 交付型原型”。

## 九、产品 / 需求阶段的里程碑（v0）

当前建议至少定义 4 个里程碑：

### M1：调研输入完成
- 标志：
  - 已形成调研记录
  - 已识别干系人与角色
  - 已有 User Case / User Story 草案

### M2：结构化需求完成
- 标志：
  - 已形成需求分析说明
  - 已形成全景图 / 故事地图
  - 已有优先级初稿

### M3：MVP 与切片完成
- 标志：
  - 已明确最小可用体验 / 最小业务闭环
  - 已明确首批 / 后续 / 暂缓项

### M4：验证收口完成
- 标志：
  - 已完成关键假设验证
  - 已形成验证结论和修订建议
  - 已具备进入设计阶段的 handoff 包

这些里程碑不等于生命周期大阶段，而是产品 / 需求阶段内部的控制锚点。

---

## 十、当前最小 MVP 建议

### 当前项目的最小 MVP，我建议还是维持偏保守版本：

> 先做一个完整但不过度铺开的“产品 / 需求阶段 Skill 包”，并让它能稳定把产物交给设计阶段。

也就是说，当前最小 MVP 可以定义为：

1. 统一契约模板
2. 统一 SOP 模板
3. 统一输出模板
4. 一个完整的产品 / 需求阶段 Skill 包
5. 产品 / 需求 -> 设计 的 handoff 跑通

### 当前不建议一口气做满的
- 完整开发阶段 Skill 包
- 完整测试执行阶段 Skill 包
- 全生命周期一次性细颗粒拆完

---

## 十一、当前 v0 草案的核心判断

### 我当前的建议是：

- 生命周期先按 5 大阶段组织
- 第一阶段“产品 / 需求”先拆成 4 个子阶段
- POC / 验证流作为产品 / 需求阶段里的一个子阶段存在
- 阶段骨架统一，差异通过能力包裁剪来体现，而不是先做 2B / 2C 二分
- 产品 / 需求阶段应显式产出优先级信息，并设置阶段内部里程碑
- 原型设计分两类：
  - 需求阶段做验证型原型
  - 设计阶段做交付型原型
- 先做“产品 / 需求阶段 Skill 包”MVP，而不是一开始把全生命周期全部落地

---

## 十二、给评审同事的问题

建议产品同事重点评审这些问题：

1. 产品 / 需求阶段拆成 4 个子阶段是否合适？
2. “需求验证 / 概念验证”是否应独立成子阶段？
3. 原型设计在需求阶段与设计阶段的分工是否合理？
4. 优先级机制放在第 2/3 子阶段是否合理？
5. 当前产品 / 需求阶段的里程碑划分是否合理？
6. 当前输入物 / 输出物设计是否太重或太轻？
7. 这个包是否足以作为第一个 MVP 落地对象？
