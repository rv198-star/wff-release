# v1.2 Operation Risk Tiering And Source Obligations

Date: 2026-04-25

## Purpose

This document defines the shared operation risk model used by P2 and P3.

P2 uses it to classify every public operation and to decide which operation-bound design sources must be produced, registered, or explicitly marked review-bound.
P3 consumes the P2-authored risk tier and source obligations to decide which behavior cards and source bindings are required before implementation-depth can be claimed.

P3 must not independently relax source requirements. A `not_required` decision is valid only when it is derived from the P2-authored operation risk row that was produced from this shared rule. If that row is missing, ambiguous, or inconsistent with P3's recomputation, P3 must report a review-bound gap instead of downgrading the source requirement.

Short rule:

> P2 and P3 must classify operation risk from the same rule set.
>
> If P2 and P3 disagree on high-risk operations, P2 diagrams and P3 behavior cards will not line up.

Operational rule:

> `risk-tiered` means `P2-authored and P3-validated`, not `P3-local heuristic relaxation`.

## Operation Risk Tiers

### `HR-MUTATION`: Mutating or durable-state operation

An operation is high-risk when it creates, updates, deletes, archives, confirms, exports, submits, starts, completes, retries, replays, schedules, or otherwise mutates durable business state.

Typical signals:

- HTTP method is `POST`, `PUT`, `PATCH`, or `DELETE`
- operation id starts with `Create`, `Manage`, `Update`, `Delete`, `Archive`, `Submit`, `Start`, `Complete`, `Record`, `Confirm`, `Export`, `Retry`, or `Replay`
- operation writes DB/cache/queue/storage/external state
- operation changes status, version, lifecycle, ownership, assignment, or audit-visible evidence

P2 required sources:

- `P2-CTR-*`
- `P2-FLOW-*`
- `P2-STATE-*` when lifecycle/status/version changes
- `P2-SEQ-*` when persistence/audit/event/external ordering matters

P3 required behavior-card posture:

- behavior card required
- missing required P2 sources are `review-bound` or blocking depending severity
- P3 must not convert a missing required source into `not_required` based on its own local heuristic

### `HR-BOUNDARY`: Trust, permission, tenant, audit, compliance, or money-like boundary

An operation is high-risk when incorrect behavior can expose, hide, authorize, deny, confirm, charge, settle, audit, or materially misrepresent protected business truth.

Typical signals:

- tenant isolation, permission, identity, role, policy, entitlement
- audit trail, compliance, legal, financial, billing, approval, evidence retention
- sensitive fields, PII, secrets, regulated record, buyer/operator trust boundary
- failure can cause unauthorized visibility or incorrect official record

P2 required sources:

- `P2-CTR-*`
- `P2-FLOW-*` for boundary decision path
- `P2-SEQ-*` when multiple components participate in boundary enforcement
- `P2-STATE-*` only if a lifecycle/status transition is involved

P3 required behavior-card posture:

- behavior card required
- tests must include allowed/denied or visible/hidden boundary behavior where applicable
- P3 must inherit source obligations from the P2 row and may only add stricter review notes

### `HR-ORCHESTRATION`: Cross-module, async, replay, idempotency, concurrency, or external dependency

An operation is high-risk when correctness depends on ordering, retries, handoff continuity, replay identity, idempotency key, concurrency/version guard, async completion, external adapter, cache freshness, or queue/event behavior.

Typical signals:

- operation crosses service/module/domain boundary
- queue/event/async workflow, callback, webhook, scheduler, batch, or external dependency
- idempotency, replay, retry, stale version, conflict, lock, cache invalidation/freshness
- operation participates in P2 replay/RBI/review-bound path

P2 required sources:

- `P2-CTR-*`
- `P2-FLOW-*`
- `P2-SEQ-*`
- `P2-RP-*` when replay/handoff continuity is part of the design
- `P2-STATE-*` only if a lifecycle/status transition is involved

P3 required behavior-card posture:

- behavior card required
- tests must cover ordering/retry/replay/idempotency/concurrency semantics where applicable
- P3 must not drop `P2-SEQ-*` or replay obligations even when a generated implementation can pass simpler tests

### `MR-READ-SENSITIVE`: Sensitive read/list/detail operation

A read/list/detail operation is medium-risk when it is read-only but still crosses a sensitive boundary or returns decision-critical business truth.

Typical signals:

- HTTP method is `GET`
- operation id starts with `Get` or `List`
- read includes tenant/permission/sensitive filtering
- read returns review, audit, compliance, recommendation, official summary, or cross-module synthesized truth
- incorrect result could mislead a business decision, but does not mutate state

P2 required sources:

- `P2-CTR-*`
- `P2-FLOW-*` when filtering/derivation/cross-module ownership is non-trivial
- `P2-SEQ-*` only when multiple components or freshness guarantees materially affect correctness
- `P2-STATE-*` not required unless the read depends on lifecycle transition semantics

P3 required behavior-card posture:

- behavior card may be required when P1/P2 marks it core, sensitive, or review-bound
- otherwise lightweight contract/scenario tests may be enough
- missing state/sequence may be `not_required` only when the P2-authored row says those sources are not required by the shared risk rule

### `LR-SIMPLE-READ`: Low-risk deterministic read or support operation

An operation is low-risk only when all conditions hold:

- read-only or deterministic transform
- no durable mutation
- no permission/tenant/audit/compliance/money-like decision beyond ordinary access already covered elsewhere
- no cross-module orchestration or async/external/cache correctness requirement
- no idempotency/replay/concurrency/version semantics
- not on a P1 core scenario, P2 RBI, P2 review-bound, or high-risk boundary path
- public contract drift has low downstream rework cost

P2 required sources:

- `P2-CTR-*`
- upstream P1 trace if applicable

P3 required behavior-card posture:

- behavior card not required by default
- contract tests and lightweight scenario coverage are enough
- P3 may treat flow/sequence/state as `not_required` only when P2 classified the operation as `LR-SIMPLE-READ` or emitted equivalent source obligations from this shared rule

## Shared Trigger Table

| Trigger | Tier | P2 source obligation | P3 obligation |
|---|---|---|---|
| durable mutation | `HR-MUTATION` | contract + flow + state; sequence if ordering matters | behavior card required |
| status/lifecycle transition | `HR-MUTATION` | contract + state mandatory | behavior card required |
| permission/tenant/identity | `HR-BOUNDARY` | contract + flow; sequence if multi-component | behavior card required |
| audit/compliance/financial/evidence | `HR-BOUNDARY` | contract + flow; sequence if persistence ordering matters | behavior card required |
| queue/event/async/external | `HR-ORCHESTRATION` | contract + flow + sequence | behavior card required |
| replay/idempotency/retry/concurrency/version | `HR-ORCHESTRATION` | contract + flow + sequence; state if lifecycle affected | behavior card required |
| sensitive/cross-module read | `MR-READ-SENSITIVE` | contract + flow when filtering/derivation is non-trivial | behavior card conditional |
| simple read/list | `LR-SIMPLE-READ` | contract only | no behavior card by default |

## Alignment Requirements

P2 must emit a per-operation risk row for every `P2-CTR-*` public operation:

- `operation_id`
- `risk_tier`
- `risk_triggers`
- `required_source_types`
- `bound_source_ids`
- `review_bound_missing_sources`
- `not_required_source_types`
- `classification_rationale`

The P2 row is the contract consumed by P3. It must be operation-addressable and traceable back to the shared rule, not embedded only in prose or implied by diagram counts.

P3 must consume the same fields when deciding behavior-card and source-binding requirements.

P3 may recompute the tier only to validate the P2 row. It must classify any disagreement as `risk_tier_mismatch` and hold the operation review-bound until P2/P3 are reconciled.

Normative order:

1. P2 classifies the operation risk tier and required source types.
2. P2 emits and registers operation-bound source IDs for required source types.
3. P3 consumes the P2 risk/source obligation row.
4. If P3 recomputes risk independently, it is only a consistency check. Any mismatch against P2 is `risk_tier_mismatch`, not a reason to silently downgrade requirements.
5. P3 may mark a source type `not_required` only when the P2-authored row, produced from the shared risk table, says it is not required for that operation.
6. If the P2 row is missing, P3 must mark `p2_operation_risk_row_missing` and treat behavior-source requirements as review-bound; it must not fall back to a private P3 risk model.


## Anti-Patterns

- P2 passes because it has enough diagrams by count, while high-risk operations lack operation-bound sources.
- P3 marks every GET/List as needing state/sequence without checking whether those sources are required by the shared P2/P3 risk tier.
- P3 unilaterally downgrades an operation to avoid missing P2 source obligations.
- P2 and P3 use different definitions of high-risk.
- A stage-level sequence diagram is treated as operation-bound authority without explicit operation/source binding.
- An aggregate state diagram is treated as operation transition authority without an explicit operation-to-transition binding.
