# 故事地图保持产品全景

- 类型：概念
- 适用阶段：需求分析 / 需求结构化
- 来源：`sources/books/original/用户故事地图.md`（Martin Fowler序、前言、第1章）
- 摘要：故事地图的核心价值不是把需求写成更多卡片，而是在需求拆分过程中持续保留产品全景图，避免只见局部功能而丢失用户目标与整体体验。

## 输入条件
- 已有初步的用户目标、业务场景或需求讨论材料
- 团队正在把需求拆成更小的工作单元

## 关键步骤
1. 先讲完整的用户故事，而不是先写零散条目。
2. 识别用户经历中的关键步骤，作为故事地图主干。
3. 在主干之下再展开细节、选项和任务。
4. 用地图持续检查拆分后的内容是否仍能表达完整体验。

## 输出结果
- 一个能同时表达“整体体验”和“局部切分”的结构化需求视图
- 为后续 MVP 切片、优先级讨论和设计沟通提供全景基线

## 适用边界
- 适用于需要在拆分需求时保持上下文的产品/业务场景
- 不适用于只需要记录静态字段、且不存在用户行为流的极简规范型材料

## 常见错误
- 只写零散用户故事，不维护全景结构
- 过早陷入卡片细节，导致用户目标消失
- 把故事地图误用为单纯的任务清单

## 关联模板
- 需求分析产物
- 需求拆解产物
- MVP 切片说明

## 下游使用建议
- 可被未来的“需求分析 Skill”引用，用于定义何时必须先构建需求全景图
- 可被“需求拆解 Skill”引用，作为拆分前的结构化前提
- 可强化需求阶段 gate：若需求已碎片化但无全景结构，则不得直接进入设计阶段
