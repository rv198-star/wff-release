# AGENTS.md

## 默认姿态

> 遇事不要慌，先搞清楚情况再说。

本项目不是要 agent 更快给答案，而是要 agent 先判断自己面对的是什么问题，再选择合适的方法镜头。

一句话理解：

> `SELA` 看整体趋势与局部优势的关系；`3L5S` 处理问题从发现到落地；`EDSP` 处理悬而不决的结构判断；`WAE` 处理 Agentic / Workflow / Evidence 的控制边界；`TVG` 和 `tplan` 是手动或 workflow 路由工具，不靠普通表达被动唤起。

## 场景化唤起表

先看用户普通表达，再选方法。用户不需要知道方法名，但不是所有 Mindthus 工具都允许被动唤起。

### 自动唤起

这些 skill 可以从普通场景信号中自动选择：

| 用户常见表达 / 场景信号 | 默认 skill | 为什么 | 不触发边界 |
| --- | --- | --- | --- |
| "这个需求越做越乱"、"反复返工"、"问题不清楚"、"任务太大"、"拆完还是不可执行" | `3l5s` | 需要先把混乱信号变成可证伪问题，或把大问题拆成可验证行动。 | 明确的一步操作不需要套 3L5S。 |
| "两个方案都对"、"边界到底在哪"、"这是伪二选一吗"、"定性判断很难"、"趋势难判" | `edsp` | 需要用极限推演建立结构坐标，再判断现实场景落点。 | 已有明确证据和确定规则时，不用 EDSP 代替验证。 |
| "旧方案局部优势很强"、"新方案系统效率更高"、"费效比正在变"、"旧范式 vs 新范式" | `sela` | 需要防止被真实但局部的优势拖住，看系统级费效比是否形成数量级差异。 | 涉及伦理底线、不可逆伤害或过渡期保护时，不能机械套 SELA。 |
| "这里该脚本控制还是 agent 判断"、"证据要不要记录"、"evidence 要不要记录"、"workflow 太重吗"、"agent 会不会漂"、"控制边界不清" | `wae` | 需要判断 Workflow / Agentic / Evidence 谁该控制哪部分工作。 | 低风险、确定性的格式化工作不需要套 WAE。 |

## Skill 地图

### `skills/sela/` — SELA

顶层战略决策参考，用来在战略方向上识别整体与局部的关系。

当局部优势真实、优秀、令人留恋，但系统级费效比正在形成数量级优势时，用 `SELA` 检查自己是否在做短视选择。

适合：重大趋势判断、战略取舍、旧范式 vs 新范式、手工卓越 vs 系统效率。

边界：它不是结论机器；不可逆伤害、尊严伦理底线、过渡期价值等场景不能机械套用。

### `skills/3l5s/` — 3L5S

通用问题处理内核，用来发现问题、定义问题、解决问题。

当问题还不清楚时，用 `Discovery -> Definition` 从混乱现象中收敛出可复述、可定位、可证伪的问题。当问题已经明确但过大、过复杂、不可执行时，用 5S / BTGSB 拆成可验证、可排期、可执行的任务。

适合：工单判断、问题诊断、复杂任务拆解、执行反复返工后的回查。

边界：不要为了形式跑完整三层；如果问题已明确，单层 BTGSB 就是高价值形态。

### `skills/edsp/` — EDSP / 极限推演与场景投影

定性判断镜头，用来处理悬而不决、难以决断的结构判断。

当 A/B 都像对、命题本身可能有问题、边界不清、趋势难判时，用 `EDSP`。先用 ED 把关键变量推到极端，建立结构坐标，再读取现实漂移方向；只有在结构坐标稳定后，才用 SP 处理具体场景选择。

适合：伪二选一、趋势判断、结构边界、原则落地到具体场景。

边界：它不能替代证据、领域研究、运行时证明或 stakeholder 判断。

### `skills/wae/` — WAE

控制边界镜头，用来处理 Agentic / Workflow / Evidence 的控制权之争。

当不确定某段工作该由流程控制、由 agent 判断，还是由证据约束时，用 `WAE`。它关心的不是“问题是什么”，而是“谁或什么应该控制这部分工作”。

适合：设计 workflow、agent、脚本、审查机制、证据门槛；判断哪里该自动化，哪里必须保留判断。

边界：低风险、确定性的格式化工作不需要套 WAE。

### `skills/tvg/` — TVG

工具型思考增强器，用来处理 AI 产物结构完整但实质浅薄的问题。

当用户显式要求使用 TVG，或 workflow 明确路由某个 AI 生成的文档、计划、方法、skill 或模块做深度审查时，用 `TVG` 做一轮有目标的价值深化。普通的“文档有点空”“判断薄”“需要深化”反馈本身不被动唤起 TVG。

适合：已经成形但缺少证据、取舍、失败路径、边界、下游可用性的 bounded artifact。

边界：TVG 不负责战略方向，不重开整个问题空间；它只深化命名清楚、有边界的产物。脚本只能做记录和校验，不能替代 judgment。

### `skills/tplan/` — tplan

Mission 运行时和任务控制平面，用来管理长期目标的状态、权限、证据边界、恢复和决策 hooks。

只有在用户显式要求使用 tplan，或 workflow 明确路由到 Mission runtime 时使用。长期目标、项目计划、todo、plan、几十步、切换任务、记录证据、保留任务状态、任务树、恢复现场、后续轮次、自动推进等自然语言需求本身不被动唤起 tplan；需要时先向用户确认是否进入 tplan，而不是自行初始化 Mission。

适合：需要 durable Mission 状态、Task/SubTask/Step 结构、human-in-loop、脚本化状态迁移和停止报告的长期执行。

边界：tplan 不是普通 todo；不因任何 plan 字样自动创建 Mission；不被长期目标表述被动唤起。

## 组合方式

这些 skill 不需要串成固定流程。常见组合是：

- 战略判断前，用 `SELA` 防短视。
- 具体处理问题时，用 `3L5S` 做默认问题内核。
- 3L5S 中遇到模糊结构判断，用 `EDSP`。
- 任何方法里需要分配控制权，用 `WAE`。
- 显式要求使用 TVG 或 workflow 明确路由时，用 `TVG` 加深有边界的产物。
- 显式要求使用 tplan 或 workflow 明确路由时，用 `tplan` 承载 Mission runtime。
