# 设计 / 架构阶段 Skill 包草案（v0）

## 文档目的

这份文档是给人类评审的 `v0` 大纲草案，用于说明设计 / 架构阶段建议如何拆子阶段、每个子阶段包含哪些 Skill 结构、输入物、输出物、SOP 与引用素材。

---

## 先看这页：设计 / 架构阶段总览

### 建议拆成 4 个子阶段 / Stage Skills

| 子阶段 | Stage Skill 名称（建议） | 目标 | 主要输入物 | 主要输出物 |
|---|---|---|---|---|
| 1 | 架构定义 / 系统边界 | 明确系统边界、关键约束、核心能力结构 | 产品 / 需求 handoff、MVP 边界、关键约束 | 架构设计说明、系统边界说明、关键决策清单 |
| 2 | 模块 / 领域 / 服务拆分 | 明确模块边界、领域边界、服务候选与依赖关系 | 架构设计说明、需求结构图、关键场景 | 模块划分说明、依赖关系图、服务/能力草案 |
| 3 | 数据 / 存储 / 接口设计 | 把结构下沉到数据模型、接口契约和交互边界 | 模块划分说明、关键流程、领域/服务边界 | 数据模型说明、接口契约、交互说明 |
| 4 | 设计收敛 / 交付型原型 | 在完成架构、模块、数据与接口设计后，用原型和结构表达收敛设计结果，形成可交付输入 | 架构与接口设计结果、关键风险 | 交付型原型、设计验证记录、设计 handoff 包 |

### 每个 Stage Skill 最小包含 4 类文档
- `skill-contract.md`
- `stage-sop.md`
- `output-template.md`
- `source-cards.md`

## 二、原型设计在本阶段的位置

设计 / 架构阶段中的原型是：
- 表达型原型
- 交付型原型

它不同于产品 / 需求阶段的验证型原型。

这里的原型目的主要是：
- 设计表达
- 实现交接
- 测试输入

### 顺序关系说明

在当前草案里，不建议把“架构设计”和“交付型原型”理解为两条并行主线。

更准确的理解是：

1. 先完成：
   - 架构定义 / 系统边界
   - 模块 / 领域 / 服务拆分
   - 数据 / 存储 / 接口设计
2. 再进入：
   - 设计收敛 / 交付型原型

也就是说，交付型原型更像是当前设计阶段内部的**后段收敛动作**，而不是与前序架构设计平级并行的一条主线。

### 当前边界原则

当前不建议把“人类与 AI 反复讨论并磨出界面”的全过程，强制纳入本项目的 Skills 覆盖边界。

更合理的边界是：

- 本项目的 Skills 负责产出：
  - 结构化设计意图
  - 页面 / 模块结构说明
  - 交互与状态规则
  - 视觉约束说明
  - 原型验收标准

- 人类 + AI / 设计工具 / 外部原型工具 负责：
  - 多轮磨稿
  - 高保真视觉收敛
  - HTML 原型生成过程本身

- 但一旦原型被确认：
  - 已确认的 HTML 原型（或同等可运行界面原型）应被视为设计阶段正式产出物
  - 并可作为开发阶段的正式输入物

### 当前推荐做法

设计阶段不强制规定原型来源，但要求明确：
- 原型是否已确认
- 原型对应哪些页面 / 模块 / 状态
- 原型如何被下游开发消费

也就是说：

> 原型生成过程可以在 Skills 体系边界之外进行，但确认后的原型成果必须进入流程之内。

## 三、主要引用素材（当前已知）

- 《启示录》：产品定义、探索前置、协作边界、产品验证
- 用户画像主题组：系统模块、平台架构、数据模型、业务 handoff
- 《精益产品开发》：价值流动、方法治理、上下文适配

---

## 四、当前 v0 判断

- 该阶段建议先拆成 4 个子阶段
- 重点不是一次性细到开发任务级别，而是先把“设计 / 架构 -> 开发输入”的桥打通
- 当前素材足够支撑 v0 大纲，但还不足以把每个 Stage Skill 细化成正式版
