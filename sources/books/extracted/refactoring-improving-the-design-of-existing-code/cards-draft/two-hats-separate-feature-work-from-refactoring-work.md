# 两顶帽子：把“加功能”和“重构”当成两种不同模式

- 类型：方法 / 边界
- 适用阶段：开发实现
- 来源：`sources/books/original/phase-others/refactoring-improving-the-design-of-existing-code.md`（第2章 2.2）
- 摘要：在开发中交替进行两种行为：**添加功能**与**重构**。添加功能时不应改动既有逻辑（除非必要）；重构时只改结构不加功能。清楚自己戴哪顶帽子，可以显著降低混乱与回归。

## 输入条件

- 你正在实现需求、修 bug 或改善结构

## 关键步骤

1. 在开始前声明当前帽子：feature-hat 或 refactor-hat。
2. feature-hat：通过新增/修改测试并让测试通过来衡量进度；尽量避免无关结构调整。
3. refactor-hat：不增加功能，只调整结构；只在接口变化必须时修改测试。
4. 频繁切换帽子是正常的，但必须显式切换（避免“边写功能边顺手大改结构”）。

## 输出结果

- 更可控的变更边界；更可审计的意图与验证路径

## 适用边界

- 如果团队强依赖“提交粒度”来做审计，可以在提交层面做更细分；但原则仍是：意图与验证方式必须清晰。

## 常见错误

- 在同一段时间里同时做功能与重构，导致测试失败时无法判断是功能错还是结构改坏了
- 用“重构帽子”逃避写测试或证明行为保持一致
