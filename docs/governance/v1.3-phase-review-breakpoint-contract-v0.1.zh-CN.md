## 目的

本文定义 P1-P4 的 `v1.3.1` phase review breakpoint contract。

该合同在 phase completion 时增加明确的 human oversight opportunities，同时保持已接受的 Agent / Skills-led autopilot mainline。它不把 human-led continuous collaboration 变成默认交付模型。

## 控制原则

每个 P phase 都必须在完成时主动呈现一个 review opportunity。

breakpoint 是 review 和 routing surface。它不是 stage artifact editor，不是 proof gate，也不是新的 phase。

phase 只有在 breakpoint 记录以下状态之一后才能继续。

展示给人类看的 review 动作必须使用中文标签。稳定英文 key 可以保留在结构化记录中，供 scripts、tests 或跨语言引用使用：

- `批准`（`approved`）：reviewer 在已声明 evidence ceiling 内接受 phase output。
- `带保留项批准`（`approved-with-review-bound-items`）：reviewer 接受继续下游推进，同时保留具名 open items。
- `要求修改`（`changes-requested`）：reviewer 请求在同一 phase 内进行 bounded changes。
- `要求返回`（`return-requested`）：reviewer 把问题发回 upstream 或当前 owning phase，并说明 rerun boundary。
- `提供干预输入`（`intervention-supplied`）：reviewer 提供新的 source truth、constraints、artifacts 或 decisions；phase 必须把它们作为新输入并 rerun 或 remediate，不能静默 patch final artifacts。
- `明确跳过审核`（`explicitly-skipped`）：reviewer 放弃 review；phase 只有在记录 skip、可用的 reviewer identity、reason 和不变的 claim ceiling 后才能继续。

## 必需 Breakpoint Payload

phase completion 时应呈现：

- phase output root 和 primary artifact list；
- phase summary 和 intended downstream consumer；
- 该 phase 的 value-bearing judgment；
- open `review-bound` items 及其 downstream effect；
- evidence basis 和 missing evidence；
- 当前 formal claim ceiling；
- recommended next step；
- reviewer reject、intervene 或 request return 时的 minimum rerun / return boundary。

payload 可以放在 phase execution report、handoff package、closeout summary，或一个小型 review-breakpoint note 中。如果现有 phase output 已经携带这些字段，不要求新增 artifact。

## Non-Mutation Rule

breakpoint 不能静默修改 generated stage artifacts、proof outputs 或 upstream truth。

如果 review 改变 product truth、architecture truth、implementation behavior 或 validation evidence，该变化必须作为新输入记录，并路由到 owning phase。

breakpoint 允许：

- 记录 reviewer decision 和 notes；
- 记录 reviewer 提供的新输入；
- 把 remediation 路由到 owning phase；
- 封顶或保留 claims；
- 带原因地明确 skip review。

breakpoint 禁止：

- 手工编辑 retained generated proof snapshots；
- 不经 rerun / remediation 就重写 upstream artifacts；
- 把 missing evidence 写成 pass；
- 把 reviewer optimism 当成 evidence；
- 在没有 external evidence 时声明 production approval、online UAT、go-live、real owner sign-off、rollback readiness、monitoring handover 或 production risk acceptance。

## Phase-Specific Intent

| Phase | Breakpoint asks | Downstream effect |
| --- | --- | --- |
| P1 | product / business / user / acceptance truth 是否足够有价值、足够清楚，可以进入 architecture？还有什么 review-bound？ | 继续 P2、回到 P1 deepening，或当 gap 属于 P1 前置输入时推迟到 `v1.3.5` source-truth admission。 |
| P2 | architecture / delivery path / constraint response / implementation-facing handoff 是否足够支撑 implementation？ | 继续 P3、执行 bounded P2 remediation，或当缺失 product truth 会迫使 design invention 时返回 P1。 |
| P3 | delivery 是否在 evidence ceiling 内足够 runnable、足够 meaningful，可以进入 validation？ | 继续 P4、执行 bounded P3 remediation，或因 upstream truth gap 返回 P2/P1。 |
| P4 | validation closure 是否诚实，risks 是否可见，claim ceiling 是否正确？ | 关闭 testing-validation、保留 review-bound items、路由 remediation，或仅在明确请求时进入 Stage-04。 |

## Human Oversight Boundary

breakpoint 给 human 或 reviewer 一个 decision point。

它不要求 human 共同编写每个 stage、持续指挥 Agent，或替代 Agent / Skills judgment。

Autopilot 仍是 mainline；breakpoint 是 oversight 和 intervention opportunity。

## Runtime Mirror Rule

最小 runtime-facing mirror 属于官方 phase entry skills：

- `skills/wff-req/SKILL.md`
- `skills/wff-arch/SKILL.md`
- `skills/wff-impl/SKILL.md`
- `skills/wff-validation/SKILL.md`

不要把完整合同复制到每个 stage pack。

只有后续子工单证明 phase entry skill 不足以执行运行时规则时，才镜像到 stage SOPs、templates 或 reference packages。

## 验收问题

如果本文正确把 phase review breakpoints 定义为以下内容，则可以接受 `WO-119A.04`：

- P1-P4 必需的 phase-completion review opportunities；
- oversight 和 routing surfaces，而不是 continuous human-led collaboration；
- 不静默修改 generated artifacts 或 upstream truth 的 decision points；
- 使用 `批准`、`要求修改`、`提供干预输入`、`要求返回`、`明确跳过审核` 等中文人类可见动作标签记录决策，同时保留结构化记录可用的稳定 protocol keys；
- claim-ceiling preservers，而不是 proof 或 production-approval substitutes；
- 范围足够窄，目前只需要在四个官方 phase entry skills 中做最小 runtime mirror。
