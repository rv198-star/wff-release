# v1.2 设计原则（v0.1）

## 这份文档是做什么的

这份文档定义 `v1.2` 期间的**最高优先级设计原则**。

它的用途不是解释历史，也不是列零散想法。
它的用途是：

- 在 `v1.2` 开发过程中反复阅读
- 在方案分歧时作为纠偏依据
- 在实现不断膨胀时作为收口依据
- 在评审时防止标准漂移

如果后续某个实现、脚本、Skill、评分规则、交付物设计与这份文档冲突，应先怀疑实现跑偏，而不是先稀释原则。

---

## 1. 核心判断

`v1.1` 建立了一个有价值但偏重的基线：

- 产出物意识强
- gate / refusal / handoff 意识强
- traceability 意识强
- 双案例证明“系统能工作”

但 `v1.1` 也暴露出明显问题：

- 主线被脚本控制面冲散
- `Phase-3` 复杂度与控制面过厚
- retained proof 与 runtime closure 语义纠缠
- 结构化正确不等于业务深度足够

因此，`v1.2` 不是继续在 `v1.1` 上补更多控制脚本。

`v1.2` 的方向是：

> **保留秩序、审计、交付和可追踪性；收短主线；降低控制面；把一部分被冻结的思考过程还给 bounded agent loops。**

---

## 2. 使用规则

凡是进入 `v1.2` 设计或实现工作，优先用这份文档回答 4 个问题：

1. 这项设计是否让主线更清楚，还是更模糊？
2. 这项设计是在降低机械成本，还是在冻结思考过程？
3. 这项设计是否减少了下游猜测，还是把不确定性推给了下游？
4. 这项设计是否真的提高了交付质量，还是只是增加了控制表面？

如果回答不清，说明设计还没有收敛。

`v1.2` 还有一条默认阅读补充：

- 如果问题涉及 `P1/P2` 该用什么思维模式、loop 强度、checklist 强度，先读：
  - `docs/v1.2-phase-thinking-mode-split-v0.1.md`
- 如果问题涉及 `P2` 如何平衡架构深度、交付效率、ROI、可扩展性和过度设计风险，先读：
  - `docs/v1.2-p2-delivery-minded-architecture-principles-v0.1.md`
- 如果问题涉及 `P3` 如何平衡真实后端实现、runtime evidence、formal state、generated runtime 和 optional lanes，先读：
  - `docs/v1.2-p3-backend-delivery-principles-v0.1.md`
- 如果问题涉及某一阶段该更偏 `Workflow` 还是更偏 `Agentic`、哪些应留在主线、哪些该退出主线，先读：
  - `runtime-deps/mindthus/source/skills/wae/SKILL.md`
- 如果问题涉及 `v1.2` 应如何评分、如何给出 `PASS / RETURN / BLOCKED`，先读：
  - `docs/v1.2-mainline-scoring-and-acceptance-matrix-v0.1.md`
- 如果问题涉及复杂诊断、trace/source binding、resolver false negative、根因不清或 release-risk 判断，先读：
  - `runtime-deps/mindthus/source/skills/3l5s/resources/three-layer-recursive-loop.md`

## 3. Principle 01: 以交付物为主线，不以脚本为主线

`v1.2` 的主线必须按**交付物**来定义，而不是按脚本序列来定义。

每个阶段都必须首先回答：

- 唯一主交付物是什么
- 下游消费者是谁
- 下游必须拿到什么
- 最小必经流程是什么
- 最小 gate 是什么
- 哪些是可选分支
- 哪些可以延期

如果一个阶段无法先用这些问题讲清，就不允许先去扩脚本控制面。

短规则：

> **先定义交付物，再定义流程；先定义流程，再决定脚本。**

---

## 4. Principle 02: 主线要短、清楚、默认可走

主线不是“所有能力的合集”。
主线是默认情况下最先被走通、最容易解释、最稳定的一条交付路径。

`v1.2` 的默认要求是：

- 主线应能被一句话讲清
- 首轮心智模型中只出现必要步骤
- 可选复杂性不得压过主线

判断标准：

- 新协作者能否快速说出 `P1 -> P4` 每阶段的唯一主交付物
- 新协作者能否快速说明什么是 default，什么是 optional
- agent 是否能快速识别“先走哪条线”

如果不能，说明主线还没有收短。

---

## 4A. Principle 02A: 当系统效率形成数量级优势时，不用局部优势逆转主线

`v1.2` 需要承认一个长期事实：

> **当某条路线在系统级费效比上形成数量级优势时，另一条路线的局部优势即使真实存在，也通常不足以决定主线方向。**

这里说的系统级优势，不是单轮“更快”这么简单，而是全链路的：

- 维护成本
- 平均产出稳定性
- 平均协作理解成本
- 迭代速度
- 泛化复用能力
- 训练/接手成本

对当前项目，这条原则的直接含义是：

- 不能因为 `v1.1` 某个 retained artifact 在局部表达上更尖锐，就逆转回 script-heavy 主方向
- 不能用少数案例的“更像专家手感”否定更可维护、更可扩展、更容易解释的主线
- `v1.2` 的默认目标仍然是：在跨过最低可接受质量线之后，押注更优的系统结构，而不是眷恋局部优势

### 使用方式

这条原则最大的风险，是被机械套用成：

- 看到 `局部优势`
- 直接推出 `应忽略局部优势`

这是错误的。

它的正确用法不是自动出结论，而是作为一个高层提醒：

- 我们是否只看到了局部最优样本
- 我们是否低估了系统级平均表现
- 我们是否把“局部更锋利”误判成“应该逆转主线”
- 我们现在为某个局部优势辩护，是不是部分出于熟悉感、偏爱、路径依赖，或者被局部卓越打动了

因此它更像：

- `decision lens`
- `review perspective`
- `anti-overreaction reminder`
- `anti-attachment check`

而不是：

- `if X then Y` 的机械规则
- 取代其他设计原则的上位开关

它最重要的实际用途是：

- 当团队开始为某个局部优势持续辩护时
- 用这条原则强制自己暂停一下
- 检查这个辩护是不是带有情感驱动或路径依赖

检查完之后，结论**完全可能仍然是保留局部优势**。

但那应该是：

- 清醒地保留
- 有边界地保留
- 明确知道自己为什么保留

而不是：

- 无意识地保留
- 因为熟悉所以保留
- 因为它曾经很亮眼，所以默认它应该统治主线

短规则：

> **先用它检查自己是不是在情感性地为局部优势辩护，再把它纳入综合判断；不要把它当成自动结论机。**

### 三个固定追问

#### 1. 对比是否公平

必须优先问：

- 是不是 `best A vs average B`
- 还是 `average A vs average B`

决定主线方向的，不应是最优样本，而应是可重复的平均表现。

#### 2. 优势是否可规模化

如果某种优势依赖：

- 少数人记住复杂脚本表面
- 高度熟练的隐性经验
- 对特定案例的长期记忆
- 难以迁移的人工调参

那它更像局部卓越，而不是系统优势。

`v1.2` 应优先投资可规模化的优势。

#### 3. 劣势是否处于可接受容忍区间

很多被直觉放大的缺点，系统层面其实是可容忍的。

因此必须问：

- 该方案的劣势是否真的跌破最低可接受质量线
- 还是只是没有达到局部最优

只要仍高于最低质量门槛，就不应用局部最优的缺失否定系统方向。

### 护栏

这条原则只有在满足以下前提时才成立：

1. 系统方案已经跨过最低可接受质量线
2. 费效比按全链路计算，而不是只看单轮输出
3. 被保留的局部卓越仍可作为显式 optional / expert lane 存在，而不是被假装不存在

这条原则不允许：

- 以“效率”之名降低业务深度下限
- 以“主线收短”之名放弃关键质量门槛
- 以“系统优势”之名掩盖已经跌破可接受线的真实退化

短规则：

> **主战场押注系统效率；局部卓越保留为可选分支；不要用局部优势逆转主线。**

## 5. Principle 03: Workflow 保秩序，不模拟智能

`Workflow` 的职责是：

- entry / exit contracts
- hard gate checks
- deterministic transforms
- artifact existence checks
- state recording
- evidence aggregation
- release/export assembly

`Workflow` 不应主要负责：

- 代替业务分析
- 代替 trade-off 判断
- 代替分支选择
- 代替对 demo 感的识别
- 代替业务细节深化

判断一个脚本值不值得存在，只问一句：

> **这个脚本是在降低机械成本，还是在冻结思考过程？**

保留前者，警惕后者。

---

## 6. Principle 04: Agentic 要负责深化，不负责失控

`Agentic` 不等于自由发挥。
`Agentic` 要负责的是高解释自由度、强判断依赖的那部分工作：

- 澄清问题
- 识别关键未知
- 比较 `2-3` 个方案
- 阐明 trade-off
- 识别 demo 风险
- 进行对抗式补漏
- 做 evidence-backed deepening

但 `Agentic` 必须是**有界循环**，而不是无限脑暴。

默认规则：

- `1` 个结构化 draft
- 最多 `3` 轮 deepening
- 最终必须进入：
  - `freeze`
  - `freeze-with-review-bound-warning`
  - `return-remediate`
  - `blocked`

短规则：

> **Loop 是为了深化判断，不是为了拖延收敛。**

### P1 truth order 必须固定

对 `P1/P2`，`v1.2` 还要再固定一条顺序：

1. `ordinary real operating baseline`
2. `product world selection`
3. `business/release truth`
4. `planning/control truth`

含义是：

- 先判断真实经营基线属于什么世界，再决定产品到底是什么
- 先把业务价值、`why-this-not-that`、类别 framing 压进 `business/release truth`
- 再把对象链、状态交接、handoff/control 信息压进 `planning/control truth`
- `P2` 只能单向消费这两个 truth pack，不能反过来用 planning truth 重新发明业务主语义

短规则：

> **先定业务世界，再定产品世界；先压业务真相，再压控制真相。**

---

## 7. Principle 05: 结构合格不等于业务深度足够

`v1.2` 必须明确区分三类质量：

- `结构充分性`
- `下游可消费性`
- `业务深度充分性`

一个产出物可以：

- 结构完整
- 术语正确
- 验收标准看似齐全

但仍然像 `demo`。

`demo 感` 的本质不是格式差，而是：

- 核心流程仍是标题级
- 操作密度不足
- 状态变化不足
- 异常路径不足
- 现实约束缺失
- 下游还要自行补关键真相

所以 `v1.2` 不允许只以结构正确来判定阶段成功。

短规则：

> **complete 不是结束，dense enough 才能 handoff。**

---

## 8. Principle 06: 收敛标准不能只有“下游不需要发明关键真相”

`v1.2` 的深度 loop 不能靠“感觉已经想得够多了”来停。

但同样重要的是：

> **“下游不需要发明关键真相”只能作为必要条件，不能作为唯一 freeze 标准。**

因为如果上游一开始就把世界切得过薄、场景选得过少、业务基线定得过浅，
下游即使完全不需要“再猜”，也仍然可能只会忠实实现一个很薄的 `demo world`。

所以 `v1.2` 的收敛至少要满足两层门槛：

### 第一层：handoff sufficiency

核心未知项已被压缩到下游不需要再发明关键真相。

### 第二层：business baseline sufficiency

当前阶段已经主动识别并覆盖了目标业务里**应进入主线基线的关键场景集合**，
而不是只把一个过薄的世界描述得很清楚。

这意味着允许 `freeze` 的条件应同时围绕：

- 核心场景集合本身不明显过薄
- 核心业务流不再标题级
- 关键 trade-off 已比较
- 高风险未知项已显式外露
- 再来一轮也不会带来结构级提升

必须 `return-remediate` 的情况包括：

- 下游仍需自己补关键实体、关键状态、关键步骤、关键约束
- 当前选定的主线场景集合本身过薄，按它开发出来仍会明显 demo 化
- 关键流程仍然明显标题级
- 关键方案没有 why-this-not-that

短规则：

> **freeze 的标准不是“下游不用再猜”这一条，而是“下游不用再猜，而且当前主线世界本身不薄到只剩 demo”。**

进一步说，当第一层和第二层门槛已经满足后，还应再问一遍：

- 再来一轮，是否还会产生新的**实质性正向业务价值收益**
- 这个收益是否大于新增复杂度

如果答案是否定的，就不应继续为了“再想更多”而延长 loop。

---

## 8A. Principle 06A: 用户输入可以决定向上探索强度，但不能降低质量底线

`v1.2` 明确区分两种深度模式：

### Baseline Mode

这是默认模式。

默认收敛标准是：

- 当前主线世界达到真实世界一般水平
- 下游不需要再发明关键真相
- 再来一轮也难以带来新的实质性结构级提升

### Creative Mode

这不是默认模式。
只有当用户**显式要求**更强创造性探索时才进入。

它的前提不是跳过 `Baseline Mode`，而是先满足 `Baseline Mode`。

在此基础上，继续迭代直到：

- 再来一轮已无新的 `positive business value gain`
- 或新增收益已明显低于新增复杂度

这条规则的含义是：

- 用户可以决定向上探索多少
- 但用户输入不能把质量下限降到真实世界一般水平以下

短规则：

> **用户输入可以抬高思考上限，但不能降低业务基线下限。**

进一步边界：

- `P1`
  - 默认支持 `Baseline Mode`
  - 用户显式要求时支持 `Creative Mode`
- `P2`
  - 当前 `v1.2` canonical stance 是：只支持 `Baseline Mode`
  - `Creative Mode` 仅作谨慎保留，不作为当前默认或正式能力，以防技术方案过度设计
- `P3`
  - 不支持 `Creative Mode`
- `P4`
  - 不支持 `Creative Mode`

当 `P1` 进入 `Creative Mode` 时，输出必须分层：

- 一层是稳定可 handoff 的 baseline truth
- 一层是可选的 creative discoveries / opportunity set

这样可以避免下游把探索性发现误读为必须实现的主线真相。

## 9. Principle 07: P3 主线先以后端高质量交付为基线

`v1.2` 明确重置 `Phase-3` baseline：

不是：

- 全栈都同时高质量闭环

而是：

- 高质量 `Backend Delivery Package`

`P3` 的主线硬目标先聚焦：

- database truth
- migration truth
- backend service/API truth
- verification truth
- delivery gate truth

前端保持重要，但默认不再作为 `P3 mainline established` 的首要决定条件。

前端可以作为：

- minimal shell
- fallback pack
- optional lane
- explicit deferred surface

短规则：

> **先把 backend 交付真相做实，再谈 frontend 完整度。**

---

## 10. Principle 07A: P1 与 P2 不是同一种思维模式

`v1.2` 不允许再默认把 `P1` 和 `P2` 当成：

- 同一套 loop
- 同一套 checklist
- 同一套思考张力

更准确的分工是：

- `P1 = value-first + deep-loop-first`
- `P2 = support-first + checklist-first + focused-review`

`P1` 更强调：

- 业务收益
- 用户体验增量
- 场景选择
- 真实世界一般水平校准
- anti-demo stress test

`P2` 更强调：

- 对 `P1` 的工程支撑能力
- 多快好省
- 架构完整性
- realizability
- extensibility
- ROI-gated extensibility
- anti-overengineering review

短规则：

> **`P1` 负责把业务世界想对、想深；`P2` 负责把它支撑好、做稳、做省。**

`P2` 的更精确角色是：

> **以交付为约束、以 ROI 为门槛的架构判断者。**

这意味着 `P2` 既不能退化成最短路径实现说明，也不能膨胀成最大化未来态架构。
它应产出足够支撑首版交付、诚实面对可实现性、并在 ROI 成立时选择性预留扩展边界的架构。

详细 companion 说明：

- `docs/v1.2-phase-thinking-mode-split-v0.1.md`
- `docs/v1.2-p2-delivery-minded-architecture-principles-v0.1.md`

## 11. Principle 08: 可选分支必须显式，不得伪装成主线

很多复杂度并不是错，而是位置错了。

以下能力可以保留，但必须作为显式可选分支存在：

- fallback UI / prototype
- dispatch / multi-worker
- stronger traceability posture
- stronger hardening / coverage
- release/export sidecars
- preview / incubation tracks

它们不应压进默认主线的第一层解释里。

短规则：

> **复杂性可以存在，但必须退出默认主线心智模型。**

---

## 12. Principle 09: 评分标准应锚定交付物与下游消费，而不是脚本执行

`v1.2` 后续所有评分、评审、quality bar，都应首先锚定：

- 主交付物质量
- 下游可消费性
- formal state 与实际证据是否一致
- 业务深度是否充分
- deferred / review-bound 是否诚实外显

不应首先锚定：

- 脚本数量
- side artifact 数量
- wrapper 是否更复杂
- 控制逻辑是否更细

短规则：

> **评交付真相，不评控制表面。**

配套评分与验收矩阵：

- `docs/v1.2-mainline-scoring-and-acceptance-matrix-v0.1.md`

---

## 13. Principle 10: 显式外露未决真相，不用“全绿幻觉”掩盖问题

`review-bound`、`return`、`blocked` 都是有效状态。

`v1.2` 不追求把所有东西都伪装成：

- 绿
- ready
- fully complete

相反，它要求：

- formal state 真实
- unresolved truth 真实
- manual signoff 真实
- deferred items 真实

短规则：

> **诚实的挂账比虚假的完成更有价值。**

---

## 14. Principle 11: runtime closure 与 retained proof 必须分开

`v1.1` 已经暴露出一个结构性问题：

- runtime closure truth
- retained proof truth

这两种真相不能再混成同一套判断。

`v1.2` 必须明确两种模式：

### runtime closure mode

用于判断一条运行链是否真的具备：

- 启动
- migration
- runtime verification
- deployment / delivery truth

### retained proof mode

用于判断一个 curated tracked proof set 是否：

- 证据链清楚
- 可解释
- 代码级 / 方法级 proof 足够
- 不依赖被刻意裁掉的运行时副产物

短规则：

> **不要再用一套 closure 语义同时审判运行时真相和仓库存档真相。**

---

## 15. Principle 12: 证据优先于臆想，方法优先于自由发挥

`v1.2` 强调 thinking runtime，但不接受无约束脑补。

当出现高风险未知项时，应优先使用：

- operator clarification
- upstream clarification
- internal method assets
- anti-pattern cards
- reliable external/domain evidence

模板和 SOP 继续保留，但不是思考深度的主要来源。

短规则：

> **不要用更长的文本伪装更深的思考。**

---

## 16. Principle 13: 真实世界一般水平是业务基线下限

`v1.2` 必须明确一个新的业务深度下限：

> **在业务密度较高、现实操作特征明显的场景中，主线产出物的业务密度和丰度，不能低于当前真实世界的一般水平。**

这条规则不是要求：

- 一上来就达到行业顶尖 SOP 深度
- 把所有现实变体都写进主线
- 无限追求业务深度直到无底洞

它要求的是：

- 当前主线不能停留在过度理想化、过度简化、明显低于现实一般水平的世界模型
- 不能只因为内部逻辑自洽，就把一个明显偏薄的业务世界冻结下来

换句话说：

- `不让下游再猜` 只是下限的一部分
- `当前上游定义出来的业务世界，不能明显低于现实世界的一般水平` 是另一部分

短规则：

> **不能只做到“写清楚一个薄世界”，还要先确认这个世界本身没有薄到低于现实一般水平。**

---

## 17. Principle 14: 高业务密度场景必须做外部基线校准

当场景具有明显现实世界操作密度时，`v1.2` 默认要求增加一条基准操作：

- 定向联网搜索 / 外部资料校准

目的不是抄行业方案。
目的也不是让 agent 被搜索结果牵着走。

目的只是：

- 获取当前真实世界的一般操作基线
- 校准主线场景集合是否过薄
- 校准异常路径、现实约束、协作动作是否被遗漏
- 防止产出物停留在“看似对，但明显偏 demo”的水平

适用场景包括但不限于：

- 医疗 / 宠物医疗
- 餐饮 / 零售
- 物流 / 仓储
- 制造 / 检测
- 财务 / 合规
- 任何具有真实操作动作、单据、设备、测量、库存、审计、角色协作的领域

短规则：

> **高业务密度场景先做现实世界一般水平校准，再决定主线 freeze。**

---

## 18. Principle 15: 优先减少下游猜测，而不是优先增加上游字数

`v1.2` 不要求每个上游产出都越来越长。
它要求的是：

- 关键问题被回答
- 关键分歧被比较
- 关键未知项被外显
- 关键 handoff 不靠猜

所以如果一个新增内容：

- 没减少下游猜测
- 没提升关键决策清晰度
- 没改善 handoff

那它就很可能只是噪音。

短规则：

> **少一点无效扩写，多一点关键真相前置。**

---

## 19. Principle 16: 设计 drift 一旦出现，先回原则，不先补局部脚本

当某个阶段表现不好时，`v1.2` 默认不先问：

- 还能补什么 gate
- 还能加什么 wrapper
- 还能写什么 sidecar script

而是先问：

- 主交付物是否定义错了
- 下游需要的真相是否没带过去
- 该留给 agent 的判断是否被过早脚本化了
- 这是不是主线与可选分支没分清导致的

短规则：

> **先修主线定义，再修控制面。**

---

## 19A. Principle 16A: fresh blocker 必须先准入，不能自动扩成下一条长修复线

`v1.2` 进入后期收口时，不允许把每一次 fresh rerun 发现的新问题，都自动升级成当前版本必须继续解决的 blocker。

否则项目会自然滑向：

- 第二案例泛化问题不断回灌 GA 主线
- case-specific gap 与 shared mainline defect 混在一起
- 统计口径问题被误当成能力缺口
- fresh rerun 持续制造新的长工单

因此，`v1.2` 对 fresh blocker 的默认顺序必须改成：

1. 先判定这个问题是否真的属于当前版本的 `GA mainline`
2. 再判定它是 shared defect、case-specific gap，还是统计/证据口径问题
3. 只有通过准入，才允许进入新的修复工单

一个 fresh 问题只有同时满足以下条件，才应被准入为当前版本 blocker：

- 它落在默认主线，而不是 optional lane / preview / 第二案例泛化层
- 它破坏硬性验收门槛，或制造 formal state 与真实证据不一致的 false green
- 它由 fresh artifact 稳定复现，而不是 stale artifact、统计口径不一致或单次噪音
- 它可以在不重开上游阶段、不扩控制面、不新增厚 wrapper 的前提下被有界修复

如果不满足以上条件，正确动作不是继续扩线，而是：

- 显式冻结 truth
- 标记为 residual / review-bound / next patch candidate
- 保持当前版本收口节奏

短规则：

> **先判定这是不是当前版本 blocker，再决定要不要继续修。**


---

## 19B. Principle 16B: 复杂系统问题必须先过三层五步递归闭环

当问题涉及 trace/source binding、resolver false negative、根因不清、跨阶段缺口、release-risk 判断时，不能直接从信号跳到修复。

默认使用三层五步递归闭环：

1. `发现层`：世界 → 信号，让异常可复述、可定位、可重现。
2. `定义层`：信号 → 问题，让问题可证伪、可度量。
3. `解决层`：问题 → 行动，让动作可执行、可验证、可复盘。

每一层都使用：

> `Baseline → Target → Gap → Strategy → Breakdown`
>
> `基 → 标 → 差 → 策 → 拆`

这条原则的直接含义是：

- 不允许在信号不可重现时定义根因
- 不允许在问题不可证伪时制定方案
- 不允许在 Action 不可验证时宣布完成
- 不允许把 `missing`、`resolver 未命中`、`not required`、`release blocker` 混成一个桶

对于 v1.2.3 P3 source precision 问题，正确顺序是：

1. 先修 trace/source binding 的眼睛和坐标系
2. 再重新判断 P2 是否真的缺设计图
3. 最后才调整 P2 SKILLS 的图产出要求

短规则：

> **先修眼睛和坐标系，再判断地图缺不缺路。**

详细方法见：

- `runtime-deps/mindthus/source/skills/3l5s/resources/three-layer-recursive-loop.md`

## 20. 反模式清单

以下现象在 `v1.2` 中应被直接视为设计偏航信号：

- 用新增脚本代替业务判断
- 用更复杂 gate 代替更清晰交付物
- 用更多 side artifacts 代替更强 downstream consumability
- 把 optional lane 混进 default mainline
- 把 frontend 完整度继续作为 `P3` 首要硬门槛
- 以“看起来很全”掩盖 demo 感
- 让 formal state 与实际证据持续脱节
- 用 retained proof 去证明 runtime closure，或反过来
- 把第二案例 fresh rerun 的新问题自动升级成当前版本必须继续修的长 blocker
- 在没有 blocker admission 的情况下继续无限扩工单
