# 故事工作坊与持续提炼模式

- 类型：步骤
- 适用阶段：需求拆解 / 需求提炼 / 交付前澄清
- 来源：`sources/books/original/用户故事地图.md`（第16章）
- 摘要：用户故事不应一次性写完定稿，而应通过持续的故事对话和工作坊逐步提炼，使故事在进入实现前获得足够但不过度的清晰度。

## 输入条件
- 已有初步故事地图或需求卡片
- 团队准备将需求进一步细分到可实现/可验证粒度

## 关键步骤
1. 基于故事地图组织故事工作坊或故事对话。
2. 对关键故事进行细分和瘦身，直至达到当前交付所需粒度。
3. 在迭代/冲刺计划前集中处理即将进入开发的故事。
4. 保持持续对话，而不是一次性批量写完所有细节。

## 输出结果
- 更适合进入设计/开发/测试阶段的故事粒度
- 可执行且可讨论的需求条目
- 与地图保持关联的细化结果

## 适用边界
- 适用于迭代式交付、逐步细化需求的产品环境
- 不适用于要求一次性冻结所有设计细节的大型制度基线文档场景

## 常见错误
- 在太早阶段把所有故事细化到底
- 把工作坊变成需求宣读会
- 在进入开发前缺少必要提炼，导致实现期大量返工

## 关联模板
- 故事工作坊记录
- 需求细化记录
- 迭代前故事准备清单

## 下游使用建议
- 可被未来“需求拆解 Skill”直接引用为提炼方法
- 可支撑“接口设计 Skill”与“测试设计 Skill”获取更细粒度输入
- 可强化 gate：未提炼到可消费粒度的故事，不进入实现相关阶段
