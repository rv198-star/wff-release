# v1.2 主线评分与验收矩阵（v0.1）

## 这份文档是做什么的

这份文档把 `v1.2` 的跨阶段评审标准正式冻结成一份统一矩阵。

它回答的不是：

- 这个阶段跑了多少脚本
- 生成了多少 side artifact
- wrapper 看起来有多完整

它回答的是：

> **这个阶段的唯一主交付物，是否足够好、足够真、足够可消费，能够支撑下游高质量继续推进。**

这份文档的用途是：

- 作为 `v1.2` 阶段评审的统一评分标准
- 作为 `PASS / PASS with review-bound items / RETURN-REMEDIATE / BLOCKED` 的统一验收依据
- 防止每次评审都重新发明标准

配套文档：

- `docs/v1.2-design-principles-v0.1.md`
- `runtime-deps/mindthus/source/skills/wae/SKILL.md`
- `docs/v1.2-phase-thinking-mode-split-v0.1.md`
- `docs/internal/improvement-reports/v1.2-mainline-deliverables-and-backend-quality-bar-v0.1.md`
- `docs/internal/improvement-reports/v1.2-business-depth-sufficiency-and-superpowers-inspired-agent-loop-v0.1.md`

## 使用规则

### Rule 1: 先验收，后评分

评分不能替代验收。

如果主交付物的硬性验收条件没有满足，即使总分看起来还可以，也不能直接判定为 `PASS`。

短规则：

> **hard acceptance gate > numeric score**

### Rule 2: 默认只评主线，不评 optional lane

默认评审对象是：

- `P1` 的 `implementation-ready PRD / UPP`
- `P2` 的 `ESP`
- `P3` 的 `Backend Delivery Package`
- `P4` 的 `Validation Closure Package`

默认不把以下内容作为主线评分必要项：

- `P1` 的额外市场研究与叙事扩写
- `P2` 的扩展架构和额外视觉表达
- `P3` 的完整前端、fallback UI、dispatch / multi-worker、额外 hardening
- `P4` 的 release-readiness extension、额外视觉证据包

除非用户明确把这些 optional lane 提升为本次任务主线。

### Rule 3: 评分围绕交付真相，不围绕脚本表面

优先评分：

- 主交付物质量
- 下游可消费性
- formal state 与实际证据是否一致
- deferred / review-bound 是否诚实外显

不优先评分：

- 脚本数量
- wrapper 数量
- side artifact 数量
- 控制逻辑的细碎程度

### Rule 4: `P1 Creative Mode` 不改变 baseline 验收底线

`P1` 若进入 `Creative Mode`：

- 必须先满足 `Baseline Mode`
- baseline 验收仍按这份矩阵执行
- creative discoveries 作为可选增益层单独呈现

它可以增加价值，不可以替代 baseline 合格线。

### Rule 5: repo-level closure 消费阶段 verdict，不重发一套阶段评分

这份矩阵冻结的是**阶段级**评分与验收口径。

因此 repo-level closure 的职责应是：

- 聚合各阶段已经产出的 `phase-verdict.json`
- 汇总 review-bound / deferred / blocker 边界
- 在 `runtime-closure` 与 `retained-proof` 模式下诚实外显缺失证据与 fallback 口径

它**不应**做的事是：

- 重新发明一套与阶段 runtime 脱节的二次评分
- 在 canonical verdict 已存在时继续以 legacy `status` 为主
- 用 repo-level 包装覆盖阶段级真实 verdict

短规则：

> **阶段怎么判，由阶段 verdict 说了算；repo-level 负责汇总，不负责重判。**

## 评分刻度

每个评分维度使用 `0-5` 分：

| 分值 | 含义 |
|---|---|
| `0` | 缺失 / 明显阻塞 / 下游无法继续 |
| `1` | 有轮廓但明显偏薄，强烈 demo-like |
| `2` | 部分覆盖，但仍需要下游自行补关键真相 |
| `3` | 达到主线可接受下限，足以支持下游继续 |
| `4` | 强，主要真相清楚，评审风险较低 |
| `5` | 很强，几乎可作为该阶段的标杆主线交付物 |

总分计算：

```text
total_score = Σ(维度得分 / 5 × 维度权重)
```

总分范围为 `0-100`。

---

## 分数档位

| 分数区间 | 解释 |
|---|---|
| `90-100` | 强 `PASS` 候选，已接近该阶段高质量主线交付物 |
| `80-89` | `PASS` 候选，仍可有少量 review-bound 项 |
| `70-79` | 最多只能是 `PASS with review-bound items` |
| `<70` | 默认应进入 `RETURN-REMEDIATE`，除非只是局部试评分而非正式验收 |

补充说明：

- 分数只能表达质量强弱
- 最终 verdict 仍由验收状态决定
- `BLOCKED` 可以发生在任何分数区间，只要缺少不可合理推断的关键真相

---

## 验收状态

| 状态 | 含义 |
|---|---|
| `PASS` | 主交付物已满足主线要求，下游可正常继续 |
| `PASS with review-bound items` | 主交付物可继续下游，但必须保留显式 review-bound / deferred 边界 |
| `RETURN-REMEDIATE` | 主交付物结构可能存在，但当前仍不足以高质量支撑下游，需要回修 |
| `BLOCKED` | 缺少不可安全推断的关键真相或关键证据，当前不应继续 |

统一判定规则：

- `PASS`
  - 所有硬性验收项通过
  - 总分通常应 `>= 80`
  - 且没有任何维度低于 `3`
- `PASS with review-bound items`
  - 所有硬性验收项通过
  - 总分通常应 `>= 70`
  - 但仍存在显式、有限、诚实外露的 review-bound 项
- `RETURN-REMEDIATE`
  - 任一硬性验收项失败
  - 或总分 `< 70`
  - 或下游仍需发明关键真相
- `BLOCKED`
  - 关键真相不可推断
  - 或关键证据缺失，继续只会制造伪完成

## P1 评分表

### 主交付物

- `implementation-ready PRD / UPP`

### 评分维度

| 维度 | 权重 | `5` 分信号 | `3` 分信号 | `0-2` 分信号 |
|---|---:|---|---|---|
| 业务基线与场景集充分性 | 30 | 主线世界达到或略高于真实世界一般水平，核心场景集合不薄 | 主线世界基本可信，核心场景足够支撑下游 | 世界过薄、场景缺族、按此实现仍明显 demo 化 |
| 场景深度与可执行性 | 25 | 核心场景已操作化，状态/异常/约束清楚，验收标准可执行 | 核心场景不再标题级，验收标准可用于设计和实现入口 | 标题级流程、验收标准空泛、下游需自行补步骤和状态 |
| `P2` 可消费性 | 20 | `P2` 可直接消费，不需重构需求世界 | `P2` 基本可启动，仅有少量 review-bound 边界 | `P2` 需要自行发明核心实体、边界或约束 |
| 证据/现实校准/诚实度 | 15 | 高业务密度场景有现实世界一般水平校准，未知项外露诚实 | 主要未知项外露，基线判断基本可信 | 无基线校准、证据姿态含混、以格式掩盖薄弱真相 |
| 主线纪律与 optional 分层 | 10 | baseline truth 清楚，optional/creative 内容不污染主线 | 主线基本清楚，附加内容可区分 | 主线和附加内容混杂，导致下游误读 |

### 硬性验收条件

| 必须通过项 | 说明 |
|---|---|
| 主 PRD / UPP 存在且自洽 | 不接受只有零散 stage 输出、没有收敛主文档 |
| 核心场景不再标题级 | 不能只剩 use case 标题和 happy path 口号 |
| 验收标准可被下游消费 | 至少能驱动 `P2` / `P3` 的后续设计与实现 |
| 范围、约束、review-bound 项显式 | 不能把未决真相藏在 prose 里 |
| 高业务密度场景完成现实世界基线校准 | 如医疗、零售、物流、制造等 |
| `P2` 不需要再发明关键产品真相 | 否则即使文档很长也不算通过 |

### `P1` 追加规则

- 如果启用了 `Creative Mode`：
  - baseline truth 必须与 creative discoveries 分层
  - creative discoveries 不计入 baseline 硬性验收替代项

### 硬性验收条件

| 必须通过项 | 说明 |
|---|---|

| 可运行 backend service boundary 存在 | 不是纯脚手架 |
| API contract 已冻结并实现 | 不接受 contract 仍漂移 |
| service + persistence verification evidence 存在 | 至少对主线能力有真实验证 |
| startup / env / delivery gate 显式 | formal state 必须与证据一致 |
| frontend 缺失被诚实标记 | 但不再默认阻塞 backend mainline |

### `P3` 追加规则

- `P3` 默认只评 backend-first 主线
- 前端、fallback UI、dispatch/multi-worker、额外 hardening 默认不计入主线硬门槛

- `Validation Closure Package`

### 评分维度

| 维度 | 权重 | `5` 分信号 | `3` 分信号 | `0-2` 分信号 |
|---|---:|---|---|---|
| 覆盖策略与证据质量 | 25 | 范围、策略、证据链、覆盖重点清楚，和实现边界一致 | 主要验证范围和证据已成立 | 测了什么、没测什么、为什么都不清楚 |
| 缺陷/风险/verdict 质量 | 30 | 风险分级清楚，verdict 可信，对 false green 有对抗性 | 有明确 verdict 和主要风险说明 | 更像 closure 包拼装，而非真正判断 |
| 下游决策可消费性 | 20 | operator 可直接据此做继续/回修/发布判断 | 基本能支持下游决策 | 下游仍需猜还能不能发、风险多大 |
| 诚实度与人工签发边界 | 15 | unresolved truth、human signoff dependency、review-bound 项全部显式 | 主要未决项已显式外露 | 伪全绿、隐性挂账、人工边界不清 |
| 主线与可选 release extension 分层 | 10 | validation closure 与 release readiness 清晰分离 | 基本可分辨 | 两者纠缠，导致主线 closure 失真 |

### 硬性验收条件

| 必须通过项 | 说明 |
|---|---|
| 验证范围与 verdict 显式 | 不接受只有零散测试记录 |
| 证据与 verdict 能对上 | 不能口头 pass、证据空心 |
| 风险、缺陷、review-bound 项显式 | 不能藏未决真相 |
| human signoff dependency 显式 | 需要人工确认的不能伪装为机器全绿 |
| release-readiness 若存在，必须与主线 closure 分开 | 避免再度语义纠缠 |

## 推荐输出物

后续 runtime / review 执行时，建议每阶段至少产出：

- `phase-mainline-scorecard.md`
  - 记录维度分、总分、主要加减分理由
- `phase-acceptance-matrix.md`
  - 记录硬性验收项是否通过，以及 review-bound / deferred / blocker
- `phase-verdict.json`
  - 记录 `phase`、`total_score`、`dimension_scores`、`verdict`、`review_bound_items_count`、`blockers_count`

短规则：

> **分数表负责表达强弱，验收表负责表达能不能继续。**

## 一句话规则

> **`v1.2` 的评分和验收，不是评谁控制得更细，而是评主交付物是否足够真、足够清楚、足够支持下游高质量继续。**
