# 重构的定义：保持可观察行为不变，以降低修改成本

- 类型：概念 / 边界
- 适用阶段：开发实现；维护 / 重构
- 来源：`sources/books/original/phase-others/refactoring-improving-the-design-of-existing-code.md`（第2章 2.1）
- 摘要：重构不是“任何代码清理”。它的核心约束是：**不改变可观察行为**的前提下调整内部结构，以提高可理解性、降低未来修改成本。

## 输入条件

- 你需要改进代码结构，但不打算改变对用户/调用方可观察到的行为

## 关键步骤

1. 明确“可观察行为”的边界：对用户/调用方来说什么算行为（输出、错误、接口、可用性等）。
2. 把当前行为锁定为可自动验证的测试（或可复现的验收样例）。
3. 以小步骤逐步调整结构；每步都保持系统可工作。
4. 如果需要改变行为，把它标记为“功能变更”，不要混入重构叙事中。

## 输出结果

- 更易理解、更易修改的结构；外部可观察行为保持一致

## 适用边界

- 如果目标是“更快”，而牺牲可理解性/可维护性可接受，则这更像性能优化，而不是重构。
- 如果必须改变接口或业务行为，要把它视为明确的功能变更，并配套不同的验证与评审。

## 常见错误

- 用“重构”掩盖行为变更（导致回归无法定位）
- 把大重写/不可用的结构调整也称为重构（违背小步可工作约束）
